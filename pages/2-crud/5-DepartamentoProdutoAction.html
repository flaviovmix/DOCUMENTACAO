Classe base abstrata de todas as actions de Produto. Centraliza o que DepartamentoProdutoList, DepartamentoProdutoForm e suas inner classes precisam compartilhar: o prefixo ROOT, os objetos Bean e WBean da requisição, e o acesso ao ID do produto em edição.

package br.xt.app.departamento.produto;

import br.xt.AppsRootAction;
import br.xt.app.departamento.DepartamentoManager;

public abstract class DepartamentoProdutoAction extends AppsRootAction {

    public static final String ROOT = DepartamentoManager.F_ACESSO_MODULO.concat("__PRO/");

    private DepartamentoProdutoWBean wbean = null;
    public DepartamentoProdutoWBean proWBean() throws Exception {
        if (wbean == null)
            wbean = (DepartamentoProdutoWBean) getManager().getInstance(DepartamentoProdutoWBean.class);
        return wbean;
    }

    private DepartamentoProdutoBean bean = null;
    public DepartamentoProdutoBean proBean() throws Exception {
        if (bean == null)
            bean = (DepartamentoProdutoBean) getManager().getInstance(DepartamentoProdutoBean.class);
        return bean;
    }

    public Integer id_pro() throws Exception {
        return getSession().getInteger(ROOT.concat(DepartamentoProdutoBean.ID_PRO));
    }

}
O que o extends AppsRootAction traz?

Toda a infraestrutura que o CRUD precisa vem da herança. Sem ela, cada action teria que implementar tudo do zero:

ROOT — montado em duas partes

O ROOT não surge do nada — ele é a junção de duas partes:

  1. No Manager: F_ACESSO_MODULO = "XT.PAINEL_CONTROLE.ACESSO_MODULO.DEPARTAMENTO"
  2. Na Action: .concat("__PRO/") junta o sufixo da entidade

Resultado: "XT.PAINEL_CONTROLE.ACESSO_MODULO.DEPARTAMENTO__PRO/"

O Manager define o endereço do módulo. A Action completa com o endereço da entidade dentro do módulo. Se tivesse outra entidade (ex: Pessoa), seria __PES/ em vez de __PRO/.

Padrão lazy: getInstance()

Os métodos proBean() e proWBean() usam o padrão de inicialização lazy: o objeto só é criado na primeira chamada e reutilizado nas seguintes.

LinhaO que faz
private ... bean = nullCampo privado inicializado como null — ainda não foi criado
if (bean == null)Só instancia na primeira chamada. Se já foi criado, pula direto para o return
getManager().getInstance(...)Retorna a instância no escopo da requisição. Todas as actions que chamam na mesma requisição recebem o mesmo objeto
Por que id_pro() lê da sessão?

O formulário tem várias ações (salvar, excluir, cancelar) e cada uma é uma requisição separada. O ID precisa sobreviver entre requisições — a sessão garante isso.

Quando o usuário clica num registro da lista, o ShowUpdate grava o ID na sessão. A partir daí, qualquer action que precisar saber qual produto está aberto chama id_pro().
