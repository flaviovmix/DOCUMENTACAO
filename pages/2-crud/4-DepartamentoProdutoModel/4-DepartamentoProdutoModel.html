O Model é a camada de regras de negócio — fica entre o formulário e o DAO. O formulário nunca chama o DAO diretamente para salvar dados — sempre passa pelo Model. A cadeia é: Formulário → Model → DAO → Banco.

Nota: neste CRUD simples não há regras de negócio, por isso o arquivo é pequeno — insert e update apenas delegam para o DAO. Conforme o módulo evoluir, as regras entram aqui sem precisar alterar o formulário nem o DAO.

CÓDIGO COMPLETO 
package br.xt.app.departamento.produto;

import br.jasap.core.AppManager;

public class DepartamentoProdutoModel extends DepartamentoProdutoDAO {

    public DepartamentoProdutoModel(AppManager manager) {
        super(manager);
    }

    public void insert(DepartamentoProdutoBean bean) throws Exception {
        daoInsert(bean);
    }

    public void update(DepartamentoProdutoBean bean) throws Exception {
        daoUpdate(bean);
    }

}
Por que o Model é tão pequeno?

O Model concentra as regras de negócio que precisam rodar antes de persistir um dado. Em módulos mais complexos, o insert() e o update() fariam coisas como:

  • Montar o qs_pro (Quick Search) antes de salvar
  • Validar campos obrigatórios ou combinações inválidas
  • Propagar status de um registro pai pros filhos
  • Atualizar contadores em outras tabelas

Neste CRUD não há nada disso — então o Model só delega para o DAO.

Você pode estar se perguntando: "por que não chamar o DAO direto do formulário?" — A resposta é separação de responsabilidades. Se amanhã precisar adicionar uma regra no insert, ela entra no Model sem tocar no formulário nem no DAO.

Por que delete não está aqui?

Delete não tem regra de negócio neste módulo. O formulário chama daoDelete() diretamente — como o Model herda do DAO, o formulário faz getFactory().departamento().proModel().daoDelete(bean) e funciona.

Se no futuro delete precisar de regra (ex: arquivar em vez de excluir, ou verificar dependências antes), um método delete() seria adicionado aqui no Model.
