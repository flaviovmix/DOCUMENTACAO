A MdFactory foi criada vazia no passo "Novo Módulo". Agora que o Model existe, adicionamos o método proModel() para que as actions consigam acessar o Model de Produto via getFactory().departamento().proModel().

CÓDIGO COMPLETO 
package br.xt.app.departamento;

import br.jasap.core.AppManager;
import br.xt.app.departamento.produto.DepartamentoProdutoModel;

public class DepartamentoMdFactory {

    protected AppManager manager = null;

    public DepartamentoMdFactory() {
    }

    public DepartamentoMdFactory(AppManager manager) {
        setManager(manager);
    }

    public AppManager getManager() {
        return manager;
    }

    public void setManager(AppManager manager) {
        this.manager = manager;
    }

    protected DepartamentoProdutoModel proModel = null;
    public DepartamentoProdutoModel proModel() {
        if (proModel == null) proModel = new DepartamentoProdutoModel(manager);
        return proModel;
    }

}
Por que proModel() fica na MdFactory e não na Action?

A MdFactory é o ponto central de acesso aos Models do módulo. Qualquer action de qualquer módulo pode acessar o Model de Produto via:

getFactory().departamento().proModel()

Se o proModel() ficasse na Action, só as actions de Produto teriam acesso. Com a MdFactory, até uma action de outro módulo (ex: Laboratório) poderia acessar o Model de Produto do Departamento se precisasse.

Padrão lazy — criado uma vez, reutilizado

O mesmo padrão que aparece no proBean() da Action:

LinhaO que faz
protected ... proModel = nullCampo começa vazio — o Model ainda não foi criado
if (proModel == null)Só cria na primeira chamada. Nas seguintes, retorna o mesmo objeto
new DepartamentoProdutoModel(manager)Cria o Model passando o manager — que repassa ao DAO via super(manager), configurando a conexão com o banco

A corrente completa de criação:

Action chama getFactory().departamento().proModel()
  → MdFactory cria DepartamentoProdutoModel(manager)
    → Model chama super(manager) → DAO configura conexão com banco

public DepartamentoProdutoModel proModel()

ElementoO que significa
protected (campo)Visível para esta classe e subclasses — mesmo padrão do campo manager
public (método)Acessível de fora — as actions chamam getFactory().departamento().proModel()
DepartamentoProdutoModelTipo de retorno — o Model que herda do DAO e contém as regras de negócio

Para cada nova entidade adicionada ao módulo, um método semelhante seria criado aqui. Exemplo: se o módulo tivesse uma entidade Pessoa, teria um pesModel() ao lado do proModel().
