package br.xt.app.departamento.produto;

import br.jasap.core.AppManager;
import br.jasap.dao.Query;
import br.jasap.util.JasapList;
import br.xt.AppsRootDAO;

public class DepartamentoProdutoDAO extends AppsRootDAO {

    public DepartamentoProdutoDAO() {
    }

    public DepartamentoProdutoDAO(AppManager manager) {
        setManager(manager);
        setDataBase(manager.getDataBase());
    }

    public void daoList(JasapList list) throws Exception {
        StringBuilder sql = new StringBuilder();
        sql.append("select * from " + DepartamentoProdutoBean.TABLE);
        sql.append(" where 1=1 <orderby>");
    
        Query query = getDataBase().getQuery(sql.toString());
        select().executeList(query, list);
    
        while (query.next()) {
            DepartamentoProdutoBean bean = new DepartamentoProdutoBean();
            query.populateBean(bean);
            list.getList().add(bean);
        }
        query.release();
    }

    public void daoSingle(DepartamentoProdutoBean bean) throws Exception {
        select().execute(bean, DepartamentoProdutoBean.TABLE);
    }

    public void daoInsert(DepartamentoProdutoBean bean) throws Exception {
        insert().execute(bean, DepartamentoProdutoBean.TABLE);
    }

    public void daoUpdate(DepartamentoProdutoBean bean) throws Exception {
        update().execute(bean, DepartamentoProdutoBean.TABLE);
    }

    public void daoDelete(DepartamentoProdutoBean bean) throws Exception {
        delete().execute(bean, DepartamentoProdutoBean.TABLE);
    }

}
Hierarquia: de onde vêm os métodos?

O DAO herda em dois níveis:

JasapDAO (Jasap) → AppsRootDAO (projeto) → DepartamentoProdutoDAO

ClasseO que fornece
JasapDAOselect(), insert(), update(), delete(), qsWhere(), getDataBase() — operações básicas de banco
AppsRootDAOgetFactory() — acesso a MdFactories de outros módulos; getUser() — usuário logado; ok() — verificar permissão

Quando o DAO faz insert().execute(bean, TABLE), o insert() vem do JasapDAO. O DAO não implementa SQL de INSERT — o Jasap faz por ele.

Por que dois construtores? (e o Bean/WBean não tem nenhum)

O Bean e o WBean não declaram construtores — o Java cria um vazio automaticamente. Basta porque são só objetos de dados.

O DAO precisa de dois construtores explícitos:

ConstrutorQuem chamaPara quê
DepartamentoProdutoDAO()Jasap (reflection)O framework pode precisar instanciar o DAO via Class.newInstance() — sem construtor vazio, falha em runtime
DepartamentoProdutoDAO(AppManager)Model (super(manager))Configura setManager() + setDataBase() — sem isso, o DAO não tem conexão com o banco

Corrente de criação: MdFactory → Model → super(manager) → DAO configurado com conexão ao banco.

Por que daoList é diferente dos outros métodos?

Os 4 métodos automáticos (daoSingle, daoInsert, daoUpdate, daoDelete) operam sobre um único registro — o Jasap monta o SQL automaticamente usando @DBInfo.

O daoList retorna vários registros e precisa:

  1. Montar o SQL na mão (com placeholders)
  2. Executar com paginação
  3. Iterar registro por registro
  4. Criar um Bean pra cada linha e popular via reflection
  5. Liberar recursos
O que é "where 1=1"?

1=1 é sempre verdadeiro — não filtra nada sozinho. É um truque clássico de SQL que serve de "âncora" pra concatenar filtros com AND depois:

WHERE 1=1 AND st_pro=1 AND nome_pro LIKE '%João%'

Sem o 1=1, o primeiro filtro precisaria saber se deve usar WHERE ou AND. Com o 1=1, todos os filtros são sempre AND — simplifica a montagem.

daoSingle(DepartamentoProdutoBean bean)

Carrega um único registro pelo ID. O bean deve ter id_pro preenchido — o Jasap usa @DBInfo(pk=true) para montar SELECT * FROM laboratorio.pro WHERE id_pro = ? automaticamente. Após a execução, os demais campos do bean ficam preenchidos.

daoInsert(DepartamentoProdutoBean bean)

Insere um registro novo. O Jasap omite o campo serial=true do INSERT e, após executar, grava o ID gerado pelo banco de volta no bean via setId_pro().

daoUpdate(DepartamentoProdutoBean bean)

Atualiza um registro existente. O Jasap monta: UPDATE laboratorio.pro SET nome_pro=..., vl_pro=..., ... WHERE id_pro = ?. Usa @DBInfo(pk=true) pro WHERE e omite a PK do SET.

daoDelete(DepartamentoProdutoBean bean)

Exclui um registro pelo ID. Monta: DELETE FROM laboratorio.pro WHERE id_pro = ?. Se houver FK em outra tabela apontando para esse registro, o banco lança SQLConstraintException.
