getManager() conceitual

0:00 / 0:00

O getManager() aparece em vários lugares do CRUD — na Action, na XtPage, no DAO, na MdFactory e no Model. Cada um usa de um jeito diferente, e isso pode confundir. Esta página explica o fluxo completo: de onde vem, como é montado, e como chega em cada lugar.

Analogia — O Assistente com a Maleta

O getManager() é a maleta do gerente do departamento (DepartamentoManager).

Fase 1 — Antes do expediente (sistema inicia):

  1. O RootManager (gerente geral) abre o prédio
  2. Ele cria os gerentes de cada departamento — o DepartamentoManager é um deles
  3. O DepartamentoManager faz o cadastro dos seus funcionários usando regAction(): "DepProdutoList pertence ao DepartamentoManager", "DepProdutoForm pertence ao DepartamentoManager", e assim por diante
  4. Neste momento ele NÃO entrega a maleta pra ninguém. Só anota quem pertence a quem

Fase 2 — Durante o expediente (requisição do usuário):

  1. Um cliente chega — o usuário clicou em algo no navegador
  2. O framework age como um assistente que navega pelo prédio:
    • Olha o quadro de cadastro: "DepProdutoList pertence ao DepartamentoManager"
    • Pega uma maleta vazia do DepartamentoManager
    • Vai passando pelos andares colocando o que precisa dentro:
      • Almoxarifado → conexão com o banco
      • RH → usuário logado
      • Ferramentaria → acesso às instâncias (getInstance)
    • Entrega a maleta pro funcionário (a action)
  3. O funcionário recebe a maleta pronta — quando chama getManager(), tudo já está dentro
  4. O cliente vai embora (requisição termina) — a maleta é descartada
  5. Próximo cliente (próximo clique): maleta nova, processo repete do zero

Ponto-chave: cada clique = uma requisição = uma maleta nova. A maleta é descartável — dura só o tempo de uma requisição.

A Ficha do Cliente — a Sessão

Se a maleta é descartada a cada clique, como o sistema lembra qual produto o usuário estava editando? Ou em qual página da lista ele estava?

A resposta é a sessão — que funciona como a ficha do cliente na recepção do prédio.

Na primeira vez que o cliente entra no prédio, a recepção cria uma ficha pra ele. A partir daí, toda vez que ele volta, o assistente pega a mesma ficha e coloca dentro da maleta nova.

Nessa ficha tem anotado:

  • Qual produto ele estava editando (id_pro)
  • Em qual página da lista ele estava
  • Quais filtros ele tinha aplicado
  • Que ele está logado como "Flávio"

A maleta morre a cada clique. A ficha sobrevive. A ficha só é descartada quando o cliente vai embora de vez (sessão expira ou faz logout).

Exemplo prático — Lista de 15 mil produtos

O cliente chega e diz: "me vê a lista de produtos disponíveis."

O sistema tem 15 mil produtos. Pra não entregar trocentos papéis, o funcionário entrega só os primeiros 500, ordenados em ordem alfabética. Antes de entregar, anota na ficha: "Flávio levou os primeiros 500 produtos, em ordem alfabética."

Maleta descartada. Ficha atualizada na recepção.

O Flávio olha a lista e percebe que o produto que queria não está ali. Volta e pede mais.

O assistente pega uma maleta nova, vai na recepção, pega a ficha do Flávio, coloca dentro da maleta, e entrega pro funcionário.

O funcionário abre a maleta, olha a ficha: "esse cara parou nos 500, em ordem alfabética." Vai no almoxarifado e pega do 501 ao 1000, na mesma ordem.

Atualiza a ficha: "agora está nos 501 a 1000, manteve ordem alfabética."

No código, quem carrega esses dados entre requisições é o proWBean() guardado na sessão.

Onde o getManager() aparece no CRUD

OndeO que faz com a maletaCódigo
ActionAbre a maleta → pega instâncias (Bean, WBean)getManager().getInstance(DepProdutoBean.class)
XtPageRecebe a maleta inteira → sabe qual módulo énew XtPage(getManager())
DAOAbre a maleta → pega a conexão com o bancosetDataBase(manager.getDataBase())
MdFactoryGuarda a maleta → repassa pros ModelssetManager(manager)
ModelRecebe a maleta → repassa pro DAO via herançasuper(manager)

Na Action

O funcionário abre a maleta pra pegar fichas em branco:

getManager().getInstance(DepartamentoProdutoBean.class)   // ficha de dados do produto
getManager().getInstance(DepartamentoProdutoWBean.class)   // ficha de filtros

Na XtPage

A página recebe a maleta inteira pra saber de qual departamento ela é — pra montar a estrutura correta (CSS, JS, título):

XtPage page = new XtPage(getManager());

No DAO

O construtor recebe a maleta e tira a conexão com o banco de dentro:

public DepartamentoProdutoDAO(AppManager manager) {
    setManager(manager);                    // guarda a maleta
    setDataBase(manager.getDataBase());     // pega a conexão com o banco
}

Na MdFactory

A MdFactory recebe a maleta e guarda pra repassar aos Models quando forem criados:

public DepartamentoMdFactory(AppManager manager) {
    setManager(manager);
}

No Model

O Model recebe a maleta e repassa pro DAO pela herança:

public DepartamentoProdutoModel(AppManager manager) {
    super(manager);    // chama o construtor do DAO
}

A corrente completa:

MdFactory guarda a maleta → cria o Model → Model repassa ao DAO via super(manager) → DAO pega a conexão com o banco

Resumo

  • getManager() é a maleta do gerente do departamento
  • O regAction() só cadastra quem pertence ao departamento — não entrega a maleta
  • A maleta só é montada e entregue quando uma requisição chega de verdade
  • Cada clique = maleta nova, descartada no final
  • A sessão (ficha do cliente) sobrevive entre requisições — é colocada dentro da maleta nova
  • A mesma maleta circula por todo o módulo: Action, XtPage, DAO, MdFactory e Model
