Camada 4 — Transporte

0:00 / 0:00

A camada 3 entrega pacote pra outra máquina. A camada 4 entrega pra processo certo dentro daquela máquina. É aqui que aparecem portas (que distinguem Tomcat de Postgres na mesma máquina), UDP (rápido, sem garantia) e TCP (confiável, com handshake e retransmissão). Toda conexão HTTP, banco, e-mail, SSH, jogo online passa por essa camada.

UDP — atire e esqueça +
0:00 / 0:00

UDP (User Datagram Protocol) é o protocolo mais simples da camada 4. Connectionless: você joga o pacote na rede e segue a vida. Sem aperto de mão antes, sem confirmação depois. Se chegou, ótimo. Se perdeu, foi-se.

Header minúsculo — só 8 bytes

CampoTamanhoO que faz
Porta origem2 bytesQuem mandou
Porta destino2 bytesPra quem é
Tamanho2 bytesComprimento total do datagrama
Checksum2 bytesDetecta corrupção (não corrige)

É só isso. Header de 8 bytes. Comparar com TCP que tem 20+ bytes de header — UDP é metade do peso.

O que UDP NÃO faz

  • Não estabelece conexão — sem handshake
  • Não garante entrega — pacote perdido fica perdido
  • Não garante ordem — pacote pode chegar fora da ordem
  • Não retransmite — se quiser retentar, é responsabilidade da aplicação
  • Não controla fluxo — manda na velocidade que quiser

Quando usar UDP

UDP brilha quando velocidade importa mais que confiabilidade:

  • DNS — pergunta curta, resposta curta (1 pacote pra cada lado). Esperar handshake TCP seria desperdício
  • VoIP / vídeo (chamada ao vivo) — perder 1 frame de vídeo é melhor que travar 200ms esperando retransmissão
  • Jogo online — posição do jogador atualiza 60 vezes por segundo; pacote velho perdido é irrelevante, o próximo já vem
  • Streaming ao vivo — mesma lógica de jogo
  • NTP (sincronização de hora) — pacotinho minúsculo
  • Multicast — TCP não suporta; UDP sim

Quando NÃO usar UDP

Tudo que precisa de garantia: HTTP, SSH, e-mail, banco de dados, transferência de arquivo. Pra esses casos, TCP é o caminho.

TCP — confiável, ordenado e com controle +
0:00 / 0:00

TCP (Transmission Control Protocol) é o oposto de UDP. Connection-oriented: estabelece uma sessão, garante entrega, mantém ordem, controla fluxo, retransmite o que perdeu. Você manda 1MB e tem certeza que o outro lado recebeu 1MB completo, na ordem certa, mesmo que internamente isso tenha custado 700 pacotes e 5 retransmissões.

O que TCP garante (que UDP não dá)

  • Entrega — pacote perdido é retransmitido até chegar (ou cair a conexão por timeout)
  • Ordem — pacotes que chegaram fora de ordem são reordenados antes de entregar pra aplicação
  • Sem duplicação — se o mesmo pacote chega 2 vezes, descarta a cópia
  • Controle de fluxo — o receptor diz quanto consegue receber por vez (sliding window); o emissor respeita
  • Controle de congestionamento — se a rede está congestionada, TCP reduz velocidade automaticamente

Header maior — 20 bytes mínimo

Pra dar tudo isso, TCP precisa de mais espaço de cabeçalho:

  • Porta origem e destino (4 bytes)
  • Sequence number (4 bytes) — qual byte do stream esse pacote começa
  • Acknowledgment number (4 bytes) — até qual byte o receptor já recebeu
  • Flags (SYN, ACK, FIN, RST, PSH, URG)
  • Window size — quanto buffer livre o receptor tem
  • Checksum + opções

Como TCP retransmite

Cada byte enviado tem um número de sequência. O receptor confirma com ACK. Se o emissor mandou bytes 1-100 e só recebeu ACK até 50, ele retransmite 51-100. Se nenhum ACK chega em X segundos, retransmite tudo desde o último ACK.

Quando usar TCP

Sempre que entrega importa mais que latência:

  • HTTP / HTTPS (todo navegador)
  • SSH, SFTP
  • SMTP, IMAP, POP3 (e-mail)
  • Banco de dados (PostgreSQL, MySQL, Redis em modo TCP)
  • Transferência de arquivo (FTP, rsync, Git)
  • Praticamente toda aplicação web tradicional

Custo da confiabilidade

TCP precisa de handshake antes (3 mensagens) e handshake de fechamento (4 mensagens). Em conexão nova, isso é 1 RTT a mais antes do primeiro byte de dado. Pra resposta curta tipo DNS, esse custo dobra ou triplica o tempo total — daí UDP fazer mais sentido.

Portas — endereçando processos dentro da máquina +
0:00 / 0:00

Cada máquina pode ter dezenas de processos rodando ao mesmo tempo: Tomcat na 8080, Postgres na 5432, Redis na 6379, SSH na 22. O IP entrega o pacote na máquina certa. Quem distingue qual processo entrega é a porta.

16 bits — 65.536 portas

Porta é um número de 16 bits, então vai de 0 a 65535. Aplicado tanto em TCP quanto em UDP — o mesmo número 5432 pode estar em TCP/5432 (Postgres) E UDP/5432 (outro serviço) sem conflito; são protocolos diferentes.

3 faixas oficiais (IANA)

FaixaNomeUso
0 – 1023Well-knownReservadas pra serviços padrão (HTTP=80, HTTPS=443, SSH=22). No Linux/Mac precisa de root pra abrir
1024 – 49151RegisteredServiços comuns mas não tão fundamentais (Postgres=5432, MySQL=3306, Tomcat=8080)
49152 – 65535Dynamic / ephemeralCliente que conecta pega uma daqui automaticamente. Você nunca configura essa

O conceito de socket

Uma conexão TCP é identificada por uma tupla de 5 elementos:

(IP origem, porta origem, IP destino, porta destino, protocolo)

Exemplo: você abre o navegador e acessa Google:

(192.168.0.42, 54321, 142.250.78.46, 443, TCP)
  • 192.168.0.42 — sua máquina
  • 54321 — porta dinâmica que o sistema operacional atribuiu pra esse cliente
  • 142.250.78.46 — Google
  • 443 — HTTPS (porta fixa do servidor)
  • TCP — protocolo

Outras conexões pro mesmo Google têm portas dinâmicas diferentes (54322, 54323...) — é assim que o sistema operacional sabe qual resposta entregar pra qual aba do navegador.

Server bind vs client connect

  • Server — escolhe uma porta fixa e escuta nela (Tomcat sempre em 8080)
  • Client — pega uma porta dinâmica automática e conecta pra porta fixa do server

Por isso "porta ocupada" é erro só de servidor — cliente nunca tem esse problema.

Three-way handshake — como o TCP abre uma conexão +
0:00 / 0:00

Antes de qualquer byte de dado fluir entre cliente e servidor TCP, eles trocam 3 mensagens pra estabelecer uma conexão. Esse é o three-way handshake.

O fluxo passo a passo

CLIENTE                         SERVIDOR
   |                                |
   |-------- 1) SYN -------------->|     "Quero conectar. Meu seq inicial é X"
   |                                |
   |<------- 2) SYN-ACK -----------|     "Aceito. Meu seq inicial é Y. Confirmo seu X+1"
   |                                |
   |-------- 3) ACK -------------->|     "Confirmo seu Y+1"
   |                                |
   |======== CONEXÃO ABERTA =======|
   |                                |
   |-------- dados ---------------->|     primeiro byte de dado real

Por que 3 mensagens (e não 2)

Os 2 primeiros pacotes (SYN do cliente, SYN-ACK do servidor) já estabeleceriam que cliente quer conectar e servidor topa. Mas falta o cliente confirmar que recebeu a aceitação do servidor. Sem o terceiro ACK, o servidor não sabe se a resposta dele chegou de fato. Os 3 passos garantem que os 2 lados sabem que os 2 lados estão prontos.

Estados da conexão (visíveis em ss / netstat)

EstadoSignificado
LISTENServidor está escutando, esperando conexão
SYN_SENTCliente mandou SYN, esperando SYN-ACK
SYN_RECVServidor recebeu SYN, mandou SYN-ACK, esperando ACK
ESTABLISHEDConexão aberta, dados podem fluir
FIN_WAIT_1 / FIN_WAIT_2Lado pediu pra fechar, aguardando confirmação
CLOSE_WAITRecebeu pedido de fechamento, mas ainda não fechou do seu lado
TIME_WAITConexão fechada, aguardando timeout pra liberar a porta de vez

Four-way handshake — fechando a conexão

Pra fechar, são 4 mensagens. TCP é full-duplex (cada lado pode mandar dado independentemente), então cada lado precisa fechar separado:

A                         B
|------- FIN ------------>|    A diz "terminei de mandar"
|<------ ACK -------------|    B confirma
|<------ FIN -------------|    B também terminou
|------- ACK ------------>|    A confirma. Conexão totalmente fechada.

O famoso TIME_WAIT

Depois do último ACK, o lado que iniciou o fechamento entra em TIME_WAIT por ~60 segundos (default Linux). Serve pra garantir que pacotes atrasados ainda no caminho não confundam uma conexão futura na mesma porta.

Sintoma clássico: servidor reiniciado fica "Address already in use" por 1 minuto. É a porta presa em TIME_WAIT. Soluções: SO_REUSEADDR no socket, ou esperar.

Custo do handshake — por que UDP existe

O handshake completo custa 1 RTT (round-trip time) antes do primeiro byte de dado. Pra ping de 50ms, são 50ms só de overhead. Pra request curto (DNS pede 1 nome, recebe 1 IP), esse 50ms duplica o tempo total. Por isso DNS usa UDP — sem handshake, manda pergunta direto.

Ferramentas de diagnóstico — camada 4 na prática +
0:00 / 0:00

Subindo o roteiro depois de camada 1, 2 e 3 — agora as ferramentas pra investigar conexões TCP e portas.

ss — quem está escutando, quem está conectado

# Listar todas as portas TCP em LISTEN
$ ss -tln
State    Recv-Q    Send-Q    Local Address:Port
LISTEN   0         128       0.0.0.0:22
LISTEN   0         128       127.0.0.1:5432
LISTEN   0         100       0.0.0.0:8080

Lê: SSH escutando em 22 em qualquer interface, Postgres em 5432 só localhost (seguro), Tomcat em 8080 em qualquer interface.

# Ver conexões TCP estabelecidas
$ ss -tn
ESTAB    192.168.0.42:54321    142.250.78.46:443
ESTAB    192.168.0.42:54322    142.250.78.46:443

netstat — versão antiga (depreciado mas ainda comum)

$ netstat -tlnp
tcp   0   0 0.0.0.0:22       0.0.0.0:*   LISTEN   1234/sshd
tcp   0   0 127.0.0.1:5432   0.0.0.0:*   LISTEN   2345/postgres
tcp   0   0 0.0.0.0:8080     0.0.0.0:*   LISTEN   3456/java

O -p mostra também qual processo (PID e nome). Útil pra descobrir "quem está usando a porta 8080".

lsof -i — quem usa qual porta (Linux/Mac)

$ sudo lsof -i :8080
COMMAND  PID  USER   FD   TYPE  NODE NAME
java     3456 flavio 42u  IPv6  TCP  *:8080 (LISTEN)

Resposta direta: porta 8080 está com o processo java (PID 3456). Pra matar: kill 3456.

telnet host porta — testar se servidor está vivo

$ telnet google.com 443
Trying 142.250.78.46...
Connected to google.com.
Escape character is '^]'.

"Connected" = TCP fechou handshake. O servidor existe, escuta na porta, e o caminho está aberto. Pra fechar: Ctrl+] seguido de quit.

$ telnet servidor.empresa.com 5432
Trying ...
telnet: Unable to connect to remote host: Connection refused

"Connection refused" = a máquina existe mas a porta está fechada (servidor caiu, ou firewall bloqueou).

nc / netcat — canivete suíço

# Testar porta (estilo telnet, mais limpo)
$ nc -zv google.com 443
Connection to google.com 443 port [tcp/https] succeeded!

# Criar listener (servidor improvisado pra teste)
$ nc -l 9999    # outra máquina pode conectar com nc IP 9999

nmap — scanner de portas

$ nmap -p 1-10000 192.168.0.1
PORT     STATE  SERVICE
22/tcp   open   ssh
80/tcp   open   http
443/tcp  open   https
8080/tcp open   http-proxy

Lista quais portas estão abertas no host. Útil pra inventário de serviço, e pra audit de segurança ("o que estou expondo?").

  2. Camada 2: ip link show — placa UP?

"Tomcat não responde" → primeiro ss -tlnp | grep 8080 pra ver se está mesmo escutando. Se não estiver, problema é de inicialização. Se estiver, próximo passo é firewall ou caminho de rede.
