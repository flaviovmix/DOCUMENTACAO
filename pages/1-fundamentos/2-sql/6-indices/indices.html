Índices — o atalho que o banco usa pra achar dados

0:00 / 0:00

Imagine um livro de 800 páginas sem índice no fim. Pra achar onde fala de "fotossíntese", você teria que folhear página por página, do começo ao fim. Com o índice atrás, você vai direto: "fotossíntese, página 412". O índice de um banco de dados é exatamente isso. Sem ele, o banco lê a tabela inteira, linha por linha, toda vez que você procura algo. Com ele, vai direto na linha certa.

O problema: ler a tabela inteira

Quando você roda uma consulta com filtro, tipo:

SELECT * FROM users WHERE email = 'ana@email.com';

sem índice, o banco faz o que se chama de sequential scan (varredura sequencial): ele pega a primeira linha, confere o email, pega a segunda, confere, e assim por diante até o fim da tabela. Ele não tem como saber onde a Ana está sem olhar todas.

Com 10 linhas, isso é instantâneo. Com 10 milhões, é uma consulta que trava a tela. O problema não aparece no começo do projeto; ele aparece quando a tabela cresce. Por isso índice é um assunto que parece teórico até o dia em que a aplicação fica lenta e ninguém sabe por quê.

O que é um índice, de fato

Um índice é uma estrutura de dados separada que o banco mantém ao lado da tabela. Ela guarda os valores de uma coluna já ordenados, cada um apontando pra localização da linha correspondente na tabela.

Voltando ao livro: a tabela é o miolo (o conteúdo na ordem em que foi escrito); o índice é a lista alfabética no fim, com "termo → página". O banco consulta a lista ordenada (que é rápida de buscar porque está em ordem), descobre o "endereço" da linha, e pula direto pra ela.

O tipo de índice padrão do PostgreSQL é o B-tree (árvore balanceada). Você não precisa entender a estrutura por dentro agora; só precisa saber que ela mantém os valores ordenados de um jeito que torna a busca, a comparação por maior/menor, e a ordenação muito rápidas. 99% dos índices que você vai criar são B-tree, e é o tipo que o banco usa quando você não diz nada.

Como se cria um índice

É um comando só:

CREATE INDEX idx_users_email ON users (email);

Lendo: cria um índice chamado idx_users_email, na tabela users, sobre a coluna email. O nome é livre, mas a convenção comum é idx_ + tabela + coluna, pra ficar óbvio do que se trata. Depois disso, aquela consulta da Ana lá de cima passa a usar o índice automaticamente; você não muda o SELECT, o banco decide sozinho usar o atalho.

Uma coisa importante: chave primária já vem indexada de graça. Quando você declara uma coluna como PRIMARY KEY, o PostgreSQL cria o índice dela sozinho. O trabalho de criar índice na mão é pras outras colunas que você filtra muito.

Onde o índice ajuda de verdade

Índice acelera três situações principais:

  • Filtro (WHERE) — qualquer coluna que aparece muito depois de WHERE é candidata. Email, CPF, status, data.
  • Junção (JOIN) — quando você liga duas tabelas por uma coluna (a chave estrangeira), indexar essa coluna deixa o JOIN rápido. Essa é a mais esquecida: a chave estrangeira não ganha índice automático no PostgreSQL, você precisa criar.
  • Ordenação (ORDER BY) — como o índice já guarda os valores ordenados, o banco pode usá-lo pra entregar o resultado na ordem certa sem ter que ordenar tudo na hora.

Tem também o índice composto, que cobre mais de uma coluna ao mesmo tempo:

CREATE INDEX idx_tasks_user_due ON tasks (user_id, due_date);

Esse serve pra consultas que filtram por usuário e data juntos ("minhas tarefas com vencimento em junho"). A ordem das colunas importa: um índice em (user_id, due_date) ajuda quem filtra por user_id sozinho ou pelos dois juntos, mas não ajuda quem filtra só por due_date. Pensa numa lista telefônica ordenada por sobrenome e depois nome: ótima pra achar "Silva, Ana", inútil pra achar todo mundo que se chama "Ana".

O preço: índice não é de graça

Se índice só trouxesse vantagem, o banco indexaria tudo sozinho. Mas tem um custo, e é por isso que você escolhe onde colocar:

CustoPor quê
Escrita mais lentaTodo INSERT, UPDATE e DELETE tem que atualizar a tabela e cada índice dela. Mais índices = escrita mais cara.
Espaço em discoO índice é uma cópia ordenada dos dados daquela coluna. Ocupa espaço próprio, que cresce junto com a tabela.
ManutençãoO banco precisa manter a estrutura organizada conforme os dados mudam.

A consequência prática: não saia indexando tudo. Índice é pra coluna que você lê e filtra muito. Tabela que recebe muita escrita e pouca leitura quer poucos índices. O equilíbrio é o ofício.

Quando índice não compensa

  • Tabela pequena — se ela cabe em poucas linhas, ler tudo já é instantâneo; o índice só atrapalha a escrita.
  • Coluna que você quase nunca filtra — índice que ninguém usa é só custo, sem benefício.
  • Coluna com pouca variedade — um campo booleano (sim/não) onde 95% das linhas são "sim" não se beneficia de índice comum, porque filtrar por "sim" ainda devolve quase a tabela inteira.

Tipos de índice no PostgreSQL

O B-tree resolve a grande maioria. Mas o PostgreSQL tem outros tipos pra casos específicos. O que mais aparece além do B-tree é o GIN, usado pra busca dentro de texto (full-text), arrays e JSON. Quando você quer "achar toda nota que contém a palavra fotossíntese em qualquer lugar do conteúdo", o B-tree não serve (ele indexa o valor inteiro, não palavras soltas); o GIN serve.

Outros detalhes úteis, ainda usando B-tree por baixo:

  • Índice único (UNIQUE) — além de acelerar, ele garante que não existam dois valores iguais. É assim que o banco impede dois usuários com o mesmo email.
  • Índice por expressão — você pode indexar o resultado de uma função, não só a coluna crua. Ex: indexar lower(name) pra que a busca funcione sem se importar com maiúscula/minúscula.
  • Índice parcial — indexa só as linhas que interessam (ex: só os pedidos com status "aberto"), economizando espaço.

Como saber se o índice está sendo usado

Você não precisa adivinhar. O comando EXPLAIN mostra o plano que o banco vai seguir pra rodar a consulta:

EXPLAIN SELECT * FROM users WHERE email = 'ana@email.com';

No resultado, você procura duas palavras:

  • Seq Scan — varredura sequencial, leu a tabela inteira. Se a tabela é grande, é sinal de que falta índice.
  • Index Scan — usou o índice, foi direto. É o que você quer ver.

Detalhe que confunde no início: às vezes o banco escolhe não usar o índice mesmo existindo, porque em tabela pequena ler tudo é mais rápido que consultar o índice e depois pular pra tabela. Isso é o banco sendo esperto, não um bug.

No Nexus, na prática

O Nexus (Spring Boot + PostgreSQL) cria índices junto das tabelas, dentro das próprias migrations. Alguns exemplos reais de lá mostram cada caso que vimos:

-- Simples: acelera login por email (e garante email unico via UNIQUE em outra parte)
CREATE INDEX idx_users_email ON users (email);

-- Composto + ordenacao: lista de notas do usuario, mais recentes primeiro
CREATE INDEX idx_notes_user_updated ON notes (user_id, updated_at DESC);

-- Unico + expressao: impede dois projetos com o mesmo nome (ignorando maiuscula)
CREATE UNIQUE INDEX uk_projects_user_lower_name ON projects (user_id, lower(name));

-- GIN: busca full-text dentro do titulo e conteudo das notas
CREATE INDEX idx_notes_search ON notes USING GIN ( ... to_tsvector('portuguese', ...) ... );

Repare o padrão: quase todo índice começa com user_id. É porque no Nexus tudo é filtrado por usuário (cada um vê só os seus dados), então user_id é a coluna mais filtrada do banco. Os índices seguem o jeito que a aplicação consulta. Essa é a lição prática: você não indexa por capricho, você indexa as colunas que as suas consultas realmente usam.

Resumo

  • Índice = estrutura separada e ordenada que aponta pra linha certa, igual o índice no fim de um livro.
  • Sem índice, o banco faz sequential scan (lê tudo); com índice, vai direto.
  • Cria com CREATE INDEX nome ON tabela (coluna); o tipo padrão é B-tree.
  • Ajuda em WHERE, JOIN e ORDER BY. Chave estrangeira não ganha índice automático, crie na mão.
  • Tem preço: escrita mais lenta e mais disco. Não indexe tudo; indexe o que você filtra muito.
  • GIN pra busca dentro de texto; UNIQUE também garante unicidade.
  • EXPLAIN mostra se a consulta usou Index Scan ou Seq Scan.
