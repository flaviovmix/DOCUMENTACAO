Visão Geral — Relacionamentos

0:00 / 0:00

Banco de dados relacional ganhou esse nome justamente porque as tabelas se conversam entre si. Uma tabela de pedido aponta pro cliente que fez o pedido. Uma tabela de produto aponta pra categoria a que pertence. Esse "apontar" é o que chamamos de relacionamento — e em SQL ele é feito com chave estrangeira.

Essa página cobre o panorama: o que é uma chave estrangeira, os 3 tipos de cardinalidade (1-1, 1-N, N-N), o papel da tabela associativa e as regras de ON DELETE/UPDATE. Os detalhes de cada tipo entram em páginas próprias da trilha.

Chave primária e chave estrangeira — o mecanismo +

Toda tabela tem uma chave primária (PK) — uma coluna (geralmente id) cujo valor é único pra cada linha. É o RG da linha.

Quando uma tabela quer apontar pra outra, ela guarda o valor da PK da outra numa coluna própria. Essa coluna se chama chave estrangeira (FK, foreign key).

-- tabela referenciada (lado "1")
CREATE TABLE categoria (
    id   SERIAL PRIMARY KEY,
    nome TEXT NOT NULL
);

-- tabela que aponta (lado "N")
CREATE TABLE produto (
    id           SERIAL PRIMARY KEY,
    nome         TEXT NOT NULL,
    categoria_id INTEGER REFERENCES categoria(id)
);

A linha REFERENCES categoria(id) é o que cria a FK. A partir desse momento o banco passa a garantir integridade: não dá pra inserir um produto com categoria_id = 999 se não existir uma categoria com id = 999. O banco bloqueia.

Esse é o ponto principal: relacionamento em SQL não é uma convenção de programador, é uma regra que o próprio banco força.

Cardinalidade — quantos de um lado, quantos do outro +

Quando duas tabelas se conectam, a primeira pergunta é: quantas linhas de um lado podem se ligar a quantas do outro? Essa resposta é a cardinalidade.

Existem 3 cardinalidades em SQL:

TipoLeituraExemplo do dia a dia
1-1Uma linha de A liga em no máximo uma linha de BPessoa ↔ Passaporte
1-NUma linha de A liga em várias linhas de BCategoria ↔ Produtos
N-NVárias linhas de A ligam em várias linhas de BAluno ↔ Disciplinas

O número antes do hífen é o lado A, depois é o lado B. 1-N é de longe o tipo mais comum num sistema real — ~80% dos relacionamentos. N-N aparece sempre que existe uma "matrícula", "associação" ou "vínculo" entre dois cadastros independentes.

Um para um (1-1) +

Cada linha de A se liga a no máximo uma linha de B, e vice-versa. É o tipo mais raro. Aparece principalmente em 2 cenários:

  1. Separar dados sensíveis ou opcionais da tabela principal (ex: usuario + usuario_perfil com bio, foto, link de redes sociais).
  2. Especialização — uma tabela genérica + uma tabela com colunas extras de um subtipo (ex: pessoa + pessoa_juridica com CNPJ).
CREATE TABLE usuario (
    id    SERIAL PRIMARY KEY,
    email TEXT UNIQUE NOT NULL
);

CREATE TABLE usuario_perfil (
    usuario_id INTEGER PRIMARY KEY REFERENCES usuario(id),
    bio        TEXT,
    foto_url   TEXT
);

Repare no truque: a coluna usuario_id é PK e FK ao mesmo tempo. Como PK ela garante que não existem 2 perfis pro mesmo usuário (limita o lado B a 1). Como FK ela aponta pro usuário.

Na prática, 1-1 quase sempre poderia ter sido uma só tabela com mais colunas. Existe quando faz sentido separar fisicamente — por desempenho, por permissão, ou por organização.

Um para muitos (1-N) +

O tipo mais comum. Uma linha do lado "1" pode aparecer em várias linhas do lado "N". O exemplo clássico é categoria de produto: uma categoria tem vários produtos, mas cada produto pertence a uma categoria só.

CREATE TABLE categoria (
    id   SERIAL PRIMARY KEY,
    nome TEXT NOT NULL
);

CREATE TABLE produto (
    id           SERIAL PRIMARY KEY,
    nome         TEXT NOT NULL,
    preco        NUMERIC(10,2),
    categoria_id INTEGER NOT NULL REFERENCES categoria(id)
);

A FK fica sempre no lado "N". Foi assim que o banco entende que o produto é o lado "muitos" e a categoria é o lado "um" — porque o produto carrega o ponteiro, não a categoria. Se a categoria carregasse o ponteiro, ela só conseguiria apontar pra um produto.

Cenário do dia a diaLado "1"Lado "N"
Pedido e itenspedidoitem_pedido
Cliente e endereçosclienteendereco
Post e comentáriospostcomentario
Departamento e funcionáriosdepartamentofuncionario
Muitos para muitos (N-N) — a tabela associativa +

Quando os dois lados podem se ligar a vários do outro lado, FK simples não resolve. Exemplo: um aluno cursa várias disciplinas, e cada disciplina tem vários alunos. Se você colocasse aluno_id em disciplina só caberia um aluno; se colocasse disciplina_id em aluno só caberia uma disciplina.

A solução é uma terceira tabela — chamada de tabela associativa ou tabela de junção — que guarda os pares.

CREATE TABLE aluno (
    id   SERIAL PRIMARY KEY,
    nome TEXT NOT NULL
);

CREATE TABLE disciplina (
    id   SERIAL PRIMARY KEY,
    nome TEXT NOT NULL
);

-- a tabela associativa
CREATE TABLE matricula (
    aluno_id      INTEGER NOT NULL REFERENCES aluno(id),
    disciplina_id INTEGER NOT NULL REFERENCES disciplina(id),
    PRIMARY KEY (aluno_id, disciplina_id)
);

Cada linha da matricula representa um vínculo entre um aluno e uma disciplina. A PK composta (aluno_id, disciplina_id) garante que o mesmo par não aparece duas vezes — não dá pra matricular o mesmo aluno na mesma disciplina duas vezes.

Olhando assim, a N-N é só um par de 1-N: aluno tem várias matrículas (1-N), disciplina tem várias matrículas (1-N), e a matrícula é o ponto onde os dois se encontram.

Tabela associativa pode ter colunas próprias

Não precisa ser só aluno_id + disciplina_id. Quando o vínculo carrega informação própria, a tabela ganha colunas extras:

CREATE TABLE matricula (
    aluno_id      INTEGER NOT NULL REFERENCES aluno(id),
    disciplina_id INTEGER NOT NULL REFERENCES disciplina(id),
    nota          NUMERIC(4,2),
    data_matricula DATE NOT NULL DEFAULT CURRENT_DATE,
    PRIMARY KEY (aluno_id, disciplina_id)
);

nota e data_matricula não pertencem nem ao aluno nem à disciplina — pertencem ao vínculo. É justamente quando você precisa guardar uma propriedade do vínculo que a tabela associativa deixa de ser "técnica" e vira parte importante do modelo.

ON DELETE e ON UPDATE — o que acontece quando o pai some +

Toda FK guarda uma referência. Pergunta natural: e se a linha referenciada for apagada? Se a categoria com id = 5 for deletada, o que acontece com os produtos que apontavam pra ela?

Esse comportamento é definido na FK com as cláusulas ON DELETE e ON UPDATE. As 4 opções principais:

OpçãoO que faz quando o pai é apagado
RESTRICT (padrão)Bloqueia o DELETE. Não deixa apagar a categoria enquanto existir produto apontando pra ela.
CASCADEApaga em cascata. Deletou a categoria, todos os produtos dela vão junto.
SET NULLDeixa o filho órfão. categoria_id dos produtos vira NULL (a coluna precisa permitir nulo).
SET DEFAULTColoca o valor default da coluna no lugar (ex: categoria_id = 1 "Sem categoria").
CREATE TABLE produto (
    id           SERIAL PRIMARY KEY,
    nome         TEXT NOT NULL,
    categoria_id INTEGER REFERENCES categoria(id)
                 ON DELETE SET NULL
                 ON UPDATE CASCADE
);

A escolha não é técnica — é de regra de negócio. Apagar pedido leva os itens junto? CASCADE. Apagar cliente é proibido enquanto tiver pedido? RESTRICT. Apagar categoria deixa os produtos sem categoria? SET NULL.

O ON UPDATE é mais raro — só importa quando a PK muda de valor (o que quase nunca acontece se a PK é SERIAL). Aparece mais quando a PK é uma string editável tipo um código.

Resumo +
  • FK é uma coluna que guarda a PK de outra tabela. Cria a ponte entre duas tabelas e o banco força a integridade.
  • Cardinalidade é "quantos de um lado, quantos do outro": 1-1, 1-N, N-N.
  • 1-N é o mais comum. FK fica sempre no lado "N".
  • N-N precisa de uma tabela associativa (matrícula, vínculo, item_pedido). Pode ter colunas próprias quando o vínculo carrega informação.
  • ON DELETE / ON UPDATE controlam o que acontece quando o pai some — RESTRICT bloqueia, CASCADE apaga junto, SET NULL deixa órfão.
