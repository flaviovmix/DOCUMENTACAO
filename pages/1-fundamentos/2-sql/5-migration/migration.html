Migration — schema versionado como código

0:00 / 0:00

Seu código tem Git: dá pra ver cada mudança, quem fez, em que ordem, e voltar atrás. O banco de dados, por padrão, não tem nada disso. Você abre o cliente, roda um ALTER TABLE na mão, e seis meses depois ninguém sabe mais o que foi feito nem em que ordem. Migration é o que dá ao schema do banco o mesmo histórico que o código já tem.

O que é uma migration

Migration é uma mudança no schema (criar tabela, adicionar coluna, criar índice) escrita como um arquivo de script versionado, guardado junto com o código no Git.

Em vez de mexer no banco pela interface gráfica, cada alteração vira um arquivo numerado. O banco passa a ter um histórico igual ao do código: uma sequência de passos, em ordem, que qualquer máquina consegue repetir.

O problema que isso resolve: schema evolui com o tempo, e você precisa que toda máquina (seu dev, o de outro dev, o servidor de produção) chegue exatamente no mesmo estado, na mesma ordem, sem ninguém esquecer um passo.

Flyway — quem aplica as migrations

O Flyway é a ferramenta que roda essas migrations automaticamente. No mundo Java/Spring é o padrão mais comum (o concorrente direto é o Liquibase). A regra dele é só uma convenção de nome de arquivo:

V{numero}__{descricao}.sql

Lendo V4__add_note_flags.sql da esquerda pra direita: o V é de versioned, o 4 é a ordem, e o __ (dois underscores) separa do nome legível. Quando a aplicação sobe, o Flyway:

  1. Olha a pasta de migrations e ordena os arquivos pelo número.
  2. Confere uma tabela de controle dele, a flyway_schema_history, pra ver o que já rodou.
  3. Aplica só os arquivos novos, em ordem, e grava cada um na tabela junto com um checksum do conteúdo.

O checksum é o detalhe importante: é uma impressão digital do arquivo. Se o conteúdo de uma migration já aplicada mudar, o checksum não bate mais, e o Flyway para com erro dizendo que o histórico foi adulterado.

A regra de ouro: append-only

Por causa do checksum, vale uma regra que não se quebra:

Migration já aplicada nunca se edita. Precisou corrigir? Cria uma migration nova.

É a mesma lógica de um commit que já foi pro repositório compartilhado: você não reescreve a história, você faz um commit por cima. Se o V37 criou uma coluna com o tipo errado, você não volta lá pra arrumar; você cria um V39 que conserta.

SituaçãoErradoCerto
Coluna com tipo errado numa migration já aplicadaEditar o arquivo antigoNova migration com ALTER ... TYPE ...
Renomear uma tabelaMexer no CREATE TABLE originalNova migration com ALTER TABLE ... RENAME
Migration ainda não commitada, só localAí pode editar à vontade — ela ainda não rodou em nenhuma outra máquina

Como é uma migration por dentro

Não tem mistério: é SQL puro, com um comentário explicando o porquê. Exemplo de uma migration que adiciona duas colunas booleanas numa tabela existente:

-- V4: flags de organizacao na nota.
-- archived: nota escondida da lista principal sem deletar.
-- pinned:   nota fixada (ordenacao posterior usa esse campo).
-- Ambos default FALSE: notas existentes ficam nao-arquivadas, nao-fixadas.

ALTER TABLE notes
    ADD COLUMN archived BOOLEAN NOT NULL DEFAULT FALSE,
    ADD COLUMN pinned   BOOLEAN NOT NULL DEFAULT FALSE;

O DEFAULT FALSE aqui não é detalhe: a tabela já tem linhas. Adicionar uma coluna NOT NULL sem default quebraria, porque as linhas antigas não teriam valor. O default resolve as linhas que já existem. Pensar nos dados que já estão lá é metade do trabalho de escrever migration.

Quem é dono do schema: Flyway, não o ORM

Frameworks como o Hibernate (o ORM do Java) conseguem criar e alterar tabelas sozinhos a partir das classes. Isso é cômodo em protótipo, mas perigoso em projeto real: você perde o controle e o histórico. A prática madura é desligar essa autonomia e deixar o schema 100% nas mãos do Flyway.

No Spring isso é uma linha de configuração, ddl-auto: validate:

jpa.hibernate.ddl-auto: validate   # Hibernate so CONFERE, nao cria nem altera

Com validate, quando a aplicação sobe o Hibernate apenas confere se as classes Java batem com as tabelas que o Flyway criou. Se não baterem, ele reclama. A ordem na subida é: Flyway aplica as migrations pendentes primeiro, depois o Hibernate valida. Quem cria tabela é só o Flyway.

No Nexus, na prática

O Nexus (projeto Spring Boot + PostgreSQL) usa Flyway exatamente assim. A configuração liga o Flyway, aponta a pasta dos scripts e diz que o banco começa do zero (sem baseline pré-existente):

flyway:
  enabled: true
  locations: classpath:db/migration   # onde ficam os V*.sql
  baseline-on-migrate: false          # banco nasce do V1, sem baseline anterior

A pasta db/migration/ tem a história inteira do produto, de V1 até V51 e além. Dá pra ler a evolução do Nexus só passando o olho nos nomes dos arquivos, em ordem:

MigrationO que entrou
V1__init.sqlBaseline: só uma tabela-marca, nada de negócio ainda
V2__create_users.sqlUsuários
V3__create_notes.sqlNotas
V4__add_note_flags.sqlFlags de arquivado/fixado nas notas
......
V13__rename_notes_to_pages.sqlRenomeação grande (notas viraram páginas)
V35__create_finance_config.sqlInício do módulo de finanças
V41__create_project_pages.sqlProjetos como hub

Cada feature do roadmap deixa rastro como uma faixa de migrations. O módulo de finanças, por exemplo, é a faixa V35 a V40. A pasta funciona como um changelog do banco que ninguém precisou escrever à mão: ela é o registro real do que aconteceu.

E o append-only aparece de verdade lá: existe um V39__fix_recorrencia_dia_do_mes_type.sql. Ele existe porque uma migration anterior criou uma coluna com o tipo errado. Não voltaram no arquivo antigo pra corrigir; criaram um passo novo, o "fix", e seguiram em frente.

Resumo

  • Migration = mudança de schema escrita como arquivo versionado, no Git junto com o código.
  • Flyway aplica os arquivos V{N}__nome.sql em ordem, uma vez cada, e registra na flyway_schema_history com checksum.
  • Append-only: migration aplicada nunca se edita; correção é sempre uma migration nova.
  • O ORM não manda no schema (ddl-auto: validate): quem cria tabela é o Flyway, o Hibernate só confere.
  • No Nexus, a pasta db/migration/ (V1→V51+) é o histórico real do produto.
