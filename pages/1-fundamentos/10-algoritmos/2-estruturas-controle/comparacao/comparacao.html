Comparação: Vetor × Lista Ligada

0:00 / 0:00

As cinco aulas anteriores construíram as duas estruturas do zero: o array (posições contíguas, acesso por índice) e a lista ligada (nós soltos na memória, ligados por ponteiro). Esta última coloca as duas na mesma corrida: a classe Comparacao cria um ArrayList e a nossa ListaLigada, joga 10000 números em cada uma e cronometra duas operações — adicionar tudo e depois ler tudo por índice.

Adicionar é empate: as duas enganchain no fim em tempo constante. O abismo aparece na leitura. vetor.get(i) é um cálculo direto de endereço; lista.get(i) precisa caminhar do primeiro nó até a posição i. Ler a coleção inteira por índice faz a lista pagar um salto a mais a cada posição — e isso cresce ao quadrado.

A corrida de leitura

O mesmo laço de leitura roda nas duas estruturas, lado a lado. Repare no contador de saltos de cada pista: no vetor ele fica parado em zero (acesso direto), na lista ele dispara — cada get(i) custa i saltos pra chegar lá.

O experimento — adicionar e ler 10000 elementos +

A classe Comparacao coloca as duas estruturas no mesmo banco de provas. Cria um ArrayList do Java e a ListaLigada que a gente escreveu, e cronometra com System.currentTimeMillis() antes e depois de cada operação:

ListaLigada<Integer> lista = new ListaLigada<Integer>();
ArrayList<Integer> vetor = new ArrayList<Integer>();

int limite = 10000;
long tempoInicial = System.currentTimeMillis();
long tempoFinal;
for(int i=0; i < limite; i++){
    vetor.add(i);          // adiciona no vetor
}
tempoFinal = System.currentTimeMillis();
System.out.println("Adicionou "+limite+" elementos no vetor");
System.out.println(tempoFinal - tempoInicial);

São 4 medições: adicionar no vetor, adicionar na lista, ler o vetor inteiro, ler a lista inteira. tempoFinal - tempoInicial dá os milissegundos gastos em cada bloco. O resultado é o que separa as duas estruturas.

Adicionar — empate técnico (os dois são O(1)) +

Inserir no fim é barato nas duas. A ListaLigada guarda um ponteiro ultimo, então adicionar() é só "linka e move" — tempo constante, sem percorrer nada:

this.ultimo.setProximo(novoElemento);
this.ultimo = novoElemento;
this.tamanho++;

O ArrayList também adiciona no fim em tempo constante amortizado: na maioria das vezes só escreve na próxima posição livre. De vez em quando enche e precisa dobrar o array interno (copiar tudo pra um maior), mas isso é raro o bastante pra diluir. Conclusão: na fase de escrita, os tempos ficam parecidos. O empate é real.

Ler — o abismo: lista.get(i) caminha toda vez +

Aqui as estruturas se separam. vetor.get(i) calcula o endereço direto (início + i) e pega o valor: 1 operação, O(1), não importa o tamanho. Já o get(i) da lista recomeça do primeiro nó e dá i saltos pelo ponteiro proximo:

public Elemento get(int posicao){
    Elemento atual = this.primeiro;
    for(int i=0; i < posicao; i++){
        if (atual.getProximo() != null){
            atual = atual.getProximo();
        }
    }
    return atual;
}

get(0) não dá salto, get(1) dá 1, get(5000) dá 5000. Quando o laço da Comparacao lê todas as posições por índice, a conta de saltos é 0 + 1 + 2 + ... + 9999, que dá ≈ 50 milhões de hops. Isso é o famoso O(n²): dobrou o número de elementos, quadruplicou o trabalho.

Por que — memória contígua vs. nós espalhados +

A diferença não é "código mais esperto", é onde os dados moram na memória:

  • Array — as posições são contíguas. A posição i está em endereço_base + i × tamanho. O computador calcula isso numa conta e pula direto. Por isso é O(1).
  • Lista ligada — cada nó foi alocado separadamente, em endereços quaisquer. A única forma de achar a posição i é seguir a corrente de ponteiros do começo. Não tem atalho. Por isso é O(n).

A lição prática — qual usar, e como ler a lista certo +

Nenhuma estrutura "ganha" sempre — cada uma otimiza um uso:

  • Precisa ler por índice toda hora (acesso aleatório, posições conhecidas) → ArrayList.
  • Só insere/remove nas pontas e percorre em ordem → lista ligada (LinkedList de fábrica) brilha.

E tem uma pegadinha de uso: o que mata a lista não é ler, é ler por índice num laço. Se você percorre seguindo os próprios ponteiros (com for-each / iterator), cada passo é 1 salto só — a leitura inteira volta a ser O(n):

// ❌ O(n²) — cada get recomeça do início
for (int i = 0; i < lista.getTamanho(); i++) {
    lista.get(i);
}

// ✅ O(n) — segue o ponteiro de onde parou
for (Integer valor : lista) {
    // ...
}

O experimento da Comparacao usa de propósito o jeito errado pra deixar o abismo visível. Na vida real, é o índice que escolhe a estrutura.
