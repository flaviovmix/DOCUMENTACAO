Nas aulas anteriores a lista ligada só guardava String — o tipo estava chumbado no código. Mas a estrutura (nós, ponteiros, adicionar, remover) não tem nada a ver com o tipo do valor. Seria desperdício escrever uma ListaLigada nova pra cada tipo. A solução é deixar o tipo como uma variável: <TIPO>. Isso é Generics. Um molde só, e quem usa decide o tipo na hora de criar. O exemplo vem da aula 05 de Estruturas de Dados em Java (Leandro Guarino).

Antes, o nó guardava String fixo. Agora ele guarda um tipo a definir, escrito como <TIPO> ao lado do nome da classe. Esse TIPO é um parâmetro de tipo — funciona como uma variável, só que em vez de guardar um valor, guarda um tipo.

O nome TIPO é livre (a convenção Java usa letras únicas: T, E, K, V). O importante é que todo lugar que era String virou TIPO — e o compilador troca TIPO pelo tipo real no momento em que você cria a lista.

Um molde, três tipos — o reaproveitamento +

Com a lista genérica, o mesmo código serve qualquer tipo. Quem cria a lista escolhe escrevendo o tipo entre < >:

ListaLigada<String>  estados  = new ListaLigada<String>();
ListaLigada<Integer> numeros  = new ListaLigada<Integer>();
ListaLigada<Cliente> clientes = new ListaLigada<Cliente>();

A partir daí o compilador amarra o tipo: estados.adicionar(3) nem compila — ele só aceita String. É a vantagem central de generics sobre guardar tudo como Object: segurança de tipo em tempo de compilação, sem precisar de cast na hora de ler.

Na aula são três main separados (Programa, ProgramaInteiros, ProgramaClientes), cada um exercitando a mesma ListaLigada com um tipo diferente.

equals() em vez de equalsIgnoreCase() +

No remover() da aula 04, a comparação era atual.getValor().equalsIgnoreCase(valor). Mas equalsIgnoreCase() só existe em String — agora o valor é TIPO, pode ser Integer ou Cliente, que não têm esse método.

A troca é por equals(), que todo objeto Java tem (vem de Object):

// antes (só String):
if (atual.getValor().equalsIgnoreCase(valorProcurado)) { ... }

// agora (qualquer tipo):
if (atual.getValor().equals(valorProcurado)) { ... }

O preço é que equals() compara exato — sem ignorar maiúsculas. E pra Cliente, o equals() padrão compara por identidade (mesmo objeto na memória), não por CPF. Se você quisesse "dois clientes com mesmo CPF são iguais", teria que sobrescrever equals() no Cliente — mas isso a aula não faz.

toString() no Cliente — por que Cliente@1b6d +

Quando você faz System.out.println(cliente), o Java chama o toString() do objeto pra virar texto. O toString() padrão (herdado de Object) devolve algo inútil: o nome da classe + um código hexadecimal da memória, tipo Cliente@1b6d3586.

Pra sair o nome do cliente, a classe sobrescreve toString():

public class Cliente {
    private String cpf;
    private String nome;
    // ...
    public String toString() {
        return this.nome;   // sem isso, imprimiria "Cliente@1b6d3586"
    }
}

Com String e Integer esse problema não aparece, porque essas classes já trazem um toString() útil de fábrica. O toString() só vira assunto quando você guarda seus próprios objetos na lista. Use o botão com / sem toString() na aba Cliente acima pra ver a diferença.
