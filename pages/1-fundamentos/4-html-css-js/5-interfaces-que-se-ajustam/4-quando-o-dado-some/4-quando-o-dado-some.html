Quando o Dado Some

0:00 / 0:00

Os três problemas anteriores foram sobre o visual: o desenho que corta, a caixa que preenche o espaço, a decisão que precisa medir. Esse troca de assunto: é sobre o dado que some, mesmo existindo. O sintoma foi um painel mostrando dois registros quando havia vários. A causa quase sempre são duas coisas se somando.

O sintoma — tenho vários, só aparecem dois +

Um painel de lançamentos mostrava dois itens. Mas existiam vários no banco. A pergunta foi direta: por que só dois? Esse tipo de sumiço costuma ter causa boba, e quase sempre são dois fatores juntos: o que é pedido e o que dispara o pedido.

O filtro implícito de escopo +

A primeira causa: um filtro escondido. A tela passava pro componente o mês atual. O componente, por sua vez, pedia ao servidor só os lançamentos daquele mês. Os outros existiam, mas nunca foram pedidos. Não é que sumiram, é que nunca foram buscados.

// a tela passa o mes atual...
<Lancamentos mes={mesAtual()} />

// ...e o componente busca SO aquele mes
api.listLancamentos({ mes }); // os outros meses nunca chegam

A lição é a primeira coisa a desconfiar quando um dado some: o escopo da consulta. Muitas vezes tem um parâmetro, um filtro de data, um "só o ativo", que você esqueceu que estava ali. Antes de achar bug grande, confira o que você está de fato pedindo.

O gatilho que nunca acontece +

A segunda causa é mais sutil. O componente tinha scroll infinito. A ideia era boa: conforme você rola pro fim, ele carrega o mês anterior, depois o anterior, e vai puxando o histórico. Mas tinha uma condição escondida: isso só dispara quando a lista transborda e dá pra rolar.

function onScroll(el) {
    // so carrega mais se faltam menos de 80px pro fim
    if (el.scrollHeight - el.scrollTop - el.clientHeight < 80) {
        carregarProximoMes();
    }
}
// com poucos itens nao ha rolagem -> onScroll nunca dispara

Com poucos itens não havia rolagem, então o gatilho nunca era acionado, e os meses antigos nunca chegavam. Dois filtros se somando: o de mês na consulta, e a condição de carregamento que dependia de um evento que não acontecia.

A correção e o trade-off +

A saída foi pedir todos os meses de uma vez e agrupar no próprio cliente, em vez de depender do scroll pra ir buscando aos poucos. Funciona, mostra tudo.

// antes: mes atual + scroll pra carregar o resto
<Lancamentos mes={mesAtual()} />

// depois: pede tudo, agrupa por mes no cliente
<Lancamentos mes="" />

Mas toda mudança de escopo tem efeito colateral, e vale dizer em voz alta. As contagens que apareciam, tipo lançados e pendentes, antes eram do mês. Agora viram totais de tudo. Mudou o filtro, mudou o significado dos números. Não é errado, mas é uma decisão, melhor tomar consciente do que ser pego de surpresa.
