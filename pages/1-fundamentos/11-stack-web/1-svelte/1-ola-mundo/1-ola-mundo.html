Olá Mundo no Svelte

0:00 / 0:00

O primeiro passo prático da trilha. A ideia não é fazer nada útil: é o agente montando o próprio front e provando que o ambiente liga. Ele cria as peças, conecta umas nas outras e testa com a frase mais simples que existe na tela: "Olá Mundo". Depois que isso aparece, ele sabe que a oficina está de pé e pode trabalhar em cima com confiança.

Por que começar pelo Olá Mundo

Quando você monta uma tecnologia nova, um monte de coisa pode dar errado antes mesmo de escrever lógica: instalação incompleta, versão errada, projeto montado torto, comando que não roda. O Olá Mundo separa esses problemas da lógica. É como dar a partida no carro antes de viajar: você não vai a lugar nenhum, só confirma que o motor pega.

Aqui o agente está construindo um espaço pra ele mesmo trabalhar depois. O Olá Mundo é o teste de que esse espaço funciona.

O scaffold — montar o esqueleto

Antes de escrever qualquer código, precisa existir um projeto. O agente não cria os arquivos um por um na mão: ele roda um comando que monta a estrutura inteira de uma vez — pastas, arquivos de configuração, dependências, um exemplo mínimo que já roda. Esse esqueleto inicial chama scaffold.

No Svelte, esse comando entrega um projeto SvelteKit com tudo no lugar. É como receber uma casa com as paredes e o encanamento prontos: o agente só decora por dentro. Montar o projeto é um passo separado de escrever a lógica.

Duas categorias de peça

O scaffold cria muita coisa. Vale separar em dois grupos, porque o agente trata cada um de um jeito:

package.json — a lista de compras

É o manifesto do projeto. Diz o nome, declara que o projeto usa módulos modernos, lista os scripts (atalhos de comando) e as dependências — tudo que precisa ser baixado.

{
  "name": "svelte",
  "type": "module",
  "scripts": {
    "dev": "vite dev",
    "build": "vite build",
    "preview": "vite preview"
  },
  "devDependencies": {
    "@sveltejs/kit": "^2.57.0",
    "svelte": "^5.55.2",
    "vite": "^8.0.7"
  }
}

Quando o agente manda instalar, é esse arquivo que ele lê pra saber o que buscar. O resultado da instalação vai todo pra node_modules/. O script dev é o que liga o servidor de desenvolvimento.

node_modules — a infra que só fica lá

É uma pasta gigante com centenas de pacotes, gerada a partir do package.json. O agente nunca abre nem edita nada ali. São as peças que as ferramentas precisam pra funcionar por baixo. Obrigatória, mas invisível: nem entra no controle de versão (fica no .gitignore), porque dá pra reconstruir a qualquer momento só rodando a instalação de novo.

A pasta .svelte-kit/ é da mesma família: o SvelteKit a gera sozinho com tipos e rotas. Não se mexe nela à mão.

svelte.config.js e vite.config.ts

São os dois arquivos que conectam as ferramentas. O agente quase não toca neles, mas vale saber o que fazem:

// svelte.config.js — config do SvelteKit
import adapter from '@sveltejs/adapter-auto';

const config = {
    compilerOptions: { runes: true },   // liga a reatividade nova do Svelte 5
    kit: { adapter: adapter() }          // decide sozinho onde rodar em produção
};
export default config;
// vite.config.ts — config do Vite (servidor de dev + empacotador)
import { sveltekit } from '@sveltejs/kit/vite';
import { defineConfig } from 'vite';

export default defineConfig({
    plugins: [sveltekit()]   // pluga o SvelteKit dentro do Vite
});

O Vite é a ferramenta que liga o servidor de desenvolvimento e, depois, empacota tudo pra produção. O adapter é o que diz pra onde o projeto vai quando sair do computador do agente.

app.html — a casca

É a única página HTML de verdade do projeto. Ela tem dois buracos que o SvelteKit preenche: um no cabeçalho e um no corpo. Toda tela que o agente montar passa por essa casca.

<!doctype html>
<html lang="en">
    <head>
        <meta charset="utf-8" />
        %sveltekit.head%
    </head>
    <body>
        <div style="display: contents">%sveltekit.body%</div>
    </body>
</html>

%sveltekit.head% e %sveltekit.body% não são HTML normal: são marcadores. O SvelteKit troca eles pelo conteúdo real na hora de servir a página.

src/routes — a pasta que vira URL

No SvelteKit, a pasta src/routes/ é o roteador. Cada subpasta vira um endereço no navegador. Arquivos que começam com + têm papel especial. Os dois que importam agora:

  • +layout.svelte — o envelope que embrulha todas as páginas. Põe o favicon e renderiza o conteúdo de cada página com {@render children()}.
  • +page.svelte — a página da rota raiz (/). Esse é o arquivo que o agente edita pra mostrar o Olá Mundo.
<!-- src/routes/+page.svelte -->
<h1>Olá Mundo</h1>

Uma linha. O resto do projeto inteiro existe pra essa linha aparecer na tela do jeito certo.

Rodar e ver na tela

O agente roda o script dev. O servidor de desenvolvimento fica de pé e devolve um endereço local, normalmente na porta 5173. Ele abre esse endereço no navegador e vê "Olá Mundo" escrito na página.

O que aconteceu: o navegador baixou o front e desenhou a frase. É isso que o front faz — pinta coisa na tela pra você ver e interagir. E tem um detalhe que torna o front agradável de estudar: enquanto o servidor de dev está rodando, se o agente muda o texto e salva, a tela atualiza sozinha, na hora.

O que esse passo prova

Mesmo sendo uma frase só, o exercício deixa três coisas claras:

  • O ambiente está montado. Se o Olá Mundo aparece, a base está de pé e o agente pode construir em cima.
  • Ficou concreto o que é "rodar o front". O front é a tela que você abre no navegador, não uma resposta que você busca num endereço (isso é o back, que tem o próprio Olá Mundo).
  • A oficina é dele. Esse front que ele montou é o mesmo que, lá na frente, vai responder a cada visitante. Por enquanto ele só provou que o motor pega.
