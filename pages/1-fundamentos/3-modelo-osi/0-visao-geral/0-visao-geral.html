Visão Geral do Modelo OSI

0:00 / 0:00

Dois computadores conversando pela internet têm muita coisa pra resolver ao mesmo tempo: o sinal no fio, o endereço de cada um, a ordem dos pacotes, o conteúdo da mensagem, o formato. O Modelo OSI é a forma de organizar isso — separa o trabalho em 7 camadas, cada uma com uma responsabilidade clara, e cada uma confiando que a de baixo já resolveu o dela.

As 7 camadas em ordem — do topo até a base +

Cada camada tem um nome, um número e uma função. A ordem é convenção: camada 7 é o topo (perto do usuário), camada 1 é a base (perto do hardware).

6ApresentaçãoFormato e segurança dos bytesTLS, charset (UTF-8), compressão
5SessãoAbrir, manter e encerrar sessão entre processos(hoje quase tudo migrou pra aplicação)

Encapsulamento — pacote descendo do navegador até o cabo +

Quando você manda uma requisição HTTP, ela desce pelas camadas. Cada camada envelopa a anterior com seu cabeçalho de controle, como uma carta dentro de vários envelopes:

  1. Camada 7 (Aplicação) — o navegador monta o HTTP: GET /index.html HTTP/1.1
  2. Camada 6 (Apresentação) — TLS criptografa os bytes (se for HTTPS)
  3. Camada 4 (Transporte) — TCP envelopa em segmentos: porta de origem, porta de destino, número de sequência
  4. Camada 3 (Rede) — IP envelopa em pacotes: IP de origem, IP de destino
  5. Camada 2 (Enlace) — Ethernet envelopa em frames: MAC de origem, MAC de destino
  6. Camada 1 (Física) — bits viram pulso elétrico no cabo (ou luz na fibra, ou onda no rádio)

A mensagem original (HTTP) fica no fundo de vários envelopes empilhados. Cada camada não sabe (nem precisa saber) o que tem dentro do envelope da anterior — só processa o seu próprio cabeçalho.

Decapsulamento — pacote subindo do outro lado +

No servidor, o caminho é inverso. O pacote sobe pelas camadas, e cada uma lê e descarta seu cabeçalho:

  1. Camada 1 — recebe o sinal do cabo e converte em bits
  2. Camada 2 — lê o frame Ethernet, confere o MAC de destino, descarta o cabeçalho do frame
  3. Camada 3 — lê o pacote IP, confere o IP de destino, descarta o cabeçalho IP
  4. Camada 4 — lê o segmento TCP, confere a porta, entrega na fila do processo certo
  5. Camada 6 — TLS descriptografa os bytes (se for HTTPS)
  6. Camada 7 — Tomcat / Apache / Nginx recebe a requisição HTTP e entrega pra aplicação

Cada camada só entende sua própria camada. A camada 4 não sabe nada sobre HTTP — só sabe que recebeu um segmento TCP pra entregar na porta X. A camada 3 não sabe nada sobre TCP — só sabe que recebeu um pacote IP com este destino.

Cada camada esconde da outra — abstração +

O ponto-chave do modelo é a abstração. Cada camada confia que a de baixo cuida da parte dela:

  • O HTTP não precisa saber se você está em Wi-Fi ou cabo. A camada 1 cuida disso.
  • Quando você troca a placa de rede do servidor, o código da aplicação não muda.
  • Quando o roteador no meio do caminho falha e o pacote pega outro caminho, o TCP não percebe.
  • Quando você ativa HTTPS, o navegador e o Tomcat não mudam a lógica — só a camada 6 entra em cena entre eles.

Isso é o que faz a internet funcionar com bilhões de máquinas diferentes: cada camada tem uma responsabilidade isolada. Você troca uma sem mexer nas outras.

OSI vs TCP/IP — modelo conceitual vs prático +

Existem 2 modelos de referência usados:

  • OSI (7 camadas) — modelo conceitual da ISO, anos 70/80. Bom pra ensinar, debugar e descrever.
  • TCP/IP (4 camadas) — modelo prático que a internet usa de verdade. Mais simples, agrupa camadas.

O TCP/IP agrupa camadas do OSI:

Na prática, dev de aplicação trabalha pensando em TCP/IP (mais simples). Mas o OSI continua sendo o vocabulário usado pra descrever em qual camada algo mora — quando alguém fala "isso é problema de camada 3", está usando OSI.

Por que importa pro dev — em qual camada cada erro mora +

Quando algo quebra, saber em qual camada está o problema acelera o debug:

Cada camada tem suas ferramentas de diagnóstico:

  • Camada 7 — DevTools do navegador, logs do Tomcat / Apache
  • Camada 4 — telnet host porta, netstat, nmap
  • Camada 3 — ping, traceroute
  • Todas as camadas — Wireshark (mostra pacote por pacote em todas as camadas, do frame Ethernet até o HTTP)
