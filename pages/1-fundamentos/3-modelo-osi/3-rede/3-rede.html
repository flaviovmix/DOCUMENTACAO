Camada 3 — Rede

0:00 / 0:00

A camada 2 entrega entre vizinhos da mesma rede local. A camada 3 entrega entre redes diferentes — atravessando roteadores no caminho. É aqui que cada máquina no mundo ganha um endereço lógico (o IP), que pode atravessar fronteiras. Aqui também vivem as ideias que mais aparecem no dia do dev: máscara de sub-rede, redes privadas, NAT, e os IPs famosos (127.0.0.1, 8.8.8.8) que todo mundo encontra.

IPv4 — o endereço de 32 bits +
0:00 / 0:00

O IPv4 é o protocolo que dá endereço lógico pra cada máquina na rede. Tamanho: 32 bits (4 bytes). Notação dotted-decimal — 4 números de 0 a 255 separados por ponto:

192.168.0.42
8.8.8.8
177.32.150.10

32 bits = 4,3 bilhões de endereços

232 ≈ 4.294.967.296. Pareceu muito em 1981. Hoje é pouco — esgotou em 2011. Por isso existe IPv6 (próxima section), mas IPv4 segue dominante por NAT (que vamos ver).

Estrutura: rede + host

Cada IP tem 2 partes — uma identifica a rede, outra identifica a máquina dentro da rede. A divisão é feita pela máscara de sub-rede (próxima section). Em 192.168.0.42 com máscara /24: 192.168.0 é a rede, 42 é o host.

Classes A/B/C (legado)

Originalmente, IPv4 era dividido em classes pelo primeiro byte. Hoje obsoleto (substituído por CIDR), mas o vocabulário ainda aparece:

ClasseFaixaTamanho da rede
A1.x.x.x a 126.x.x.x16 milhões de hosts (gigantesca)
B128.x.x.x a 191.x.x.x65 mil hosts (média)
C192.x.x.x a 223.x.x.x254 hosts (pequena)

O papel do roteador

Roteador é o equipamento da camada 3 — equivalente do switch (camada 2) no nível de redes. Ele lê o IP de destino do pacote e decide pra qual rede vizinha encaminhar, consultando uma tabela de roteamento. Cada salto entre roteadores é um hop; um pacote pode dar 10-20 hops até atravessar o mundo.

IPv6 — o sucessor de 128 bits +
0:00 / 0:00

IPv6 resolve o esgotamento do IPv4 — endereço de 128 bits (16 bytes). 2128 ≈ 340 undecilhões de endereços. Tantos que dá pra cada átomo de matéria do planeta ter o seu.

Notação hexadecimal

Em vez de decimal, IPv6 usa hexadecimal, separado por dois-pontos, em 8 grupos de 4 dígitos:

2001:0db8:85a3:0000:0000:8a2e:0370:7334

Pra ficar legível, existem 2 atalhos:

  • Zeros à esquerda de cada grupo podem sumir: 0db8 vira db8
  • Sequência de zeros vira :: (uma vez só por endereço)

O exemplo acima fica:

2001:db8:85a3::8a2e:370:7334

IPs IPv6 famosos

::Endereço não especificado (equivalente a 0.0.0.0)
fe80::/10Link-local — só dentro do segmento de rede
2001:db8::/32Reservado pra exemplos/documentação (igual ao "555-0100" do telefone)

Por que ainda usamos IPv4?

IPv6 já existe há mais de 20 anos, mas a adoção é lenta — em 2024 ronda 40% globalmente. NAT permitiu que IPv4 esticasse muito além do esgotamento (com sua casa toda usando 1 IP público), tirando a urgência. Sites grandes (Google, Facebook, Cloudflare) já são dual stack — falam IPv4 e IPv6 ao mesmo tempo.

Máscara de sub-rede e CIDR — separando rede de host +
0:00 / 0:00

A máscara de sub-rede é a régua que diz: "destes 32 bits do IP, X bits são da rede, e o resto é do host". Sem máscara, o IP é só um número — não dá pra saber onde acaba a rede e começa a máquina.

Notação CIDR — o jeito moderno

CIDR (Classless Inter-Domain Routing) escreve a máscara como /N — onde N é quantos bits são da rede:

192.168.0.0/24

O /24 diz: 24 bits de rede, 8 bits de host. Num IP de 32 bits, isso significa:

BitsConteúdo
1–24192.168.0 — fixo (identifica a rede)
25–320–255 — varia (identifica a máquina)

Notação antiga — máscara em dotted-decimal

Antes do CIDR, escrevia-se a máscara como um IP de 32 bits onde os bits "1" são da rede e os "0" são do host:

CIDRMáscara antigaHosts disponíveis
/8255.0.0.016.777.214
/16255.255.0.065.534
/24255.255.255.0254
/30255.255.255.2522

Como calcular: regra dos -2

Pra uma máscara /N:

  • Bits de host = 32 − N
  • Total de endereços = 2(32−N)
  • Hosts utilizáveis = 2(32−N) − 2 (descontando endereço de rede e broadcast)

Exemplo: /24 → 32−24 = 8 bits → 28 = 256 endereços → 254 hosts utilizáveis.

Endereços especiais dentro da rede

  • Primeiro endereço (todos os bits de host = 0) — endereço da rede em si. Ex: 192.168.0.0 em /24
  • Último endereço (todos os bits de host = 1) — broadcast da rede. Ex: 192.168.0.255 em /24
  • Os do meio — utilizáveis pra hosts. Ex: 192.168.0.1 a 192.168.0.254

Atalho mental

As máscaras mais comuns valem decorar:

  • /24 — 256 endereços (rede de casa típica, "192.168.0.x")
  • /16 — 65k endereços (rede grande)
  • /8 — 16M endereços (gigante)
  • /30 — 4 endereços, 2 utilizáveis (link ponto-a-ponto entre roteadores)
Redes privadas e NAT — por que sua casa é 192.168 +
0:00 / 0:00

A internet pública não tem IPs suficientes pra todo aparelho do mundo. A solução: redes privadas + NAT. Sua casa, o escritório, a empresa toda — usam IPs reservados que NÃO atravessam a internet, e um único IP público representa todo mundo lá fora.

RFC 1918 — as 3 faixas privadas

Em 1996, a RFC 1918 reservou 3 blocos de IP que NUNCA são roteados na internet pública:

FaixaCIDRTamanhoOnde aparece
10.0.0.0 a 10.255.255.25510.0.0.0/816 milhõesEmpresa grande, cloud (AWS VPC)
172.16.0.0 a 172.31.255.255172.16.0.0/121 milhãoEmpresa média, Docker default
192.168.0.0 a 192.168.255.255192.168.0.0/1665 milCasa, escritório pequeno

Essas faixas podem ser usadas livremente por qualquer um, em qualquer lugar do mundo. Sua casa em 192.168.0.x, a do vizinho em 192.168.0.x também — não há conflito porque nenhum desses IPs aparece na internet pública.

Por que sua casa é 192.168

O roteador de casa vem configurado de fábrica com a faixa 192.168.0.0/24 ou 192.168.1.0/24. É convenção dos fabricantes. Você poderia trocar pra 10.0.0.0/24 e ia funcionar igual — só não é o default.

NAT — a tradução de IPs

NAT (Network Address Translation) é o que faz vários IPs privados compartilharem 1 IP público. Mora no roteador. Funciona assim:

  1. Sua máquina (192.168.0.42) manda pacote pra 8.8.8.8
  2. O pacote chega no roteador, que tem 2 IPs: o privado da rede de casa (192.168.0.1) e o público da rua (177.32.150.10)
  3. O roteador reescreve o IP de origem: troca 192.168.0.42 por 177.32.150.10
  4. Anota numa tabela: "porta X = computador 192.168.0.42"
  5. O servidor responde pra 177.32.150.10, porta X
  6. O roteador olha a tabela, vê que aquela porta era do computador 192.168.0.42, traduz de volta e entrega

Resultado: dezenas de aparelhos da casa parecem 1 IP só pra internet. Por isso IPv4 esticou tanto.

Efeito colateral: você não pode ser servidor sem ajuda

Se alguém da internet quer falar com seu computador (não você indo pra fora), o pacote chega em 177.32.150.10 — mas o roteador não sabe pra qual máquina interna entregar. O pacote morre. Por isso pra rodar servidor em casa precisa de port forwarding manual no roteador, ou serviço externo (ngrok, Cloudflare Tunnel, VPN).

O teste do "ping 8.8.8.8"

Quando alguém diz "minha internet está caindo", o primeiro teste é ping 8.8.8.8:

  • Funciona → camadas 1-3 OK, internet OK. Problema é em camada superior (DNS? aplicação?)
  • Não funciona → problema é roteamento, gateway, ou link com o provedor

Em seguida, ping google.com:

  • Funciona → DNS está OK, tudo OK
  • Não funciona, mas IP funciona → DNS quebrado (camada 7)

Esses 2 comandos diagnosticam metade dos casos de "tá sem internet".

Ferramentas de diagnóstico — camada 3 na prática +
0:00 / 0:00

Subindo o roteiro: depois das ferramentas de camada 2 (cabo, link, ARP), agora as de camada 3.

ping — está vivo?

$ ping -c 4 8.8.8.8
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
64 bytes from 8.8.8.8: icmp_seq=1 ttl=117 time=12.3 ms
64 bytes from 8.8.8.8: icmp_seq=2 ttl=117 time=11.9 ms
...
4 packets transmitted, 4 received, 0% packet loss

Manda pacote ICMP echo. Se volta, o destino está vivo e o caminho funciona. Tempo (RTT) e perda de pacote (packet loss) são as métricas. 0% loss e tempo estável = saúde boa. Loss alto ou tempo irregular = caminho com problema.

traceroute / tracert — qual caminho?

$ traceroute google.com
 1  192.168.0.1   1.2 ms     ← gateway de casa
 2  10.50.0.1     8.5 ms     ← rede do provedor
 3  177.32.0.1    12.1 ms    ← borda do provedor
 4  ... (10 a 15 hops até Google) ...
15  142.250.78.46 22.0 ms    ← Google

Mostra cada roteador (hop) entre você e o destino. Útil pra:

  • Saber onde está o gargalo (qual hop demora mais)
  • Ver onde a conexão morre (último hop que respondeu = onde quebrou)
  • Entender geograficamente por onde passa

No Windows o comando é tracert.

mtr — ping + traceroute juntos

mtr é a versão "live" — mostra ping contínuo pra cada hop ao mesmo tempo. Detecta perda intermitente que ping sozinho mascara.

$ mtr google.com

ip route / route -n — tabela de roteamento da máquina

$ ip route show
default via 192.168.0.1 dev wlan0
192.168.0.0/24 dev wlan0 proto kernel scope link src 192.168.0.42

Mostra pra onde a máquina envia cada faixa de IP. default via X é o gateway — pra onde vai TUDO que não é da rede local. Linha por linha:

  • default — qualquer IP fora da rede local vai pra 192.168.0.1
  • 192.168.0.0/24 — IPs da rede local saem direto pela placa Wi-Fi (sem passar pelo gateway)

whois — quem é dono daquele IP?

$ whois 8.8.8.8
NetName: GOGL
OrgName: Google LLC
Country: US

Lookup público de quem registrou aquele IP. Útil pra identificar IP suspeito em log, ver país de origem, descobrir provedor.

Roteiro de debug atualizado (camadas 1-3)

  1. Cabo plugado, LED aceso? (camada 1)
  2. ip link show — placa UP? (camada 2)
  3. ip neigh show — gateway REACHABLE? (camada 2 / ARP)
  4. ping 192.168.0.1 — gateway responde? (camada 3 local)
  5. ping 8.8.8.8 — internet sai? (camada 3 internet)
  6. ping google.com — DNS funciona? (camada 7)
  7. traceroute google.com — onde para? (camada 3 caminho)

Cada falha aponta exatamente em qual camada o problema mora. Esse roteiro resolve a maioria dos sintomas de "rede está estranha".
