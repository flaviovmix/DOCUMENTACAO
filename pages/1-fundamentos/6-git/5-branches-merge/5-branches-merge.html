Git — Branches e Merge

0:00 / 0:00

Até aqui a linha do tempo do git foi uma reta só: um commit atrás do outro. Esse card abre a parte que faz o git valer a pena de verdade: branches (ramos paralelos da linha do tempo) e merge (juntar dois ramos de volta num só). É o que permite você desenvolver numa branch dev sem mexer na main, e só trazer pra main quando estiver pronto.

O que é uma branch

A intuição comum é errada: uma branch não é uma cópia da pasta do projeto. Branch é só um ponteiro — uma etiqueta leve que aponta pra um commit. Quando você commita estando numa branch, a etiqueta anda junto, sempre marcando o commit mais recente daquele ramo.

Por isso criar branch no git é instantâneo e barato: não copia arquivo nenhum, só cria mais uma etiqueta. Os commits ficam todos no mesmo repositório; o que muda é qual etiqueta aponta pra qual ponto da linha do tempo.

A branch padrão, criada no primeiro commit, costuma se chamar main (antigamente master). A partir dela você cria outras pra trabalhar em paralelo.

HEAD — onde você está agora

Além das etiquetas de branch, existe um ponteiro especial chamado HEAD. Ele aponta pra branch em que você está no momento. Quando você troca de branch, o HEAD muda de alvo e o git reescreve os arquivos do disco pra refletir aquele ramo.

Resumindo os três tipos de ponteiro que aparecem no git log: main e dev são etiquetas de branch; HEAD é "você está aqui". Quando o log mostra (HEAD -> dev), significa "você está na branch dev".

Criar e trocar de branch

Dois passos conceitualmente separados: criar a branch e mover pra ela.

git branch dev      # cria a branch dev (você CONTINUA onde está)
git switch dev      # move pra ela (HEAD passa a apontar pra dev)

O git branch dev sozinho só cria a etiqueta — você não sai da branch atual. O git switch dev é que faz o HEAD pular pra lá. Tem o atalho que junta os dois num comando só:

git switch -c dev   # -c = create: cria e já move

E o git checkout? O git checkout dev faz a mesma troca de branch. Ele é o comando antigo, que acumula muitas funções (trocar branch, restaurar arquivo, navegar pra um commit). O git switch foi criado depois pra isolar só a parte de trocar de branch, deixando o comando mais claro. Os dois funcionam; switch é o recomendado hoje pra essa tarefa específica.

Por que ter uma branch dev

A ideia: a main guarda só o que está pronto e estável; o desenvolvimento do dia a dia acontece numa branch dev (ou em branches de feature). Você commita à vontade na dev — inclusive coisas pela metade, com bug, em teste — sem sujar a main. Quando o conjunto fica pronto, você junta dev na main com um merge.

Isso te dá uma main sempre confiável (dá pra publicar a qualquer momento) e liberdade pra experimentar na dev sem medo.

git merge — juntando os ramos

Merge é trazer os commits de uma branch pra dentro de outra. Você fica na branch de destino (a que vai receber) e manda mergear a de origem:

git switch main        # vai pro destino
git merge dev          # traz os commits da dev pra main

O git resolve isso de duas formas, dependendo da situação:

  • Fast-forward — se a main não recebeu nenhum commit novo desde que a dev saiu dela, não há o que conciliar: o git só "empurra" a etiqueta main pra frente, até o último commit da dev. A linha do tempo continua reta. Nenhum commit novo é criado.
  • Merge commit — se as duas branches andaram (cada uma ganhou commits próprios), o git cria um commit de merge novo, com dois pais, amarrando as duas histórias. É aqui que o git log --graph mostra os ramos convergindo.

Em ambos os casos, todos os commits individuais da dev ficam preservados no histórico da main.

Squash merge — achatar tudo num commit só

Às vezes a dev tem 15 commits bagunçados ("wip", "ajuste", "corrige typo") e você não quer poluir a main com esse rastro. O squash pega todas as mudanças da branch e deixa preparadas como um único conjunto, sem comitar:

git switch main
git merge --squash dev      # joga todas as mudanças no stage, NÃO commita
git commit -m "mensagem do conjunto inteiro"

O --squash não cria commit sozinho — ele só prepara as mudanças. O git commit seguinte é que gera um commit limpo com tudo dentro.

Merge normal vs squash, lado a lado:

  • Merge normal preserva cada commit da dev na história da main. Bom quando o histórico detalhado importa.
  • Squash achata todos num só. Bom quando você quer a main com um commit por feature, sem o ruído do desenvolvimento.

Detalhe: depois de um squash, o git não considera a dev "mergeada" (porque não há commit de merge ligando as histórias). Se for continuar usando a dev, o normal é apagá-la e criar uma nova a partir da main atualizada.

Clone — começar a partir de um repo que já existe

Até agora a gente partiu de git init (transformar uma pasta local em repo). Mas quando o repositório já existe (no GitHub, por exemplo), o caminho é git clone:

git clone git@github.com:usuario/projeto.git
cd projeto

O clone faz tudo de uma vez: baixa todos os commits e todas as branches, cria a pasta, já configura o remoto (origin) apontando pro endereço, e já te deixa na branch principal com o rastreamento pronto. Você cai num repositório completo, com a linha do tempo inteira.

init vs clone, a diferença em uma frase: init nasce vazio a partir de uma pasta sua; clone nasce já preenchido a partir de um repo que existe. Depois do clone você trabalha igual: cria branch, commita, e (próximo card) sincroniza com push e pull.

Erros comuns

  • Achar que branch é cópia de arquivos. Não é. É ponteiro. Trocar de branch reescreve o disco, mas os commits moram todos no mesmo .git.
  • Commitar na branch errada. Antes de commitar, um git status mostra na primeira linha em que branch você está. Conferir vira hábito barato.
  • Esperar squash e levar merge normal (ou vice-versa). Sem --squash o merge preserva todos os commits. Com --squash você ainda precisa do git commit depois, senão nada é gravado.
  • Tentar trocar de branch com trabalho não commitado que conflita. O git pode barrar. Resolve commitando ou guardando de lado (git stash, no próximo card).
