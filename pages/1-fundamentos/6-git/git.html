Git no projeto

O XT trabalha com duas branches: dev (onde o trabalho do dia a dia acontece) e main (a branch limpa, com histórico em squash). Toda a sequência de comandos que aparece no /commit gira em torno disso — commitar na dev, atualizar a main com um único commit que resume tudo, e voltar pra dev.

O fluxo dev → main

Em vez de fazer merge direto da dev na main (que carregaria N commits intermediários), o XT usa merge squash: a main recebe um commit único que resume toda a mudança da dev. No fim, um merge -s ours "fecha o histórico" — sem ele, o próximo merge tentaria trazer os commits antigos de novo e daria conflito.

  • dev: histórico real, com todos os commits do dia.
  • main: histórico limpo, um commit por sessão de trabalho.

git status — ver o que mudou

Lista os arquivos modificados, novos (untracked) e os que já estão no stage pra commitar. É o primeiro comando do fluxo — antes de adicionar qualquer coisa, você olha o status pra entender o que o git "vê".

git status

No /commit, esse passo serve também pra filtrar: arquivos grandes, .env, secrets ou pastas que não deveriam ir (tipo PROEJTOS PARA CLAUDE/) são identificados aqui antes do add.

git add — selecionar pra commitar

Coloca um arquivo (ou vários) no stage — a área de espera do próximo commit. Só o que está em stage entra no commit; o resto fica como modificação solta.

git add caminho/do/arquivo.java
git add .
  • git add <arquivo>: adiciona um item específico — preferido quando você quer controle do que vai.
  • git add .: adiciona tudo da pasta atual pra baixo. Mais rápido, mas perigoso — pode pegar arquivos sensíveis sem você ver.

No fluxo do XT, a Fase 2 (commit na dev) usa adições específicas pra evitar incluir lixo. Já a Fase 3 (squash na main) pode usar git add . sem medo — porque a main só recebe o que já foi filtrado e commitado na dev.

git commit -m — gravar a mudança

Pega tudo que está em stage e cria um commit no histórico, com a mensagem que você passar. Só depois do commit a mudança vira "permanente" no histórico local.

git commit -m "feat: adiciona DepartamentoForm com 4 campos"

A flag -m ("message") permite passar a mensagem direto na linha. Sem ela, o git abriria o editor (vim por padrão) — é por isso que -m está em todo lugar.

git push — enviar pro GitHub

Manda os commits que estão só no seu computador pro repositório remoto (no caso, o GitHub). Antes do push, o trabalho existe só localmente — depois do push, está sincronizado e visível pra outras máquinas.

git push

Se você tentar push sem antes ter dado commit, não acontece nada — push só envia o que já foi commitado.

git checkout — trocar de branch

Move o "ponteiro" pra outra branch. Os arquivos no disco mudam pra refletir o estado daquela branch.

git checkout main
git checkout dev

No fluxo do XT, o checkout aparece três vezes: pular pra main (Fase 3, passo 8), e voltar pra dev no fim (Fase 4, passo 15) — e mais uma variante especial com --theirs (próxima seção).

git merge --squash — empilhar tudo num commit só

Diferente do merge normal (que cria um "merge commit" trazendo todos os commits da branch de origem), o --squash achata tudo: as mudanças aparecem no stage como se fossem uma só, prontas pra um commit único.

git checkout main
git merge --squash dev

É isso que mantém a main com histórico limpo: cada sessão de trabalho da dev (com 5, 10, 20 commits) vira um commit na main.

git checkout --theirs — resolver conflitos do squash

Quando o squash gera conflitos (porque a main e a dev divergiram), --theirs resolve aceitando a versão da branch que está sendo mergeada — no caso, a dev.

git checkout --theirs .

Faz sentido aqui porque a dev é a fonte da verdade — a main só recebe o que já foi aprovado lá. Sem isso, você teria que abrir cada arquivo conflitado e escolher manualmente.

commit do squash — fechar a fase 3

Depois do --theirs e do git add . que recoloca tudo no stage, vem o commit que cria o commit único da sessão na main:

git add .
git commit -m "merge: sessão do dia X — feature Y"

Esse é o commit que vai aparecer no histórico limpo da main.

git merge -s ours — fechar o histórico

Esse é o passo crítico do fluxo — sem ele, o próximo merge tentaria trazer todos os commits antigos da dev de novo, gerando conflitos enormes.

git merge -s ours dev

A estratégia ours diz pro git: "registra que a main absorveu tudo da dev, mas mantém o conteúdo atual da main". O resultado é um merge commit "vazio" que só serve pra marcar no histórico que a dev até esse ponto já está dentro da main.

  • Sem -s ours: o próximo squash mostra todos os commits da dev outra vez, como se nada tivesse sido mergeado.
  • Com -s ours: o git sabe que a dev já entrou; o próximo squash só mostra o que veio depois.

git pull — sincronizar de outra máquina

Quando você troca de computador (ou alguém mais commita no remoto), pull traz as mudanças do GitHub pra sua máquina. É o oposto do push.

git pull

No XT, isso aparece no fluxo simétrico do /sync-pull — depois de um git pull, é preciso copiar as memórias e settings.json do espelho versionado de volta pra pasta do harness.

A sequência inteira em ordem

Os 16 passos do /commit resumidos pra fixar a ordem:

  1. Backup de memórias e settings.
  2. git status — ver o que mudou.
  3. git add <files específicos> — selecionar.
  4. git commit -m "msg" — commitar na dev.
  5. git push — enviar a dev.
  6. git checkout main — pular pra main.
  7. git merge --squash dev — empilhar.
  8. git checkout --theirs . — resolver conflitos.
  9. git add . — restagear.
  10. git commit -m "msg" — commit do squash.
  11. git merge -s ours dev — fechar o histórico.
  12. git push — enviar a main.
  13. git checkout dev — voltar pra dev.
  14. Repetir o ciclo no submódulo de documentação se houver mudança lá.

Por que esse fluxo e não merge direto

O git permite vários modelos de branching. O XT escolheu este por três razões:

  • Histórico limpo na main: abrir o log da main mostra um commit por sessão, fácil de varrer.
  • Histórico real preservado na dev: nada se perde — todos os commits do dia continuam lá.
  • Sem conflito acumulado: o merge -s ours garante que cada sessão começa do zero, sem arrastar conflitos antigos.

A regra prática: nunca improvisar a ordem dos passos. Pular o -s ours ou trocar a ordem do --theirs com o add quebra o histórico de um jeito difícil de desfazer. Sempre que o pedido for "atualizar com o GitHub", invocar o /commit — ele tem os 16 passos certinhos.
