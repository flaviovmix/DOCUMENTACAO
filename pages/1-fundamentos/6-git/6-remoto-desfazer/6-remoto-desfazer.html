Git — Remoto e Desfazer

0:00 / 0:00

Até aqui tudo aconteceu na sua máquina. Esse card cobre as duas frentes que faltam: o remoto (mandar e trazer commits do GitHub, lidar com divergência e conflitos) e desfazer (voltar atrás antes ou depois de commitar, sem perder trabalho por acidente). É a parte onde mais gente se assusta, e quase todo susto vem de agir sem olhar o estado antes.

O remoto — ligando o local ao GitHub

Um remoto é um repositório espelho hospedado em outro lugar (GitHub, GitLab). O apelido padrão dele é origin. Se você criou o repo local com git init, precisa ligá-lo ao remoto uma vez:

git remote add origin git@github.com:usuario/projeto.git
git branch -M main          # garante que a branch principal se chama main
git push -u origin main     # primeiro envio

O -u (de upstream) amarra sua main local à main do origin. Feito uma vez, os próximos envios viram só git push, sem repetir o resto. (Quem começou com git clone já tem o origin e o rastreamento configurados — pula esse passo.)

push e pull — sincronizar

Dois sentidos:

  • git push — envia seus commits locais pro remoto. "Sobe" o que você fez.
  • git pull — traz os commits do remoto pro seu local e já tenta juntar. "Desce" o que mudou lá.

Por baixo, pull = fetch (baixar os commits novos, sem mexer no seu trabalho) + merge (juntar). Dá pra fazer só o git fetch quando você quer ver o que veio antes de juntar.

Divergência local/web — o push recusado

O problema clássico: você fez commits locais e o remoto também ganhou commits (você editou direto na web, ou outra máquina deu push). As duas histórias andaram em paralelo — divergiram. Nenhuma é continuação direta da outra.

Aí o git push é recusado, com algo como "rejected — remote contains work that you do not have". O git se recusa a sobrescrever o que está no remoto. A solução é trazer o de lá primeiro e juntar com o seu:

git pull origin main
# resolve o que vier...
git push

Regra que evita dor: pull antes de começar a trabalhar e antes de cada push. Quanto mais sincronizado, menos chance de divergir.

Conflitos — quando o git não decide sozinho

Se os dois lados mexeram em arquivos/linhas diferentes, o git junta sozinho. O conflito só acontece quando os dois mexeram na mesma linha do mesmo arquivo — aí o git não tem como saber qual vale, e pausa pra você decidir. Ele marca o trecho no arquivo:

<<<<<<< HEAD
versão do seu computador
=======
versão que veio do remoto
>>>>>>> origin/main

Você abre cada arquivo conflitado, em cada trecho marcado, deixa só o que vale (um lado, o outro, ou uma mistura escrita à mão) e apaga as linhas <<<, ===, >>>. Depois fecha o merge:

git add .
git commit          # conclui o merge
git push

Se forem 10 arquivos com vários conflitos cada, são dezenas de decisões manuais. Esse é o custo que assusta — e o motivo de sincronizar cedo e em pedaços pequenos.

Escolher um lado inteiro — ours e theirs

Quando você tem certeza de que um lado inteiro vale e o outro pode ser descartado sem olhar, dá pra mandar o git resolver os empates sempre pro mesmo lado:

git merge -X ours origin/main      # em empate, mantém o MEU (local)
git merge -X theirs origin/main    # em empate, mantém o de LÁ (remoto)

A pegadinha está no nome: ours = o lado de quem dá o comando (seu local); theirs = o outro lado (o remoto). Pra escolher um lado de um arquivo específico durante um conflito:

git checkout --ours caminho/arquivo.txt
git checkout --theirs caminho/arquivo.txt

O risco de escolher no escuro: o -X ours/theirs não apaga arquivos inteiros nem o trabalho que não conflita — só decide os empates. Mas "sei que o meu está certo" costuma valer pro arquivo que você editou, e o outro lado pode ter um commit legítimo em outro arquivo que você nem lembra. Ao escolher cego, você descarta silenciosamente as edições conflitantes do outro lado e nunca vê o que perdeu. É seguro quando você controlou os dois lados (ex: seu próprio fluxo dev → main); arriscado quando outra pessoa ou outra máquina pode ter mexido. O meio-termo: deixar o pull listar os arquivos em conflito e decidir vendo a lista.

.gitignore — o que o git deve ignorar

Arquivo de texto na raiz do repo, uma linha por padrão, listando o que não deve ser versionado:

node_modules/
dist/
.env
*.log

O git passa a ignorar esses caminhos: não aparecem no status, não entram no git add .. Serve pra dependências (baixáveis de novo), arquivos de build (gerados) e segredos (senhas, tokens).

Pegadinha: o .gitignore só vale pra arquivo que ainda não foi commitado. Se você já commitou node_modules, adicionar no ignore não tira — precisa parar de rastrear primeiro, sem apagar do disco:

git rm -r --cached node_modules

Desfazer antes de commitar — git restore

Você editou e se arrependeu, ainda não commitou:

git restore arquivo            # joga fora a edição, volta ao último commit
git restore --staged arquivo   # tira do stage, mas MANTÉM a edição

A diferença: com --staged ele só desfaz o git add (o arquivo volta pra "modificado, não preparado"), mantendo seu conteúdo. Sem --staged, ele apaga a mudança de vez — e como ela não estava em commit nenhum, é irreversível. Cuidado com essa.

Desfazer depois de commitar — revert vs reset

Já commitou e quer voltar atrás. Dois caminhos opostos:

git revert <hash>    # cria um NOVO commit que desfaz aquele. Histórico intacto.
git reset <hash>     # APAGA commits, movendo a branch pra trás.

revert é o seguro, principalmente se já deu push: não reescreve história, só adiciona um commit "anti". É o certo pra repo compartilhado.

reset reescreve a história. Tem três intensidades, e a diferença está no que ele faz com as suas mudanças:

  • --soft — desfaz o commit, mas mantém as mudanças preparadas (no stage). Útil pra refazer a mensagem ou juntar commits.
  • --mixed (padrão) — desfaz o commit, mantém as mudanças no working tree, mas tira do stage.
  • --hard — desfaz o commit e apaga as mudanças. É o mais perigoso: o que não estava commitado some. Antes dele, sempre git status; nunca rode no escuro.

git stash — guardar trabalho de lado

Você está no meio de uma edição, não quer commitar ainda, mas precisa pular pra outra branch. O stash guarda as mudanças numa gaveta e limpa a área de trabalho:

git stash          # guarda tudo e volta ao último commit
git switch outra-branch
# faz o que precisa...
git switch dev
git stash pop      # traz a gaveta de volta e a esvazia

git stash list mostra o que está guardado (dá pra empilhar vários). pop aplica a última e remove da gaveta; apply aplica mas mantém guardada.

force push — sobrescrever o remoto

O push normal é recusado quando seu local diverge do remoto (a divergência lá de cima). O --force ignora isso e manda o seu por cima, apagando o que estava no remoto:

git push --force              # perigoso: apaga o que estava lá
git push --force-with-lease   # mais seguro: aborta se mexeram no remoto

O --force-with-lease só força se ninguém mexeu no remoto desde seu último fetch — se mexeram, ele aborta em vez de apagar. Regra: em repo só seu, --force-with-lease é aceitável; em repo compartilhado, evite, prefira resolver com pull.

Ver antes de agir

A maioria dos sustos com git (reset que apagou trabalho, merge que comeu um arquivo) vem de agir sem olhar o estado. A trinca de leitura — vista no card de Histórico — é o seguro barato antes de qualquer merge, reset ou push --force:

git status              # onde estou, o que está pendente
git diff                # o que exatamente mudou
git log --oneline       # como cheguei aqui

status responde "onde estou"; diff responde "o que mudou"; log responde "como cheguei aqui". Rodar os três antes de uma ação destrutiva custa segundos e evita perder horas.

Erros comuns

  • Dar push --force em repo compartilhado. Apaga commits de outras pessoas sem aviso. Use --force-with-lease ou, melhor, resolva com pull.
  • Confundir ours e theirs. ours é o seu lado, theirs é o do outro — e quem é "o outro" depende de você estar em merge ou rebase. Na dúvida, veja a lista de conflitos antes.
  • git reset --hard no escuro. Apaga o que não estava commitado. Sempre git status antes.
  • Adicionar ao .gitignore esperando que tire arquivo já commitado. Não tira. Precisa de git rm --cached primeiro.
  • Commitar segredo (.env, senha). Uma vez no histórico, fica lá mesmo se você apagar depois. Prevenir com .gitignore desde o início.
