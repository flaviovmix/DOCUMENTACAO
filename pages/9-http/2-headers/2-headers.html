HTTP — Cabeçalhos (headers)

0:00 / 0:00

Se o método e a URI dizem o que a requisição quer, os cabeçalhos dizem como: que formato o cliente aceita, em que idioma, com qual credencial, em qual cookie e por aí vai. Quase tudo de sofisticado no HTTP — autenticação, cache, compressão, CORS, conteúdo dinâmico — vive nos headers.

O que são headers

Cabeçalhos são pares Nome: valor, um por linha, vindos logo depois da linha inicial (request-line ou status-line) e antes do corpo da mensagem. A separação entre os headers e o corpo é uma linha em branco — é literalmente o que o HTTP usa pra saber onde os metadados terminam.

GET /api/produtos HTTP/1.1
Host: www.meusite.com.br
Accept: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

{ corpo aqui, se houver }

Os headers descrevem a mensagem sem fazer parte do conteúdo. O corpo é a carga; os headers são o envelope com remetente, destinatário, tipo de conteúdo e instruções de manuseio.

Onde aparecem

Headers podem aparecer na requisição, na resposta, ou em ambas. Existem quatro categorias úteis na hora de classificar:

  • General — fazem sentido tanto em request quanto em response e não falam sobre o corpo. Ex: Date, Connection.
  • Request — só em requisições, descrevem o cliente ou condicionam a resposta. Ex: Host, User-Agent, Accept.
  • Response — só em respostas, dão informação extra do servidor. Ex: Server, Set-Cookie, Location.
  • Entity / Representation — descrevem o corpo (request ou response). Ex: Content-Type, Content-Length, Content-Encoding.

Na prática raramente importa decorar a categoria — basta saber que cada header tem um lado em que faz sentido aparecer.

Formato e regras

  • Nome do header é case-insensitive. Content-Type, content-type e CONTENT-TYPE são equivalentes. Convenção: cada palavra capitalizada com hífen no meio.
  • Valor é livre, mas o significado do valor depende do header. Alguns aceitam case-insensitive (Connection: keep-alive); outros são sensíveis (ETag, tokens, hashes).
  • Múltiplos valores na mesma linha separados por vírgula: Accept: text/html, application/xhtml+xml, application/json.
  • Mesmo header repetido em linhas separadas é raro mas válido — o caso clássico é Set-Cookie, que aparece uma vez por cookie.
  • Q-values (qualidade) indicam preferência: Accept: text/html;q=0.9, application/json;q=1.0 diz "prefiro JSON, mas aceito HTML". Sem q, assume-se 1.0.
  • Linha em branco (CRLF duplo) marca o fim dos headers e o início do corpo.

Cabeçalhos de requisição mais comuns

Header O que diz
HostDomínio do site sendo pedido. Obrigatório a partir do HTTP/1.1 — é o que permite vários sites compartilharem o mesmo IP.
User-AgentIdentifica o cliente (navegador, app, bot). Servidores às vezes mudam o conteúdo conforme essa string.
AcceptTipos de conteúdo que o cliente aceita receber. Ex: text/html, application/json, */* (qualquer coisa).
Accept-LanguageIdiomas preferidos. Ex: pt-BR,pt;q=0.9,en;q=0.8.
Accept-EncodingCompressões que o cliente sabe descomprimir. Ex: gzip, deflate, br.
AuthorizationCredencial do cliente. Formatos comuns: Basic base64(user:senha), Bearer <token>.
CookieCookies que o navegador devolve pro servidor (foram setados por Set-Cookie antes).
RefererURL da página de onde o usuário veio. (Sim, está escrito errado — "referrer" no inglês — mas o erro virou padrão.)
OriginOrigem (esquema + domínio + porta) que disparou a requisição. Central no CORS.
Connectionkeep-alive (reaproveita TCP) ou close (fecha após a resposta).
Content-TypeTipo do corpo enviado (em POST/PUT). Ex: application/json, application/x-www-form-urlencoded, multipart/form-data.
Content-LengthTamanho do corpo em bytes. Servidor usa pra saber quanto ler.

Cabeçalhos de resposta mais comuns

Header O que diz
Content-TypeFormato do corpo retornado. Ex: text/html; charset=UTF-8, application/json, image/png.
Content-LengthTamanho do corpo em bytes.
Content-EncodingCompressão aplicada (gzip, br). O cliente descomprime antes de mostrar.
ServerSoftware do servidor. Ex: Apache/2.4.18 (Ubuntu), nginx. Muitos servidores escondem por segurança.
Set-CookiePede pro navegador guardar um cookie. Aparece uma vez por cookie sendo setado.
LocationPra onde redirecionar. Acompanha status 301, 302, 303, 307 ou 308.
Cache-ControlRegras de cache: no-store, no-cache, max-age=3600, public, private.
ETag"Impressão digital" da versão atual do recurso. Cliente reusa em chamadas futuras pra checar se algo mudou.
DateData e hora em que a resposta foi gerada (em GMT).
WWW-AuthenticateAcompanha 401 Unauthorized dizendo o tipo de autenticação esperada (Basic, Bearer).
Access-Control-Allow-OriginFamosa no CORS — diz quais origens podem ler a resposta via JavaScript.

Negociação de conteúdo

Um dos truques mais elegantes do HTTP: a mesma URL pode servir conteúdos diferentes dependendo do que o cliente pediu. É a content negotiation, e ela funciona em pares:

  • Accept (cliente) ↔ Content-Type (servidor) — formato (HTML vs JSON vs XML).
  • Accept-Language ↔ Content-Language — idioma.
  • Accept-Encoding ↔ Content-Encoding — compressão.

O cliente diz suas preferências, com pesos opcionais (q=0.9), e o servidor escolhe a melhor opção que consegue oferecer. Se nada bater, devolve 406 Not Acceptable.

Um exemplo prático: a mesma URL /api/produtos pode devolver HTML quando aberta no navegador (Accept: text/html) e JSON quando chamada por uma SPA (Accept: application/json) — sem mudar de endpoint, sem versionar URL.

Cabeçalhos customizados

Aplicações criam headers próprios pra carregar informação que não cabe nos padrões. Por convenção antiga, custom headers começavam com X-:

  • X-Frame-Options — proteção contra clickjacking.
  • X-Powered-By — qual framework gerou a resposta.
  • X-Forwarded-For — IP original do cliente quando a requisição passou por proxy.

O problema: alguns desses X- viraram tão comuns que foram padronizados — e aí o X- ficou marcando uma "promessa de não-padrão" que se quebrou. Em 2012 a RFC 6648 desencorajou o prefixo. Hoje, custom headers novos usam o nome direto, sem X-.

Na prática, dá pra inventar qualquer header — só não inventar nome que conflite com algum padrão já existente.

Como inspecionar headers

Saber ler headers é a habilidade que separa quem chuta de quem debuga. As três formas mais práticas:

  • DevTools do navegador (F12) → aba Network/Rede → clicar numa requisição → painel Headers separa Request Headers e Response Headers. Mostra também URL, método, status e tempo.
  • curl -v na linha de comando — imprime os headers enviados (>) e recebidos (<). Comando: curl -v https://meusite.com.br.
  • Wireshark / mitmproxy pra inspeção mais profunda — vê os pacotes na rede, útil quando o problema está abaixo do HTTP (TLS, DNS, conexão).

Casos típicos em que isso salva: API que devolve 401 sem motivo aparente (falta o Authorization?), CORS bloqueando o front (Origin bate com Access-Control-Allow-Origin?), cache que não atualiza (Cache-Control agressivo?), upload que falha (Content-Type errado no formulário?).

Lição geral

Headers são o canivete suíço do HTTP. O método e a URI definem a operação; os headers carregam todo o resto: identidade, formato, idioma, compressão, cache, autenticação, segurança. Praticamente toda funcionalidade não-trivial da web é uma combinação específica de headers funcionando juntos.

Não precisa decorar nomes — precisa saber que existe um header pra cada coisa, e saber inspecionar pra descobrir qual quando o sistema reclamar. Os próximos tópicos (cookies, cache, CORS) são, na essência, capítulos de "o que esses headers específicos fazem".
