HTTP — Como funciona

0:00 / 0:00

HTTP (Hypertext Transfer Protocol) é o idioma que navegador e servidor usam pra trocar páginas, imagens e dados na web. Toda vez que você digita um endereço, abre um link ou clica num botão que carrega algo, é HTTP rodando por trás.

O que é HTTP

A sigla vem de Hypertext Transfer Protocol — "Protocolo de Transferência de Hipertexto". Hipertexto é texto com conexões clicáveis (links) que permitem pular de um documento pra outro de forma não linear. O HTTP é o conjunto de regras que define como esse hipertexto trafega entre máquinas.

Na prática, é o protocolo que permite a um estudante num café em São Paulo ler um artigo armazenado num servidor no Japão. É a base da World Wide Web e o protocolo mais usado em aplicações web e APIs.

A origem e a evolução

Em 1989, no CERN (Organização Europeia para Pesquisa Nuclear), o britânico Tim Berners-Lee propôs um sistema global pra compartilhar informação entre cientistas. Em 1990 nasceu o projeto WorldWideWeb, junto com a primeira versão do HTTP.

As principais versões:

  • HTTP/0.9 (1990) — só método GET, sem cabeçalhos, só conteúdo HTML.
  • HTTP/1.0 (1996) — cabeçalhos mais ricos, novos métodos como POST.
  • HTTP/1.1 (1997) — conexão persistente: várias requisições reusando a mesma conexão TCP.
  • HTTP/2 (2015) — multiplexação, cabeçalhos binários comprimidos, server push.
  • HTTP/3 (2018) — adota o protocolo QUIC (sobre UDP) pra reduzir latência.

Cliente, servidor e proxies

O HTTP envolve três tipos de ator:

  • Cliente — quem inicia a conversa pedindo um recurso. Geralmente um navegador, mas também pode ser um app mobile, um script ou qualquer coisa que faça requisição.
  • Servidor — quem hospeda o recurso e responde. Pode ser uma máquina única ou um cluster. O cabeçalho Host (introduzido no 1.1) permite que vários sites compartilhem o mesmo IP.
  • Proxy — intermediário entre cliente e servidor. Forward proxy age em nome do cliente (esconder identidade, filtrar saída); reverse proxy age em nome do servidor (cache, balanceamento, terminação SSL).

Como funciona o ciclo

O HTTP segue o modelo request–response:

  1. O cliente abre uma conexão TCP com o servidor e envia uma requisição (método + caminho + cabeçalhos + corpo opcional).
  2. O servidor processa e devolve uma resposta (versão + código de status + cabeçalhos + corpo).
  3. A conexão pode ser reaproveitada (a partir do 1.1) ou fechada.

Cada requisição é tratada de forma independente — o protocolo é stateless, ou seja, o servidor não guarda informação entre uma chamada e outra. Quando uma aplicação precisa de estado (login, carrinho), usa mecanismos extras como cookies e sessões.

Os métodos HTTP

O método indica a ação desejada sobre o recurso:

Método O que faz
GETSolicita um recurso. Parâmetros vão na URL, raramente no corpo.
HEADIgual ao GET, mas devolve só os cabeçalhos (sem corpo). Útil pra metadados.
POSTEnvia dados pra processamento. Usado pra criar recursos ou submeter formulários.
PUTCria ou substitui completamente o recurso na URI especificada.
DELETERemove o recurso indicado pela URI.
OPTIONSPergunta quais métodos e cabeçalhos o servidor aceita pra aquele recurso.
TRACEReflete a requisição de volta pro cliente — diagnóstico/debug.
CONNECTEstabelece um túnel (geralmente SSL/TLS) através de um proxy.

Anatomia de uma requisição

Toda mensagem HTTP tem três partes: linha inicial (status ou solicitação), cabeçalhos (metadados) e corpo opcional. Exemplo de uma requisição completa que o navegador envia:

GET / HTTP/1.1
Host: www.meusite.com.br
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: */*
Accept-Language: pt-BR,pt;q=0.9
Accept-Encoding: gzip, deflate
Connection: keep-alive

E a resposta correspondente que o servidor devolve:

HTTP/1.1 200 OK
Date: Sun, 09 Apr 2023 12:28:53 GMT
Server: Apache/2.4.18 (Ubuntu)
Content-Length: 612
Content-Type: text/html; charset=UTF-8

<html>
<head><title>Exemplo</title></head>
<body><h1>Olá, Mundo!</h1></body>
</html>

Os códigos de status mais comuns: 200 (OK), 301 (redirecionamento), 404 (não encontrado), 500 (erro do servidor). Pra inspecionar requisições reais: abrir o DevTools do navegador (F12) → aba Rede / Network.

HTTP vs HTTPS

O HTTPS (HTTP Secure) é o mesmo HTTP, mas trafegando dentro de uma camada de criptografia SSL/TLS. A diferença prática:

  • HTTP puro — dados em texto claro. Quem consegue interceptar a conexão (rede pública, ISP, atacante na mesma rede Wi-Fi) lê tudo.
  • HTTPS — dados criptografados ponta a ponta. O interceptador vê só ruído.

Pra qualquer coisa que envolva senha, dado pessoal, pagamento ou sessão de usuário, HTTPS deixou de ser opcional há anos. Navegadores modernos marcam HTTP puro como "não seguro" e bloqueiam recursos sensíveis.

Lição geral

HTTP é um protocolo simples na essência: o cliente pede, o servidor responde, e cada par de mensagens é independente. Toda a riqueza da web — cookies, autenticação, cache, APIs REST, streaming de vídeo — é construída em cima dessas regras básicas usando cabeçalhos e métodos.

Entender o que está dentro de uma requisição (método, URI, cabeçalhos, corpo, status) é o que separa quem só usa o navegador de quem consegue debugar uma API que não responde, configurar um cache corretamente ou ler um log do servidor sem se perder.
