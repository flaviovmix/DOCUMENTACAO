Refatorar no VS Code

0:00 / 0:00

Um aquecimento sobre o editor. No anterior a gente largou o botão "play" do NetBeans e rodou Java na mão. Agora que o código vai morar no VS Code, surge uma dúvida prática: como renomear uma variável (ou método, ou classe) e trocar em todos os lugares certos, do jeito que o NetBeans fazia num clique. A resposta curta: o VS Code puro não faz isso sozinho, quem faz é uma extensão.

O problema — renomear em vários lugares +

Você chamou uma variável de x e quer que vire contador. Só que esse nome aparece em vários pontos: na declaração, dentro de uma conta, dentro de um if, talvez em outro arquivo. No NetBeans era um clique e ele trocava tudo de uma vez. A pergunta é: como repetir isso no VS Code sem deixar nenhum lugar pra trás.

A tentação do find and replace — e por que quebra +

A primeira ideia é o buscar e substituir (Ctrl + H): põe x, troca por contador em tudo. O problema é que ele troca texto, não entende código. Três jeitos de dar errado:

  • existe outra variável x em outra função, sem relação com essa, e ele troca ela também;
  • a letra x aparece no meio de uma palavra maior (max, exemplo) ou num comentário, e ele mexe ali;
  • a letra x aparece dentro de uma string (um texto que ia pra tela), e ele estraga o texto.

O buscar e substituir não sabe a diferença entre dois x diferentes, nem entre código e comentário. Por isso é perigoso pra renomear.

O que falta: entender o código, não o texto +

O que o NetBeans fazia (e o que a gente quer) é o rename semântico. "Semântico" aqui quer dizer que a ferramenta entende a estrutura do código: sabe qual x é qual, sabe o escopo de cada variável (onde ela vive), e sabe que comentário e string não são código. Com isso ela troca só as ocorrências certas daquela variável específica e ignora o resto. É a diferença entre achar e trocar no cego e alguém que leu e entendeu o texto.

A solução no VS Code — a extensão +

O VS Code sozinho é só um editor de texto esperto, não entende Java de cara. Quem dá esse poder é uma extensão: o Extension Pack for Java (da Microsoft), que traz, entre outras coisas, o Language Support for Java by Red Hat.

Essa extensão sobe por baixo um language server: um programa que fica lendo seu código e entendendo a estrutura. O motor dele é o mesmo do Eclipse (o JDT), que faz rename há muitos anos, e é por isso que dá pra confiar pra renomear.

Detalhe prático: ao abrir o projeto, o language server demora alguns segundos pra carregar e indexar. Tem um aviso de status embaixo que vai de "carregando" até Java: Ready. Renomear antes disso pode trocar só uma parte. Regra: espere o Java: Ready.

Como usar — F2 (Rename Symbol) +

Com a extensão instalada e o projeto carregado, renomear é assim:

  1. clique em cima do nome da variável, método ou classe;
  2. aperte F2 (o atalho do Rename Symbol); dá também por botão direito, Rename Symbol;
  3. digite o nome novo e aperte Enter.

Ele troca em todos os lugares certos, em todos os arquivos, respeitando o escopo. O F2 é genérico: funciona pra qualquer linguagem que tenha um language server instalado, não só Java. Pra Java, quem dá o poder é a extensão do Red Hat.

Os limites (sendo honesto) +

Pra renomear variável, método e classe, é seguro e equivalente ao NetBeans (é o mesmo tipo de rename semântico). Mas dois pontos valem a honestidade, e valem pro NetBeans também:

  • espere o language server carregar (Java: Ready), senão a troca pode sair incompleta;
  • nenhum rename pega referência que está fora do código tipado: nome dentro de string, código que usa reflection, ou arquivo de configuração (XML, Spring). Nesse ponto os dois são igualmente cegos.

Na dúvida, dá pra revisar os arquivos afetados antes de aplicar.
