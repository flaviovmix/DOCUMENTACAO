Montar o ambiente: do zero até pronto pra rodar

0:00 / 0:00

Esse episódio dá um passo atrás. Todos os trajetos até aqui (rodar com o npm run dev, clicar, adicionar, remover) já partem de um projeto montado na máquina. Aqui a gente volta ao começo de tudo: como esse ambiente nasce do zero, do nada até o projeto pronto pra um npm run dev. Ele está numerado por último, mas na prática é o passo zero, vem antes de todos os outros. O ator aqui não é servidor nem navegador: são as ferramentas de linha de comando (npx, sv e npm).

Parada 1 — o pré-requisito: Node e npm na máquina +

Antes de qualquer comando do projeto, a máquina precisa ter o Node.js instalado. Ele não vem sozinho: traz junto duas ferramentas que a gente usa o tempo todo.

  • npm — instala e gerencia os pacotes (as dependências).
  • npx — roda um pacote na hora, sem precisar instalar de forma permanente.

Dá pra conferir se está tudo no lugar:

node -v   # neste projeto: v24.16.0
npm -v    # neste projeto: 11.13.0

Sem Node, nenhum dos próximos passos roda. Ele é o chão de tudo.

Parada 2 — criar o projeto: npx sv create +

O projeto não foi montado arquivo por arquivo na mão. Um comando faz o scaffold, ou seja, monta a estrutura inteira de uma vez. O comando exato que criou este projeto:

npx sv@0.15.4 create --template minimal --no-types --no-install 01-ola-web-svelte

Lendo por partes:

  • npx sv — o npx baixa o CLI do Svelte (o sv) na hora e executa, sem instalar global na máquina.
  • create ... 01-ola-web-svelte — cria o projeto nessa pasta.
  • --template minimal — o esqueleto mais enxuto.
  • --no-types — JavaScript puro, sem TypeScript (menos ruído pra aprender).
  • --no-install — adia o download das dependências pra um passo separado (a Parada 5).

Rodado sem essas flags, o sv create faz as mesmas perguntas no terminal (template, TypeScript, add-ons). As flags só respondem tudo de uma vez.

Parada 3 — o que o scaffold gera +

Em um comando, ele monta o projeto todo. As peças:

PeçaO que éVocê mexe?
src/Onde você trabalha (as rotas e páginas)Sim
static/Arquivos servidos como estão (imagens, robots.txt)Às vezes
package.jsonLista de dependências e os scripts (dev, build)Raramente
svelte.config.js, vite.config.jsConfigs que ligam as ferramentasQuase nunca
.npmrcAjustes do npm pra este projetoQuase nunca
jsconfig.jsonAjuda o editor a entender o projetoNão
.gitignoreO que o git deve ignorar (ex: node_modules)Raramente

Repare que ainda não existe a pasta node_modules/. Foi o --no-install que segurou isso. É o próximo passo.

Parada 4 — o .npmrc e o engine-strict +

Entre os arquivos gerados tem um pequeno e importante, o .npmrc:

engine-strict=true

Ele liga uma regra: o npm passa a exigir a versão de Node que o projeto pede (declarada nas dependências). Se a sua versão não bate, ele falha cedo, com um aviso claro, em vez de deixar o projeto quebrar de um jeito estranho lá na frente. É uma rede de segurança da versão.

Parada 5 — instalar as dependências: npm install +

O scaffold só listou as dependências no package.json. Baixá-las é um passo separado, que o --no-install tinha adiado. Dentro da pasta do projeto:

cd 01-ola-web-svelte
npm install

O npm install lê a seção de dependências do package.json e baixa tudo pra dentro da pasta node_modules/. É lá que mora, entre outras coisas, o binário do Vite, que o npm run dev vai executar no próximo episódio.

Essa pasta node_modules/ é gerada, não entra no git (está no .gitignore). Quem clona o projeto roda npm install e refaz a pasta a partir do package.json.

Parada 6 — ambiente pronto +

Fim do trajeto do ambiente. Agora existe um projeto completo: a estrutura de pastas, as configs, um exemplo que já roda e as dependências baixadas em node_modules/. A máquina tem tudo que precisa.

O próximo passo é ligar o servidor de desenvolvimento com npm run dev, que é justamente o episódio que abre essa sequência de trajetos (numerado antes deste, mas que só faz sentido depois deste). Daqui pra frente o caminho é o que os outros episódios já seguiram.

npx vs npm install: duas formas, dois usos +

Os dois aparecem no setup e é fácil confundir:

ComandoPra quê
npx sv createRodar uma ferramenta uma vez (o scaffold), sem instalar nada permanente na máquina
npm installBaixar as dependências do projeto pra node_modules/ (fica no projeto)

E o que entra no git: o código (src/), as configs e o package.json entram. O node_modules/ e as pastas geradas (.svelte-kit/, build/) não, são reconstruídas a partir do package.json.
