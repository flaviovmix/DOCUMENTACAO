Mandar pro GitHub

0:00 / 0:00

Até aqui a página existe só no seu PC. Agora dois movimentos: versionar com git (guardar o histórico das mudanças) e publicar no GitHub (o código na nuvem). É aqui que mora uma pegadinha clássica de quem está começando — e a gente vai cair nela de propósito pra você nunca mais tropeçar.

Git, em uma ideia +

Git é um histórico de versões do seu projeto. Cada commit é uma foto do estado dos arquivos num momento — com uma mensagem dizendo o que mudou. Você pode voltar a qualquer foto.

Dois lugares importam:

  • Working tree — seus arquivos como estão agora na pasta.
  • Staged (área de preparação) — o que você marcou pra entrar na próxima foto.

E dois "lugares" de armazenamento: o local (a pasta .git no seu PC) e o remoto (o GitHub). Commit é local; mandar pro GitHub é um passo separado (push).

Versionar local — init, add, commit +

Dentro da pasta do projeto, transforme-a num repositório e tire a primeira foto:

# 1. transforma a pasta num repositório git (cria a pasta escondida .git)
git init

# 2. marca os arquivos pra próxima foto (o "." = tudo nesta pasta)
git add .

# 3. tira a foto, com uma mensagem em inglês (convenção)
git commit -m "counter with manual DOM update"

Por que add e commit? O add coloca os arquivos na área staged (você escolhe o que entra); o commit grava de fato no histórico. Pra ver o histórico:

git log --oneline

No Windows pode aparecer warning: LF will be replaced by CRLF. É só sobre formato de fim de linha — inofensivo, pode ignorar.

Criar o repositório no GitHub — vazio! +

No site do GitHub: botão New → dá um nome → Create repository. O ponto crítico:

⚠️ Como você já tem o repo local com commits, crie o do GitHub VAZIO — não marque "Add a README", nem .gitignore, nem licença.

Por quê? Se o GitHub já vem com um commit (o README) e o seu local também já tem commits, são duas histórias separadas, cada uma com seu próprio "primeiro commit". Na hora do push o git recusa, porque não sabe como encaixar uma na outra.

Regra de bolso:

  • Já tenho código local → crio o GitHub vazio e dou push. (nosso caso)
  • Quero começar pelo GitHub → crio com README e faço git clone pra minha máquina.
HTTPS ou SSH — as duas formas de conectar +

Ao copiar a URL do repositório, o GitHub oferece duas abas (não é uma só):

FormaURLComo autentica
HTTPShttps://github.com/usuario/repo.gitUsuário + token (pede nas operações)
SSHgit@github.com:usuario/repo.gitUma chave SSH que você gera uma vez e cadastra; depois nem pede senha

As duas fazem a mesma coisa. SSH é cômodo depois de configurado; HTTPS funciona sem setup de chave. Escolha uma e use a URL correspondente no próximo passo.

Ligar e empurrar — remote, branch, push +

Com o repo vazio criado, conecte sua pasta a ele e mande os commits:

# registra o remoto, apelidado de "origin" (use a URL que você copiou)
git remote add origin git@github.com:usuario/repo.git

# renomeia o ramo local de "master" pra "main" (padrão do GitHub)
git branch -M main

# envia os commits pro origin, no ramo main. O -u liga os dois.
git push -u origin main

Depois desse primeiro push -u, as próximas vezes são só git push (ele já sabe pra onde). Atualize a página do repositório no GitHub: seu código está lá.

Na primeira conexão SSH o git pergunta "Are you sure you want to continue connecting?" — responda yes uma vez (ele guarda a "host key" do GitHub).

Quando dá errado — "histórias separadas" +

Se você criou o repo com README e tentou empurrar, o git recusa assim:

! [rejected]        main -> main (fetch first)
error: failed to push some refs to ...
hint: Updates were rejected because the remote contains work that you
hint: do not have locally.

Tradução: "o GitHub tem um commit (o README) que você não tem". Duas saídas:

  • Recriar vazio (o jeito limpo): no GitHub, Settings → Danger Zone → Delete repository, e crie de novo sem README. Aí o push entra liso.
  • Juntar as histórias: git pull origin main --allow-unrelated-histories --no-edit traz o README e funde com o seu (são arquivos diferentes, sem conflito); depois git push.
Resumo dos comandos +

O caminho feliz, do zero ao GitHub:

git init
git add .
git commit -m "minha primeira foto"

# (criar o repo VAZIO no site do GitHub)

git remote add origin git@github.com:usuario/repo.git
git branch -M main
git push -u origin main
