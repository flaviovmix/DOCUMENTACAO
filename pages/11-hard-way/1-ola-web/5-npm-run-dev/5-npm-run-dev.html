npm run dev: o que acontece

0:00 / 0:00

Você digita npm run dev, dá Enter, e em menos de um segundo aparece um endereço pra abrir no navegador. Mas entre o Enter e a tela, um monte de peça entra em cena. Vamos seguir o trajeto parada por parada, do comando até o app montado na tela. O ator aqui não é o agente que nasce e morre a cada requisição: é o Vite, o servidor de desenvolvimento, um processo que sobe e fica vivo segurando a porta.

Parada 1 — o comando no terminal +

Dentro da pasta do projeto, você roda:

npm run dev

Quem recebe isso é o npm, o gerenciador de pacotes que veio junto com o Node. O npm run <nome> não é um comando do Vite: é o npm dizendo "vou procurar um atalho chamado dev e executá-lo". Ele ainda nem sabe o que dev faz.

Parada 2 — npm lê o package.json e acha o atalho +

O npm abre o package.json na raiz do projeto e procura na seção scripts a chave dev:

"scripts": {
    "dev": "vite dev",
    "build": "vite build",
    "preview": "vite preview",
    "prepare": "svelte-kit sync || echo ''"
}

Achou: "dev": "vite dev". Então npm run dev é só um apelido pra vite dev. O build e o preview são outros apelidos ali do lado, pra quando você for gerar a versão final.

Arquivo: package.json (raiz do projeto).

Parada 3 — npm executa o vite +

O npm executa vite dev. Mas qual vite? Não é um programa instalado no sistema todo: é o binário que o npm install baixou pra dentro do projeto, em node_modules/.bin/vite. Quando você roda um script, o npm coloca essa pasta na frente do caminho, então vite resolve pro binário local.

É por isso que o npm install tinha que rodar antes: sem ele, a pasta node_modules/ não existe e não há vite pra executar.

Arquivo: node_modules/.bin/vite (gerado pelo install, não entra no git).

Parada 4 — o Vite lê as configs +

Ao subir, o Vite procura o vite.config.js pra saber como se comportar:

import { sveltekit } from '@sveltejs/kit/vite';
import { defineConfig } from 'vite';

export default defineConfig({
    plugins: [sveltekit()]
});

A peça-chave é o plugin sveltekit(). É ele que ensina o Vite a entender arquivos .svelte, a tratar a pasta src/routes/ como rotas e a compilar tudo. E esse plugin, por sua vez, lê o svelte.config.js:

import adapter from '@sveltejs/adapter-auto';

const config = {
    compilerOptions: { runes: true },   // (resumido)
    kit: { adapter: adapter() }
};

export default config;

Aqui mora a configuração do SvelteKit: o adapter (pra onde o projeto vai quando for pra produção) e o modo runes ligado, que é o que faz o $state funcionar nos seus componentes.

Arquivos: vite.config.js e svelte.config.js (raiz).

Parada 5 — o servidor sobe e abre a porta +

Com as configs lidas, o Vite monta o servidor de desenvolvimento (um processo Node), abre uma porta e imprime o endereço:

  VITE v8.0.16  ready in 753 ms

  ➜  Local:   http://localhost:5173/

A porta padrão é a 5173. É aqui que nasce o ator de longa duração: o processo fica vivo, escutando essa porta. Por isso o terminal trava ali, ocupado segurando o servidor. Pra parar, Ctrl + C.

Repare: até agora nenhuma página foi servida ainda. O servidor só está de pé, esperando alguém bater na porta.

Parada 6 — o navegador pede a página +

Você abre http://localhost:5173/ no navegador. Agora sim existe uma requisição: o navegador bate na porta do Vite pedindo a rota /.

O Vite serve o esqueleto da página (src/app.html) e, pra rota /, compila na hora o arquivo src/routes/+page.svelte, transformando o seu componente em JavaScript que o navegador entende, e manda esse JS junto.

Detalhe de fidelidade: quem responde é o Vite servindo e compilando arquivos, não um backend processando a lógica do seu app. É outro tipo de servidor.

Arquivos: src/app.html (esqueleto), src/routes/+page.svelte (a rota /).

Parada 7 — o app monta na tela +

O navegador recebe o JS, executa, e o Svelte cria os elementos na tela: o contador e a lista de frutas aparecem. O trajeto do npm run dev termina aqui, com a página montada.

Daqui pra frente, clicar no +1 ou no adicionar roda dentro do navegador, sem voltar ao Vite. O servidor não fica "esperando o clique": quem espera é o navegador. (É exatamente isso que os próximos dois episódios seguem de perto.)

O Vite continua vivo só pra uma coisa: o hot reload. Você salva um arquivo .svelte, ele recompila aquele pedaço e a tela atualiza sozinha, sem reiniciar nada.

Quem é o ator: o Vite, processo de longa duração +

Vale separar dois tipos de servidor pra não confundir mais pra frente:

Servidor de aplicação (ex: o back)Servidor de desenvolvimento (Vite)
Processa a lógica do app a cada requisiçãoServe e compila os arquivos do front
Você sobe pra rodar a regra de negócioVocê sobe pra desenvolver com hot reload
Vive enquanto o app está no arVive enquanto você está codando (Ctrl + C encerra)

O ponto comum: os dois são processos de longa duração, não o agente que nasce e morre numa requisição só. O npm run dev sobe um deles e o deixa de pé.

dev vs build — dois atalhos diferentes +

No mesmo package.json moram outros atalhos. O dev é pra desenvolver: rápido de subir, compila sob demanda, hot reload. O build gera a versão final, otimizada, pra colocar no ar; e o preview serve essa versão final pra você conferir antes de publicar.

AtalhoPra quê
npm run devDesenvolver agora, com hot reload (:5173)
npm run buildGerar a versão final otimizada
npm run previewConferir a versão final localmente

É o mesmo npm run dev que sobe o front do Nexus (também na :5173). Lá tem ainda o back, que sobe por outro comando, na :8080. O atalho que você aprendeu aqui é o mesmo.
