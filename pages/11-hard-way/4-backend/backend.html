Backend na Unha

0:00 / 0:00

Fase 4 da trilha: o backend que responde. Já temos o front (Fase 1) e os aquecimentos de Java na linha de comando. Agora a gente sobe a outra ponta: um servidor HTTP escrito na unha, em Java puro, sem nenhum framework. A regra da trilha é a de sempre: fazer o backend cru primeiro, sentir a dor, pra só depois o Spring entrar como alívio. Esta página anda pelo arquivo Servidor.java peça por peça.

O que a gente queria

A regra do The Hard Way é sempre a mesma: fazer a coisa crua primeiro, bater na dor, e só depois trazer o framework como resposta. Então antes de ligar o Spring, a gente escreve um servidor HTTP na mão, pra sentir na pele o que o Spring faz por baixo.

Surpresa boa logo de cara: não precisou baixar nada. O Java 25 já estava na máquina, e desde o Java 11 dá pra rodar um arquivo .java direto, sem compilar à parte. O comando é só java Servidor.java. Nada de Maven, nada de download. É o aquecimento do javac e do java na mão que explica por que isso funciona.

cd "NEXUS - THE HARD WAY/04-backend"
java Servidor.java

Subir o servidor

O servidor não veio de biblioteca de fora: ele vem dentro do próprio JDK. O import é com.sun.net.httpserver.HttpServer. O Java já traz um servidor HTTP simples de fábrica, bom pra estudo e coisa pequena.

HttpServer servidor = HttpServer.create(new InetSocketAddress(8080), 0);

servidor.createContext("/ping", Servidor::ping);
servidor.createContext("/notas", Servidor::notas);

servidor.start();
  • HttpServer.create(...) prepara o servidor na porta 8080. O segundo argumento, 0, é o tamanho da fila de conexões esperando atendimento. Zero quer dizer "usa o padrão".
  • createContext é o coração da Fase 4: registra rota por rota, na mão. Cada chamada liga um caminho a uma função que responde por ele. São duas agora, dá pra fazer no dedo. Num app com cinquenta rotas, isso vira uma lista enorme de createContext amarrada à mão. Guarda essa dor.
  • servidor.start() liga de verdade e começa a atender.

As duas rotas e a dor de montar a resposta

A rota /ping é o "tá vivo?": devolve a palavra pong com status 200. Abriu, respondeu, provou que o servidor responde.

A rota /notas é onde mora a dor. Ela devolve uma lista em JSON, só que o JSON a gente monta no braço:

static void notas(HttpExchange troca) throws IOException {
    var repositorio = new RepositorioDeNotas();   // o "banco" fake (Fase 3 ainda não existe)
    List<String> itens = repositorio.listar();

    StringBuilder json = new StringBuilder("[");
    for (int i = 0; i < itens.size(); i++) {
        json.append("\"").append(itens.get(i)).append("\"");
        if (i < itens.size() - 1) json.append(",");
    }
    json.append("]");

    responder(troca, 200, "application/json", json.toString());
}

Funciona, mas é frágil: nenhum objeto virou JSON sozinho. Se você esquecer uma aspas ou uma vírgula, o JSON sai quebrado e o compilador não reclama, porque pra ele é só um texto qualquer. Você só descobre o erro quando o outro lado tenta ler.

O ajudante responder escreve a resposta HTTP na mão, em três passos: vira o texto em bytes (UTF-8, pra acento não virar lixo), seta o Content-Type, e manda status + tamanho + corpo. Repara em quanta etapa manual pra devolver um texto.

static void responder(HttpExchange troca, int status, String tipo, String corpo) throws IOException {
    byte[] bytes = corpo.getBytes(StandardCharsets.UTF_8);
    troca.getResponseHeaders().set("Content-Type", tipo + "; charset=utf-8");
    troca.sendResponseHeaders(status, bytes.length);   // 1a linha (status) + headers
    try (OutputStream saida = troca.getResponseBody()) {
        saida.write(bytes);                            // o corpo
    }
}

A teia de new

Volta naquele new RepositorioDeNotas() dentro da rota. Hoje é um new só, inocente. Mas mostra um problema que cresce: pra rota entregar o dado, ela precisa do repositório, e quem cria o repositório é a própria rota, dando new na mão. Num app de verdade isso empilha: a rota precisa de um serviço, o serviço precisa do repositório, o repositório precisa da conexão. Cada um dando new no próximo e passando adiante. Vira uma teia de new amarrada no braço.

Isso tem nome: é o problema que a injeção de dependência resolve. Em vez de cada peça dar new na seguinte, alguém de fora monta tudo e entrega pronto. No próximo passo da jornada é exatamente o que o Spring faz: você só diz "preciso do repositório" e ele entrega, montado.

O beco: o erro que não deixou compilar

Na primeira vez que mandamos rodar, nem compilou:

Servidor.java:22: error: incompatible thrown types Exception in functional expression
        servidor.createContext("/ping", Servidor::ping);

A causa é fina e vale ouro. O createContext espera um HttpHandler, cuja função handle só permite lançar um tipo de erro: IOException. Mas os métodos ping e notas estavam com throws Exception, que é o pai de todas as exceções, mais amplo. Quando você liga um método a uma interface por referência, esse método não pode lançar um erro mais amplo do que a interface aceita. Exception é mais amplo que IOException, então o Java barrou.

O conserto foi trocar throws Exception por throws IOException nos três métodos e importar o IOException. A lição: quando você conecta um método a uma interface, a assinatura tem que caber inteira, inclusive a parte do throws. Não dá pra prometer lançar mais do que o contrato permite.

Onde isso encosta no front

Por que esse backend existe? Lá no front, na Fase 1, a lista de frutas estava chumbada no código da tela, escrita à mão dentro da página. O backend existe pra ser o dono dessa lista: a rota /notas devolve ela como JSON, num lugar só.

O próximo passo (Fase 5) é o front parar de ter a lista chumbada e passar a buscar do servidor, com um fetch em /notas. Aí o dado mora num lugar só, e depois sai de um banco de verdade. E é nessa hora que aparece a primeira pedra real: o navegador bloqueia esse fetch, porque o front roda numa porta e o back noutra. Isso se chama CORS, e é a dor da próxima fase.

Resumo

Subir um servidor são três coisas: HttpServer.create na porta, createContext pra cada rota, e start. Responder é montar status, cabeçalho e corpo na mão. Virar JSON no braço com StringBuilder é frágil, e é o que um framework faz sozinho. Dar new no repositório dentro da rota é a teia de new que a injeção de dependência desfaz. E o erro do throws Exception virando IOException mostrou que referência de método tem que respeitar até as exceções do contrato. Tudo isso junto é a dor que o Spring Boot vai tirar na sequência da Fase 4.
