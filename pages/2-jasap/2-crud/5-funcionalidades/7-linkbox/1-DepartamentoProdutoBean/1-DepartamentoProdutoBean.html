O Bean é o arquivo que começa a sequência do LinkBox. A mudança é pequena em linhas, mas conceitual: o Produto deixa de guardar o fk_pessoa_produto como um campo próprio e passa a guardar um objeto DepartamentoPessoaBean embutido. A FK vira apenas uma delegação: "pega o id_pessoa da pessoa embutida".

Esse arranjo permite que o LinkBox do formulário consiga exibir o nome da pessoa vinculada — não só o ID. Ao carregar o produto, o Form chama daoSingle(proBean().getPessoa()) pra popular o objeto pessoa com os dados completos.

Visão Geral — o que muda nesta sequência

Quatro blocos novos são adicionados ao Bean:

BlocoO que é
import ... DepartamentoPessoaBeanNovo import — o Bean do Produto passa a conhecer o Bean da Pessoa
private DepartamentoPessoaBean pessoaCampo novo — objeto embutido que vai carregar os dados da pessoa responsável
getPessoa() / setPessoa()Getter com lazy init (cria o objeto na primeira chamada) + setter padrão
getFk_pessoa_produto() / setFk_pessoa_produto()A FK deixa de ter campo próprio. Delega pro id_pessoa da pessoa embutida
FK_PESSOA_PRODUTOConstante com o nome da coluna no banco. Usada pelo LinkBox e pelo Manager

O que não muda: a coluna no banco (fk_pessoa_produto) continua igual. Pra quem lê o banco, o Bean segue exatamente o mesmo contrato — quem chamar setFk_pessoa_produto(42) vai ter o mesmo efeito de antes (grava 42 na coluna). A diferença é interna: o valor passa por uma ponte (objeto pessoa) em vez de ficar num campo isolado.

Conceito — Lazy init do objeto relacionado 
public DepartamentoPessoaBean getPessoa() {
    if (pessoa == null) pessoa = new DepartamentoPessoaBean();
    return pessoa;
}

Lazy init = "criar só quando precisar". O campo pessoa começa null. Só vira um objeto de verdade na primeira vez que alguém chamar getPessoa().

Por que não inicializar já no campo (private DepartamentoPessoaBean pessoa = new DepartamentoPessoaBean())?

  • Economia — se ninguém usar a pessoa (ex: listagem que só mostra o nome do produto), o objeto nunca é criado
  • Compatível com populateBean — quando o Jasap popula o Bean a partir do ResultSet, o campo pessoa fica null. Quem chamar getPessoa() depois garante que retorna um objeto válido (não null), evitando NullPointerException
  • Encapsulamento — o código externo nunca vê o pessoa cru. Sempre passa pelo getter, que garante a invariante "nunca retorna null"

Esse padrão é chamado de "null object via getter" — muito comum no Jasap pra relacionamentos entre Beans.

Conceito — Delegação da FK (sem campo próprio) 
public Integer getFk_pessoa_produto()           { return getPessoa().getId_pessoa(); }
public void    setFk_pessoa_produto(Integer fk) { getPessoa().setId_pessoa(fk); }

Reparar: não existe mais um private Integer fk_pessoa_produto no Bean. O getter e o setter operam direto no id_pessoa da pessoa embutida.

O motivo é prático: o LinkBox do Form precisa exibir o nome da pessoa. Se o Bean guardasse só a FK como Integer, o Form teria que fazer uma consulta extra toda vez pra descobrir o nome. Com o objeto embutido, basta o DAO chamar daoSingle(proBean().getPessoa()) uma vez e o nome já fica disponível em getPessoa().getNome_pessoa().

Do ponto de vista do banco, nada muda: quando o DAO vai gravar, chama getFk_pessoa_produto() e recebe o inteiro correto. Quando lê do banco, chama setFk_pessoa_produto(42) — o setter propaga pra getPessoa().setId_pessoa(42) e preenche o objeto embutido (só com o ID, por enquanto).

SituaçãoO que acontece
DAO lê do banco (populateBean)setFk_pessoa_produto(42) → getPessoa().setId_pessoa(42). O objeto pessoa é criado (lazy) com só o ID preenchido
Form carrega o nome (daoSingle)daoSingle(proBean().getPessoa()) preenche os demais campos da pessoa (nome, apelido, etc)
Form exibe no LinkBoxproBean().getPessoa().getNome_pessoa() — já tá lá
Form grava no banco (update)O DAO chama getFk_pessoa_produto() → getPessoa().getId_pessoa() → retorna o ID correto pra coluna
Constante FK_PESSOA_PRODUTO 
public static String FK_PESSOA_PRODUTO = "fk_pessoa_produto";

Mesmo padrão das outras constantes do Bean (ID_PRODUTO, NOME_PRODUTO, etc): nome da coluna no banco, pronto pra ser referenciado sem strings soltas pelo código.

Dois consumidores principais:

  • LinkBox no Form — usa como chave do campo hidden: addKey(DepartamentoProdutoBean.FK_PESSOA_PRODUTO, proBean().getFk_pessoa_produto())
  • Manager — não usa diretamente, mas a convenção do projeto exige a constante pra qualquer coluna do banco
