Setup Inicial — Subir o Nexus em Docker making-of

A máquina foi formatada. O projeto estava no disco, mas a máquina estava limpa: sem Java, sem Node, sem banco. Aqui está o passo a passo real de colocar o Nexus pra rodar do zero, com o banco num container Docker e backend/frontend direto na máquina. Cada comando explicado pedaço por pedaço.

0:00 / 0:00

As 3 peças (e o conceito que amarra tudo)

O Nexus só funciona com três coisas ligadas ao mesmo tempo:

  • Banco PostgreSQL — onde a informação mora. Vai rodar num container Docker.
  • Backend Spring Boot (porta 8080) — o cérebro: recebe pedido, fala com o banco, responde. Roda direto na máquina com Java 25.
  • Frontend SvelteKit (porta 5173) — a tela no navegador. Roda direto na máquina com Node.

O conceito que torna tudo isso possível: a aplicação não sabe nem liga se o Postgres é um container ou um programa instalado. A config dela (application.yml) só diz "conecta em localhost:5432, banco nexus, usuário nexus, senha nexus_dev". Quem aparecer nesse endereço com nome e senha certos, serve. Por isso dá pra usar Docker numa máquina e instalação normal em outra, sem mudar uma linha do projeto.

Passo 1 — O banco no Docker

Primeiro, baixar a imagem do Postgres (a "caixinha" pronta com o banco dentro):

docker pull postgres:16-alpine

docker pull baixa uma imagem. postgres:16-alpine é qual: Postgres versão 16, na variante alpine (enxuta, leve).

Depois, criar e ligar o container:

docker run -d --name nexus-postgres -e POSTGRES_DB=nexus -e POSTGRES_USER=nexus -e POSTGRES_PASSWORD=nexus_dev -p 5432:5432 -v nexus_pgdata:/var/lib/postgresql/data postgres:16-alpine

Parte por parte:

  • docker run — cria e liga um container novo.
  • -d — detached: roda em segundo plano e devolve o terminal. Sem isso, o terminal ficaria preso no log do banco.
  • --name nexus-postgres — apelido do container, pra achar ele depois.
  • -e POSTGRES_DB=nexus — variável de ambiente (-e = environment). Já cria um banco chamado nexus.
  • -e POSTGRES_USER=nexus — cria o usuário.
  • -e POSTGRES_PASSWORD=nexus_dev — define a senha. Esses três juntos são exatamente o que o backend espera achar.
  • -p 5432:5432 — abre a porta. Esquerda = porta no Windows, direita = porta dentro do container. Mapeadas, o Windows enxerga o banco.
  • -v nexus_pgdata:/var/lib/postgresql/data — o volume. Container é descartável; se você apagar, os dados sumiriam junto. O volume guarda os dados num cofre que sobrevive a recriar o container.
  • postgres:16-alpine — a imagem usada.

Pra conferir se subiu (procurar nexus-postgres com status Up):

docker ps

Passo 2 — O backend (Java 25)

O backend é Java, então precisa do JDK 25 instalado (usamos o Eclipse Temurin 25, gratuito). No instalador, marcar a opção de configurar o JAVA_HOME e o PATH. Pra conferir, num terminal novo:

java -version

Tem que mostrar a versão 25. Detalhe importante: um terminal aberto antes da instalação não enxerga o java. As variáveis de ambiente só valem em terminais abertos depois. (Mais sobre isso nos becos, lá embaixo.)

Com o Java no lugar, entrar na pasta e subir:

cd "C:\src\PROJETOS\nexus\backend"
.\mvnw.cmd spring-boot:run
  • .\ — executa um arquivo da pasta atual (o PowerShell só roda algo local se você apontar).
  • mvnw.cmd — o Maven Wrapper (o "w" é de wrapper). Maven é a ferramenta que compila e roda projetos Java; o wrapper é um atalho que já vem no projeto e baixa a versão certa do Maven sozinho, sem você instalar Maven. O .cmd é a versão pro Windows.
  • spring-boot:run — a tarefa: compila e sobe a aplicação Spring Boot, deixando ela rodando.

A primeira vez demora, e aqui tem um conceito: o projeto não carrega as bibliotecas junto, só a receita delas, no arquivo pom.xml. O Maven lê essa receita, vai no repositório público (Maven Central) e baixa cada biblioteca pra um cache na máquina (a pasta .m2). Da segunda vez em diante, sobe rápido.

Pronto quando o log mostra Started NexusApplication e Tomcat started on port 8080. Nesse momento o Flyway cria as tabelas no banco automaticamente.

Passo 3 — O frontend (Node)

O frontend é SvelteKit, que roda sobre o Node. Instalar o Node na versão LTS (a estável recomendada). A tela do instalador que oferece ferramentas extras pra compilar módulos nativos (Python + Build Tools) pode ficar desmarcada: o Nexus não precisa. Conferir num terminal novo:

node -v

Entrar na pasta e baixar as dependências:

cd "C:\src\PROJETOS\nexus\frontend"
npm install

npm é o gerenciador de pacotes do Node. npm install lê o package.json (a receita do frontend, igual o pom.xml era a do backend) e baixa tudo pra pasta node_modules.

Numa máquina recém-formatada, o PowerShell costuma bloquear o npm com erro de segurança, porque no Windows o npm é, por baixo, um script. A correção, rodada uma vez só:

Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
  • Set-ExecutionPolicy — muda a política de quais scripts podem rodar.
  • RemoteSigned — scripts que você criou/instalou na máquina rodam livres; só os baixados da internet precisariam de assinatura. É o nível recomendado pela própria Microsoft pra desenvolvimento.
  • -Scope CurrentUser — vale só pro seu usuário, sem mexer no sistema nem precisar de admin.

Depois disso, o npm install roda. Os avisos amarelos de "vulnerabilities" são ruído; não rode npm audit fix (pode atualizar pacotes e quebrar). Por fim, subir o frontend:

npm run dev

Aparece Local: http://localhost:5173. Abrir no navegador e o Nexus está de pé.

Religar o container depois

O container já existe. Ao reiniciar o PC, não rode docker run de novo (ele tentaria criar outro com o mesmo nome e reclamaria). Use:

docker start nexus-postgres

run = criar do zero. start = ligar o que já existe. Os dados continuam lá por causa do volume. No Docker Desktop, o botão de play faz exatamente o mesmo que o docker start.

Becos e o que aprendemos

O terminal velho que não via o Java. Depois de instalar o JDK, o primeiro terminal jurava que java não existia. Quando um instalador mexe nas variáveis de ambiente, isso só vale pra terminais abertos depois. O que já estava aberto carregou o ambiente antigo. Lição: instalou algo que mexe no PATH, abre um terminal novo pra testar.

O npm bloqueado. Máquina nova chega com o PowerShell travando scripts. Assusta de cara, mas a correção é uma linha e é segura (mexe só no seu usuário). Vai acontecer toda vez que formatar e for usar npm.

A sobra de uma limpeza anterior. No pom.xml ainda havia a biblioteca OpenPDF, que o projeto não usa mais. Numa sessão passada o código que a usava saiu, mas a linha no pom ficou. Removemos agora. Detalhe fino: o jar já baixado continua no cache .m2 (inofensivo); remover do pom só garante que a próxima compilação não o inclui. "Limpei tudo" precisa incluir a receita, não só o código.

Pendente — trazer os dados

O banco subiu vazio. O Flyway criou a estrutura (as tabelas), mas os dados reais (notas, lançamentos) estão em outra máquina, onde o Postgres era instalado normalmente. Próximo passo: gerar um pg_dump na máquina antiga e restaurar dentro do container daqui.
