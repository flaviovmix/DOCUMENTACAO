Do Mês pro Dia

0:00 / 0:00

O seletor de período do dashboard do Financeiro já fazia "Últimos 12 meses", um mês da lista e um intervalo. Só que o intervalo era de mês a mês: você escolhia "de Fevereiro até Junho" e pegava meses inteiros. O Flávio queria escolher por dia (tipo "de 13/11/2025 até 22/02/2026"). Este making-of é só isso: trocar o intervalo de mês pra dia, no front e no back. Arquivos reais: FinanceMesSeletor.svelte, types.ts, FinanceDashboardController.java e FinanceDashboardService.java.

O que mudou

Antes, o painel "Intervalo" tinha dois dropdowns de mês e o filtro pegava meses inteiros. Agora ele tem dois seletores de data e o recorte é exato no dia. Nada de gráfico, nada de revisão: só o filtro.

A descoberta que encolheu a tarefa

Antes de mexer, fomos ver o quão fundo o "mês" estava cravado. A descoberta foi o contrário do esperado: os cards (entradas, saídas, saldo, limite, categorias) já eram calculados por intervalo de dias. O método que faz a conta, o buildDashboard, já recebia duas datas LocalDate (data com dia, mês e ano, ao contrário do YearMonth, que só tem mês e ano), e a consulta usava data BETWEEN inicio AND fim, que inclui as duas pontas.

O que prendia tudo no mês era só a borda: o front mandava o período como YYYY-MM (só ano e mês) e o controlador fazia YearMonth.parse. O plano ficou pequeno: front manda o dia (YYYY-MM-DD), controlador parseia como LocalDate. O resto já estava pronto.

O front, peça por peça

O tipo. O FinancePeriodo é uma união: ou { mes }, ou { de, ate }. O de/ate antes guardava YYYY-MM; agora guarda YYYY-MM-DD. A forma do tipo não mudou (continua string); mudou o que a string carrega.

O seletor. Os dois dropdowns de mês (StyledSelect) viraram <input type="date">, o calendário nativo do navegador, que já devolve YYYY-MM-DD. Com os dropdowns saíram a função abbrFor e a lista de meses abreviados que ela usava (código morto). Entrou a dmyFor, que monta o rótulo no formato brasileiro:

// Intervalo agora é por dia: rótulo 'DD/MM/AAAA'.
function dmyFor(value: string): string {
    if (!value) return '';
    const [y, m, d] = value.split('-');
    return `${d}/${m}/${y}`;
}

Aplicar. A aplicarIntervalo compara as duas datas como texto e troca se vierem invertidas. Funciona sem converter pra data porque o formato YYYY-MM-DD já ordena certo na comparação de string (ano, depois mês, depois dia, tudo com zero à esquerda):

function aplicarIntervalo() {
    if (!intDe || !intAte) return;
    // 'YYYY-MM-DD' compara lexicograficamente na ordem certa.
    const [de, ate] = intDe <= intAte ? [intDe, intAte] : [intAte, intDe];
    periodo = { de, ate };
    open = false;
}

O re-sync. Os inputs guardam intDe/intAte num estado próprio. Se você aplica um intervalo, clica num mês e reabre o painel, os inputs mostrariam o intervalo velho. Um $effect resolve: toda vez que o painel abre, re-sincroniza com o período atual. Ele só lê intervaloOpen e periodo (não os próprios inputs), senão digitar uma data dispararia o reset e apagaria o que você acabou de escrever:

$effect(() => {
    if (intervaloOpen) {
        intDe = 'de' in periodo ? periodo.de : '';
        intAte = 'de' in periodo ? periodo.ate : '';
    }
});

O back, peça por peça

O controlador. A mudança é quase boba: onde era YearMonth.parse virou LocalDate.parse. YearMonth só entende ano e mês; LocalDate entende o dia também. Como o front agora manda o dia, aceita numa boa (o import trocou junto):

if (de != null && !de.isBlank() && ate != null && !ate.isBlank()) {
    return dashboardService.getDashboard(user, LocalDate.parse(de), LocalDate.parse(ate), contaId);
}
return dashboardService.getDashboard(user, mes, contaId);

O service. O overload do intervalo recebia dois YearMonth e esticava o mês inteiro com de.atDay(1) / ate.atEndOfMonth(). Virou um overload de LocalDate: como já chega o dia exato, passa de/ate direto pro buildDashboard (que já somava por dia). Sumiu o atDay/atEndOfMonth; sobrou só o swap defensivo:

@Transactional(readOnly = true)
public FinanceDashboardResponse getDashboard(User user, LocalDate de, LocalDate ate, Long contaId) {
    if (de.isAfter(ate)) { LocalDate tmp = de; de = ate; ate = tmp; }
    return buildDashboard(user, de, ate, contaId, "");
}

Resumo

O FinancePeriodo passou a carregar dia no de/ate; o seletor trocou os dropdowns de mês por <input type="date"> e ganhou a dmyFor; o controlador trocou YearMonth.parse por LocalDate.parse; e o service trocou o overload de YearMonth (que esticava o mês) por um de LocalDate (que usa o dia direto). A coisa pra levar: antes de achar que uma mudança vai ser grande, veja o quão fundo a suposição antiga está cravada. Aqui o miolo já trabalhava por dia; a tarefa era só parar de arredondar pra mês na borda.
