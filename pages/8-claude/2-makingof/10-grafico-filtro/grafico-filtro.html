Gráfico que Segue o Filtro

0:00 / 0:00

No dashboard do Financeiro do Nexus tem um gráfico de linha (entrada, saída, saldo) e um seletor de mês no topo. O gráfico ignorava o seletor: mostrava sempre os últimos 12 meses até hoje. A gente ligou os dois e, de quebra, refez o seletor e generalizou o endpoint do backend. Arquivos reais: FinanceMesSeletor.svelte, FinanceHistoricoChart.svelte e FinanceDashboardService/Controller.java.

O que a gente queria

Você trocava o mês lá em cima, os cards mudavam, mas o gráfico ficava parado. Queríamos duas coisas. Em "Últimos 12 meses", o gráfico mostra os 12 meses até hoje, com o mês atual na ponta direita. Num mês específico, ele passa a mostrar 5 meses centrados naquele mês: dois antes, o escolhido no meio, dois depois. Assim dá pra ver o entorno do mês (se foi pico ou vale), em vez de um ponto solto.

Os conceitos

1. Janela de dados: ancorada no presente vs centrada

Pensa numa fileira longa de meses. O gráfico nunca mostra a fileira inteira, mostra um pedaço: uma janela. A pergunta é onde ela para. Ancorada no presente: a borda direita é sempre hoje e ela se estica pra trás (o "últimos 12 meses"). Centrada: você escolhe um mês e a janela fica em volta dele, com o mês no meio. Mesmo gráfico, mesma fileira; muda só onde a janela está parada.

2. Endpoint por intervalo, não "os últimos N"

Antes o servidor só sabia responder "me dá os últimos N meses até hoje". Isso atende a janela ancorada, mas não a centrada. Generalizamos: o servidor passou a aceitar um intervalo, um mês de início e um de fim. Quem pergunta decide as duas pontas. A ancorada vira "de 12 meses atrás até hoje"; a centrada vira "de 2 antes até 2 depois". O servidor só preenche o intervalo que pediram.

3. Reatividade: um estado, dois componentes

O mês selecionado mora num lugar só. O seletor escreve nele ($bindable), o gráfico lê dele (a prop mes). Quando o valor muda, o gráfico recalcula a janela sozinho e busca os dados de novo. Ninguém avisa o gráfico na mão; ele reage porque depende daquele valor.

Como montamos

O seletor. Estava com um item "Todos" mais catorze meses (incluindo um futuro), uma lista enorme. Agora: o primeiro item é Últimos 12 meses (valor vazio), depois só os 6 meses mais recentes, e no fim um "Exibir todos" que não é filtro, só expande a lista pra 13 meses. Um interruptor (showAll) alterna entre 6 e 13.

O backend. O método que ancorava em hoje virou um atalho: ele calcula as pontas e chama uma versão nova que recebe um intervalo de/ate (teto de 24 meses). Um chama o outro, sem duplicar a agregação. O controlador aceitou os dois parâmetros novos; sem eles, cai no modo antigo:

@GetMapping("/historico")
public List<FinanceHistoricoMesItem> historico(
    @RequestParam(required = false) String de,
    @RequestParam(required = false) String ate,
    @RequestParam(required = false, defaultValue = "6") int meses,
    @AuthenticationPrincipal User user
) {
    if (de != null && !de.isBlank() && ate != null && !ate.isBlank()) {
        return dashboardService.getHistorico(user, YearMonth.parse(de), YearMonth.parse(ate));
    }
    return dashboardService.getHistorico(user, meses);
}

O gráfico. Ganhou a prop mes. De um $derived sai a janela: null → modo legado (rolante, usado pela outra tela); '' → 12 meses até hoje; 'YYYY-MM' → 2 antes até 2 depois. O efeito pede os dados já com as duas pontas, e o título muda junto.

const range = $derived.by(() => {
    if (mes === null) return null;
    if (mes === '') { const a = nowYm(); return { de: ymAdd(a, -11), ate: a }; }
    return { de: ymAdd(mes, -2), ate: ymAdd(mes, 2) };
});

E a ligação: na página do Financeiro, o mês que o seletor escreve é o mesmo que entregamos pro gráfico. Trocou no seletor, o gráfico segue.

Decisões e becos

O fallback silencioso. Com tudo pronto, o gráfico mostrava 6 meses, não 12. A pista estava no número: 6 era o valor padrão do modo antigo. O backend não tinha sido reiniciado. O frontend já mandava de e ate, mas o servidor velho não conhecia esses parâmetros, ignorou (Spring ignora query params extras) e caiu no default meses=6. Não quebrou, não deu erro, só respondeu com o comportamento velho. Reiniciar resolveu. Lição: resultado "quase certo" depois de mexer no servidor, desconfie que o código novo nem subiu.

"Exibir todos" é só expandir. No começo eu tinha posto um item "Todos" (período inteiro) no fim da lista. O Flávio cortou: o "Exibir todos" serve só pra escolher qualquer mês disponível, não é um filtro. O agregado virou o primeiro item, "Últimos 12 meses".

Por que 5 meses no centrado. A ideia é ver o entorno: 2 antes e 2 depois bastam pra sacar se o mês foi pico ou vale, sem encher a tela.

Manter os 12 fixos com vazios. Com histórico curto, "Últimos 12 meses" mostra meses zerados. Dava pra cortar os vazios do começo, mas 12 é 12: o vazio onde não houve lançamento é informação honesta, não defeito. E a prop mes opcional preserva a outra tela (o dashboard inicial usa o gráfico do jeito velho, sem nem saber da mudança).

Resumo

O gráfico do Financeiro agora segue o seletor. Em "Últimos 12 meses" mostra a janela ancorada no presente (12 até hoje); num mês específico, a janela centrada (5 meses com o escolhido no meio). Pra isso o backend deixou de só saber "os últimos N" e passou a aceitar um intervalo de/ate, com o método antigo virando atalho do novo. A coisa pra levar: pensar em gráfico temporal como uma janela que desliza sobre uma fileira de meses. Onde ela para é só uma conta de duas pontas, e um servidor que aceita intervalo serve qualquer um desses casos.
