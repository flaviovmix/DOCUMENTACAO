Olá Mundo em React

0:00 / 0:00

O par do episódio Anatomia do Projeto Svelte: o mesmo "Olá Mundo" do Stack Web, antes feito em SvelteKit, agora em React. A tela é idêntica (só um título). O valor está na comparação: o que é conceito universal de frontend e o que é só jeito de cada ferramenta. Projeto real: PROJETOS/ESTUDO/STACK WEB/REACT/.

O que a gente queria

O Stack Web é o lab onde o Flávio aprende a stack do Nexus do zero. O Nexus é Svelte, então Svelte é o foco. Mas o plano tem uma parte de comparação: a mesma coisa feita nos dois, lado a lado, pra separar a ideia universal da sintaxe de cada ferramenta. No episódio anterior montamos o "Olá Mundo" em SvelteKit. Aqui a tarefa é refazer ele idêntico na tela, só que em React: um <h1>Olá Mundo</h1> e nada mais. React entra pra comparar e aprender, não pra produzir o Nexus.

Os conceitos

Quatro ideias separam os dois mundos.

1. Biblioteca vs framework

SvelteKit é um framework: o carro montado, já vem com rotas, estrutura de pastas e build decididos. React é uma biblioteca: cuida só de desenhar a tela. Não decide rota, nem pasta, nem deploy. É uma peça de Lego solta, ótima no que faz, mas você escolhe o que encaixar em volta. Por isso React quase nunca vem sozinho.

2. Vite, a peça em volta do React

A peça que juntamos ao React se chama Vite: é o gerador que cospe o projeto pronto (o papel do sv create no Svelte) e o servidor de dev que atualiza a tela ao salvar. O detalhe que fecha o círculo: o Vite é o mesmo motor que estava por baixo do SvelteKit no episódio anterior. Mesmo motor embaixo; o que muda é a camada por cima.

3. JSX e componente como função

No React o HTML mora dentro do JavaScript (isso é o JSX). Um componente é simplesmente uma função que devolve um pedaço de tela. A função App devolve o título e pronto, virou componente. No Svelte o HTML e o código moram em arquivos separados; no React moram juntos, dentro da função.

4. Como a página chega na tela

A maior diferença. No SvelteKit você não liga nada na mão: cria o arquivo +page.svelte com o nome certo e o framework acha por convenção. No React é o contrário, você liga explicitamente: o main.tsx pega o componente raiz e manda renderizar dentro de uma div com id root no HTML. É como plugar um aparelho na tomada: o React é o aparelho, o root é a tomada, o main é o fio que você conecta na mão.

Como montamos

Primeiro, gerar. Rodei o gerador do Vite com o modelo react-ts, dentro da pasta REACT/, ao lado da SVELTE/. Mesmo ambiente do irmão: TypeScript ligado e eslint junto. Depois npm install.

npm create vite@latest REACT -- --template react-ts
npm install

Aí veio o passo que o Svelte não teve: a faxina. O gerador do Vite não entrega um Olá Mundo limpo, entrega uma página de demonstração cheia (contador com useState, logos girando, textos, links, CSS e SVGs). A gente reduziu o App.tsx a três linhas, igual ao +page.svelte que era só o título:

function App() {
  return <h1>Olá Mundo</h1>
}

export default App

Tirei do main.tsx a linha que carregava o CSS da demo e apaguei o que sobrou sem uso (os dois .css, os SVGs dos logos, os ícones). Sobraram dois arquivos: o App (a tela) e o main (o fio que liga a tela no HTML):

import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import App from './App.tsx'

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>,
)

O index.html é curtinho: tem a div root vazia (o buraco) e o <script> que chama o main. O fluxo final é uma fila de quatro peças: o navegador abre o index, o index chama o main, o main manda o App renderizar dentro do root, e o App devolve o "Olá Mundo".

Decisões e becos

Por que Vite e não Next. No mundo React tem várias ferramentas pra montar o projeto (o Next é a mais famosa). Escolhemos o Vite por ser o mais simples pra um Olá Mundo e por ser o mesmo motor que já roda por baixo do Svelte. Não é tecnologia estranha: é a peça que o Flávio já viu, agora aparecendo na frente.

A faxina do boilerplate. No Svelte o gerador entregou um projeto quase vazio, só faltava o título. No React foi o oposto: apagar até sobrar o título. É uma diferença de cultura entre as ferramentas, e vale ter visto, porque todo projeto React novo cai nessa página de demonstração e precisa de limpeza.

A lição conceitual. A tela ficou idêntica, mas o caminho até ela é diferente. No Svelte a página aparece porque o arquivo tem o nome certo na pasta certa. No React aparece porque conectamos, na mão, a árvore num ponto do HTML. Nenhum é melhor: Svelte aposta em convenção (te dá mais pronto, pede menos decisão); React aposta em ser peça solta (te dá menos pronto, te deixa escolher).

Resumo

O mesmo Olá Mundo, agora em React. React é uma biblioteca que só desenha a tela; pra virar projeto juntamos o Vite, que é o mesmo motor do Svelte aparecendo na frente. O componente é uma função que devolve JSX, e a tela só aparece porque ligamos a árvore React no buraco root do HTML, na mão. A coisa pra levar: muito do que parecia "do Svelte" é do frontend inteiro (componente, TypeScript, montar a tela). O que muda de verdade é quem decide por você. Svelte decide mais; React deixa você decidir.
