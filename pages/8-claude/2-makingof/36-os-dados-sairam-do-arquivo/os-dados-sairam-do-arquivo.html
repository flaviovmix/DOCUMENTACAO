Os Dados Saíram do Arquivo

0:00 / 0:00

Etapa 5 da migração do site do Kaco pra Java: matar a fonte estática de dados. Até aqui, toda página lia de um arquivo no repo (kids-stories.js) que definia quatro globais na janela: KIDS_CATEGORIAS, KIDS_PERSONAGENS, KIDS_STORIES e KIDS_COLORIR. Neste capítulo esse arquivo vira um endpoint que escreve JavaScript: um controller monta os mesmos globais direto do banco, com o mesmo contrato (mesmas chaves), e por isso nenhum componente do site precisou mudar. E a vitrine da etapa anterior liga de verdade: anônimo recebe a amostra completa e o resto vem como stub trancado.

A contradição no próprio plano

O plano mandava duas coisas que não fecham juntas: trocar nas páginas o <script src="components/kids-stories.js"> pela URL nova, e manter o site estático (live-server) funcionando com o arquivo antigo até a virada final. Se as páginas apontam pro endpoint, o estático quebra — lá o endpoint não existe. Duas saídas foram postas na mesa: servir o JS gerado no mesmo caminho do arquivo antigo (controller tem precedência sobre arquivo estático no Spring — zero páginas tocadas), ou seguir o plano com a URL nova e aceitar a quebra. Decisão do dono: "pode quebrar por enquanto, já tenho um backup". URL nova: /js/kids-dados.js.

O controller que escreve JavaScript

KidsDadosController, um @RestController com um método só. O truque todo está no produces e no corpo: em vez de JSON pra alguém interpretar, ele responde código executável — pro navegador, era um script e continua sendo um script:

@GetMapping(value = "/js/kids-dados.js", produces = "application/javascript")
@Transactional(readOnly = true)
public ResponseEntity<String> dados(Authentication auth) throws Exception {
    boolean anonimo = auth == null || !auth.isAuthenticated()
            || auth instanceof AnonymousAuthenticationToken;
    // ... monta os 4 globais ...
    String js = "window.KIDS_CATEGORIAS = " + json.writeValueAsString(cats) + ";\n"
              + "window.KIDS_PERSONAGENS = " + json.writeValueAsString(pers) + ";\n"
              + "window.KIDS_STORIES = " + json.writeValueAsString(stories) + ";\n"
              + "window.KIDS_COLORIR = " + json.writeValueAsString(colorir) + ";\n";
    return ResponseEntity.ok()
            .cacheControl(CacheControl.maxAge(Duration.ofSeconds(60)))
            .contentType(MediaType.parseMediaType("application/javascript;charset=UTF-8"))
            .body(js);
}

Peças: o Authentication injetado decide anônimo vs logado (nulo, não autenticado ou AnonymousAuthenticationToken); o ObjectMapper do Jackson serializa cada global (aspas, acentos e caracteres especiais por conta dele — JSON concatenado na mão é bug esperando); o Cache-Control de 60s faz publicação nova aparecer em até um minuto; e o @Transactional(readOnly = true) mantém a transação aberta pras coleções lazy (cenas, personagens) carregarem dentro do método. Os caminhos saem prontos por um helper content() que prefixa /content/ no caminho relativo guardado no banco.

O stub trancado

O plano dizia "anônimo recebe só a amostra" e também "o resto aparece com cadeado na home". As duas juntas não fecham: sem receber o resto, a home não tem o que desenhar. A saída: história fora da vitrine vira stub sem conteúdo pra anônimo:

s.put("src", trancada ? "" : content(h.getAudioPath()));
s.put("cenas", trancada ? List.of()
        : h.getCenas().stream().map(c -> content(c.getImagemPath())).toList());
s.put("tx", trancada ? "" : nz(h.getTranscricao()));
if (trancada) s.put("trancada", true);

A capa fica (o card precisa dela, e /content/** já era público). Desenho de colorir fora da vitrine é diferente: nem vira stub — no colorir a arte É a experiência; mandar o caminho da imagem já entregaria o desenho pra pintar. História trancada sem cenas não entrega nada; desenho trancado com imagem entregaria tudo.

O front mínimo

Oito páginas trocaram uma linha de script; as sete de experiência ganharam também o kids-acesso.js (o modal, que antes só existia na home). No SecurityConfig, um bloco novo inverte a lógica da etapa 4 — a página é pública, os dados é que são filtrados:

// ETAPA 5 (D7): páginas de experiência viram públicas — quem é
// filtrado agora são os DADOS (/js/kids-dados.js manda só a amostra
// vitrine pra anônimo). Rotinas/emoções/família seguem trancadas.
.requestMatchers("/historias.html", "/historia.html", "/colorir.html",
        "/quebra-cabeca.html", "/quebra-cabeca-jogo.html",
        "/personagens.html", "/audio-historias.html").permitAll()

E o interceptor de cliques do kids-acesso.js, que na etapa 4 trancava qualquer .html, agora tranca por item: whitelist de páginas públicas + lookup no próprio global:

function itemTrancado(arquivo, params) {
    var id = arquivo === 'historia.html' ? params.get('id')
           : arquivo === 'quebra-cabeca-jogo.html' ? params.get('h')
           : null;
    if (!id) return false;   // sem item na URL (hubs, modo livre) = página aberta
    var story = (window.KIDS_STORIES || []).filter(function (s) { return s.id === id; })[0];
    return !story || !!story.trancada;
}

Não achou a história (nem stub veio) ou veio trancada? preventDefault() e o modal desce. Página pública sem item na URL navega normal; rotinas, emoções e família continuam caindo no modal.

O cadeado com blur

Card de item trancado ganha a classe hist-trancado e um <span class="hist-lock">🔒</span>. O pedido do blur veio na validação visual:

.hist-trancado .hist-head img {
    filter: blur(7px);
    transform: scale(1.08);   /* esconde as bordas lavadas do blur */
}

E um beco clássico de duplicação: o blur nasceu nos grids da home (builders inline) e o dono reportou que o hub do quebra-cabeça não borrava. Claro — os hubs usam o componente compartilhado kids-historias. A correção subiu pro componente (o card builder aceita trancada no item e troca o selo de status pelo cadeado); os hubs só passam a flag adiante.

A carga mínima

O banco estava vazio (a carga de verdade é a etapa 7). Um SQL descartável adiantou o essencial: 8 personagens, 5 histórias, 1 desenho — e as 43 cenas de uma tacada só, sem digitar caminho nenhum:

INSERT INTO cena (historia_id, ordem, imagem_path)
SELECT h.id, n, 'historias/' || h.slug || '/' || n || '.jpeg'
FROM historia h
JOIN (VALUES ('o-passeio-feliz', 9), ('o-gato-o-rato-e-o-pato', 7),
             ('kaco-anda-de-bicicleta', 8), ('kaco-vai-a-praia', 10),
             ('kaco-com-tio-thiago', 9)) AS q(slug, total) ON q.slug = h.slug
CROSS JOIN LATERAL generate_series(1, q.total) AS n;

Os vínculos história-personagem saíram de uma tabela VALUES com join por slug (nenhum id na mão). Vitrine marcada: O Passeio Feliz, O Gato o Rato e o Pato, e o desenho1.

A prova por curl

Anônimo: as duas histórias da vitrine completas, as outras três como stub (cenas: [], trancada: true). Logado exige o ritual do CSRF na mão: GET na página de login guardando cookie, extrair o token escondido do HTML, POST com e-mail + senha + token — e aí o mesmo endpoint respondeu as 5 histórias completas, sem a palavra trancada no corpo. As 7 páginas públicas: 200 pra anônimo; rotina, emoção e família: 302 pro login. Becos do caminho: a porta 8080 ocupada por um app antigo de outra sessão, e o banco recusando conexão depois de um reinício porque o Docker Desktop nem estava aberto.

Resumo

O arquivo estático virou um endpoint que escreve JavaScript com o mesmo contrato (mesmos globais, mesmas chaves) — por isso nenhum componente mudou. O Authentication decide entre o cardápio completo e a degustação; história trancada vira stub com capa mas sem conteúdo (a home desenha o cadeado com blur), desenho trancado nem aparece; o interceptor tranca por item, não por página; e a carga mínima com generate_series encheu o banco pra prova por curl. O kids-stories.js segue no repo aposentado em vida, até a virada final. Próximo capítulo: as 15 páginas clonadas de rotina e emoção começam a virar templates.
