O Kaco Parou de Se Repetir

0:00 / 0:00

O site do Kaco é estático: HTML, CSS e JavaScript puro, sem framework e sem build. Cada página é um arquivo solto. Com o tempo, copiar e colar virou rotina e o mesmo código foi parar em vários lugares. A cor azul da marca, o hex #2aa9e0, estava escrita mais de cem vezes. A sombra escura dos cartões repetia umas trinta. O menu do topo estava à mão em cada página, o hero estava copiado em cinco arquivos, e a lista das seis histórias vivia em três. A faxina foi tirar tudo isso, lendo o código peça por peça. O objetivo: mudar qualquer coisa num lugar só.

Cores e sombra num lugar só (os tokens)

No topo do style.css entrou um bloco :root, que é a raiz do documento. Quem define uma variável aí deixa ela disponível pra página inteira, por causa da cascata do CSS. Primeiro os acentos da marca, cada cor com um nome:

:root {
    --c-blue:   #2aa9e0;
    --c-green:  #22b07d;
    --c-amber:  #f59e0b;
}

Depois as categorias das histórias. Elas não repetem a cor, elas apontam pro acento:

--c-rotina:   var(--c-blue);
--c-aventura: var(--c-green);
--c-emocoes:  var(--c-amber);

Repara que --c-rotina não é um hexadecimal, é um var() pro --c-blue. A categoria rotina é azul porque aponta pro azul. Se o azul da marca mudar, muda só o --c-blue e a categoria acompanha sozinha.

A sombra tem um truque. O alpha (a transparência) muda de cartão pra cartão: às vezes .07, às vezes .14. Se a gente guardasse a sombra inteira numa variável, não dava pra variar o alpha. Então guardamos só os três números da cor, como um trio, e soltamos o alpha na hora de usar:

--shadow-rgb: 15, 39, 64;

/* uso: */
box-shadow: 0 10px 26px rgba(var(--shadow-rgb), .12);

Pra trocar os mais de cem hex pelos var() sem fazer na mão, rodamos um script. Aí veio o primeiro beco: se ele trocasse todo #2aa9e0 por var(--c-blue), trocaria também a própria definição no :root, virando --c-blue: var(--c-blue), uma variável apontando pra ela mesma, quebrada. A solução foi o script dividir o arquivo: o bloco de cima (as definições) ele não toca; só troca do reset pra baixo. No fim, cada cor de marca aparece uma vez (a definição) e o resto é var().

Uma decisão consciente: a gente não tokenizou tudo. Branco e preto puros ficaram como estão (são primitivos, não cor de marca), e os gradientes decorativos da home, que aparecem uma ou duas vezes, também ficaram hex. Tokenizar cada tonzinho criaria uma sopa de variáveis, que é outro tipo de bagunça. A regra: vira token o que é identidade da marca e o que repete muito; o resto fica.

Os dados das histórias (kids-stories.js)

A lista das seis histórias estava copiada em três páginas. Criamos o kids-stories.js com dois globais. Um é o mapa de categorias, com nome e cor (e a cor aponta pro token):

window.KIDS_CATEGORIAS = {
    rotina:   { nome: 'Rotina',   cor: 'var(--c-rotina)' },
    aventura: { nome: 'Aventura', cor: 'var(--c-aventura)' },
    emocoes:  { nome: 'Emoções',  cor: 'var(--c-emocoes)' },
};

O outro é a lista das histórias, cada uma com um id estável, título, categoria, imagem, duração, caminho do áudio e a transcrição:

window.KIDS_STORIES = [
    { id: 'passeio-feliz', t: 'O Passeio Feliz', cat: 'rotina',
      img: 'img-slide/0 (1).png', dur: '3:12', src: 'audio/o-passeio-feliz.m4a',
      tx: 'Kaco sai pra passear...' },
    // ...mais cinco
];

Agora as páginas de histórias e colorir não têm mais o array copiado. Elas leem o KIDS_STORIES e só colam o texto que é próprio delas. A página de colorir faz um .map() na lista e adiciona a descrição e o status:

itens: window.KIDS_STORIES.map(function (s) {
    return { t: s.t, cat: s.cat, img: s.img, status: 'em breve', desc: DESCS[s.id] };
})

Título, categoria e imagem vêm da fonte única; só o "em breve" e o textinho de pintar ficam na página. Adicionar uma sétima história agora é mexer num arquivo, não em três.

Menu e hero viram molde

O rodapé do site (kids-footer.js) já era um componente que se injeta sozinho. O menu do topo, não: estava à mão em toda página. Fizemos o kids-menu.js no mesmo molde. O arquivo inteiro é uma IIFE (função que se executa na hora) pra não sujar o escopo global, e os links vivem num array único:

var LINKS = [
    { href: 'index.html',     label: 'Início' },
    { href: 'historias.html', label: 'Histórias' },
    // ...
];

Uma função currentPage() pega o fim da URL com location.pathname.split('/').pop() pra saber em que página a gente está e marcar aquele link como ativo. O build() monta o nav como texto, com um guard no começo pra não duplicar, e põe o menu no topo da página:

if (document.querySelector('.menu')) return null; // guard: nao duplica
// ...
document.body.insertBefore(nav, document.body.firstChild);

Esse insertBefore é o detalhe: o rodapé usa appendChild pra cair no fim, o menu usa insertBefore pra subir pro começo. A parte de abrir e fechar (o hamburger no celular) saiu de dentro do script.js e entrou no próprio componente, numa função wire() em volta de um setOpen(open) que liga e desliga a classe open. Como o componente agora cuida do próprio menu, o trecho velho no script.js virou código morto e foi removido.

O hero seguiu a mesma ideia, com um detalhe a mais: ele é usado de dois jeitos. Nas páginas familia e áudio, ele se monta sozinho por configuração (a página põe um <div id="kidsHero"> e um objeto kidsHeroConfig). Mas a página de histórias tem outro componente (kids-historias) que precisa montar o mesmo hero por dentro. Pra não ter dois códigos do hero, o kids-hero.js expõe uma função pública:

window.KidsHero = { build: build };

Assim o kids-historias chama window.KidsHero.build() e recebe o cabeçalho pronto, em vez de ter a própria cópia. Um hero, montado de dois jeitos (por config automática ou chamando a função), com o código num lugar só. A onda em SVG e os emojis flutuantes que se repetiam em cinco arquivos agora vivem só dentro desse componente.

Decisões e becos

O beco da auto-referência no script de tokens já foi contado: trocar cor por variável sem pisar na própria definição. A lição é que substituição em massa precisa saber o que não tocar.

O player de áudio merece uma nota sobre a regra dos três. Ele tem mais de cem linhas de lógica (tocar, pausar, velocidade, volume) e a gente extraiu pro kids-player.js mesmo ele aparecendo numa página só. Por que não esperou a terceira cópia? Porque player é uma daquelas peças obviamente reutilizáveis, igual botão e menu: já nasce como componente. A regra dos três vale pra lógica de negócio específica, não pra esses blocos que qualquer projeto vai querer de novo.

E teve uma decisão de segurança, não de código. O site tinha quatro arquivos de rascunho (versões antigas de design, com link quebrado). O caminho óbvio era apagar. Mas esses quatro não estavam no controle de versão, então apagar não teria volta. Em vez disso, movemos pra uma pasta _rascunhos, fora da parte que vai pro ar. O site fica limpo e nada se perde. Apagar de verdade, só quando der pra desfazer.

Resumo

A faxina foi três movimentos. As cores e a sombra viraram tokens num :root único, com as categorias apontando pros acentos e a sombra guardada como trio de números pra variar o alpha. Os dados das histórias viraram fonte única no kids-stories.js, e as páginas passaram a só configurar. E o menu e o hero viraram componentes que se injetam sozinhos, com o hero exposto por uma função pra outro componente reusar. Por cima, a regra que guia tudo: procurar antes de copiar, extrair antes de clonar, e só abstrair de verdade na terceira cópia. Tudo isso é código que serve qualquer site estático, não só o do macaquinho.
