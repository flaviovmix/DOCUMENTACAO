Salvando para Sempre

0:00 / 0:00

No módulo de orçamento do Nexus, editar um orçamento e clicar em Salvar deixava o botão preso em "Salvando..." pra sempre: o modal não fechava e nenhum erro aparecia na tela. A causa estava longe de onde o problema dava as caras. Arquivos reais: OrcamentoService.java (backend), OrcamentoDetailModal.svelte e api.ts (frontend).

O que a gente queria

Era pra ser banal: abrir um orçamento que já existe, mudar alguma coisa, clicar em Salvar. Só que, ao clicar, o botão virava "Salvando..." e ficava ali, congelado. O modal não fechava, a tela não mudava, e o pior, nenhuma mensagem de erro. Esse é o tipo de bug mais chato: um erro na tela ao menos te diz por onde começar; aqui não havia pista. O episódio conta como saímos do "nada acontece" até a causa, que estava num lugar bem diferente de onde o problema aparecia.

Os conceitos

1. Quando o banco dá o id de uma coisa nova

Pensa numa fila de senha. Quando você cria uma informação nova no programa (um item de orçamento), ela nasce sem número. O id (1, 2, 3...) não é o programa que inventa: é o banco de dados que dá, e só no momento em que a linha é de fato gravada. É igual ao número da senha que você só recebe ao passar no caixa. A ferramenta que faz a ponte entre objeto e tabela (o Hibernate) junta as gravações e manda pro banco num momento que ela escolhe (o flush). Então existe uma janela: o item já foi criado na memória, mas ainda não foi gravado, e por isso o id dele ainda é null.

2. Lista com etiqueta única no front

Pra desenhar uma lista, o Svelte pede uma etiqueta única por item (o (it.id) do {#each}), pra saber quem é quem quando a lista muda. É como dar um crachá de nome pra cada pessoa numa fila. Regra de ferro: dois crachás iguais e ninguém sabe quem é quem. Se duas etiquetas são iguais, o Svelte não desenha "mais ou menos", ele para tudo e acusa erro (each_key_duplicate). Aqui a etiqueta era o id do item.

3. O erro aparece longe de onde nasce

O conceito mais útil do episódio. Um problema pode dar as caras num lugar que não tem nada a ver com onde começou. O sintoma estava na tela (botão travado, que é front), então o reflexo é mexer no front. Mas a causa estava no servidor. Tratar o sintoma no lugar errado é como o médico que ataca a dor de cabeça quando o problema é o óculo errado: horas no lugar errado.

Como a gente caçou

Desconfiamos do banco. Primeira teoria: um travamento. Talvez uma gravação anterior estivesse pendurada, segurando a linha do orçamento, e o novo Salvar esperava ela soltar pra sempre. Faz sentido. Mas teoria não se confirma com palpite, e sim com evidência: fomos direto no Postgres perguntar se havia consulta presa ou linha travada. Nenhuma. Banco limpo, teoria descartada.

Olhamos o servidor. Backend no ar e respondendo. O log mostrava que, ao salvar, ele rodou as gravações certinho, sem nenhuma exceção. Do ponto de vista do servidor, salvar funcionou. Era o que mais confundia: "salvar" soa como coisa de servidor, e o servidor jurava ter feito o trabalho dele.

A virada: o console do navegador. Atrás do F12, em vermelho, a frase que resolveu tudo: "chave duplicada, valor null, nos índices 0 e 1". Em português claro: a lista de itens tinha pelo menos dois itens com a mesma etiqueta, e essa etiqueta era null. As peças se encaixaram com os conceitos 1 e 2.

Uncaught Svelte error: each_key_duplicate
Keyed each block has duplicate key `null` at indexes 0 and 1
    in OrcamentoDetailModal.svelte

Por que os ids vinham null. Ao salvar uma edição, o servidor faz algo simples e meio bruto: em vez de descobrir o que mudou item por item, ele joga fora a lista inteira e recria do zero. Itens recém-criados, todos sem id. E logo em seguida montava a resposta pra mandar de volta pra tela, só que antes de gravar no banco. Resultado: resposta com todos os itens de id null. A tela recebia, tentava desenhar, via duas etiquetas null e abortava o desenho no meio. O desenho abortado é o que congelava o botão: o "Salvando..." está amarrado a uma chavinha que voltaria ao normal quando a resposta chega, mas a tela quebrou bem na hora de redesenhar, antes de mostrar o botão de volta.

Decisões e becos

A correção foi de uma linha. No servidor, antes de montar a resposta, trocamos um "salva quando der" por um "salva agora": de save (que adia a gravação) pra saveAndFlush (que força o banco a gravar naquele instante). Gravou agora? O banco já deu os números, e os itens da resposta saem com id de verdade. Etiquetas diferentes, a tela desenha, o modal fecha.

replaceItems(o, req.items());     // joga fora a lista e recria (itens sem id)
orcamentos.saveAndFlush(o);       // FIX: grava agora, ids viram reais (antes era .save)
return OrcamentoResponse.from(o); // antes, isto rodava com itens id=null

O toque de fora. Repara que a correção foi no backend, e o sintoma estava no front. O instinto, vendo um botão travado, é consertar a tela ("põe um prazo na chamada, troca a etiqueta da lista"). Isso até evitaria a tela travar, mas trataria só o sintoma: a causa de verdade (o servidor mandando id null) continuaria viva e o orçamento voltaria errado. Quem já viu mais bug pergunta primeiro "de onde isso nasce?" antes de mexer em "onde isso aparece".

Por que dois nulls e não um. Se fosse um item só, a etiqueta null nem daria conflito (uma sozinha não colide com ninguém). O bug só apareceu porque o orçamento tinha vários itens, todos sem id ao mesmo tempo. Bug que só aparece com dois ou mais é traiçoeiro: num teste com um item só, parece tudo bem.

Fragilidades anotadas (não consertadas agora). A chamada de salvar no front não tem prazo máximo: se um dia o servidor de fato travar, dá o mesmo "pra sempre" por outro motivo. E usar o id como etiqueta da lista é frágil justamente pra item recém-criado. As duas valem um endurecimento depois, mas a correção de raiz foi a do servidor.

Resumo

O "Salvando..." travava pra sempre não porque o servidor demorava, mas porque a tela quebrava ao desenhar a lista de itens. A lista usava o id como etiqueta única, e o servidor devolvia os itens com id null, porque montava a resposta antes de gravar (e id de coisa nova só nasce na gravação). Dois nulls viravam duas etiquetas iguais, e isso aborta o desenho. A correção foi forçar a gravação antes de montar a resposta (saveAndFlush no lugar de save). A coisa pra levar: o erro aparece longe de onde nasce. O atalho não foi adivinhar, foi seguir a evidência (banco limpo, servidor sem erro, console do navegador apontando a frase exata).
