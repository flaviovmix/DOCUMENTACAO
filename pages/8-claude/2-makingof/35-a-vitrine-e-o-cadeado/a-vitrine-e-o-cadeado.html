A Vitrine e o Cadeado

0:00 / 0:00

Etapa 4 da migração do site do Kaco pra Java: o login, requisito central do projeto. O plano original dizia "tudo autenticado", mas a decisão de produto mudou o desenho antes do código: a home vira vitrine pública com uma amostra por categoria (1-2 histórias, 1-2 desenhos, quebra-cabeça usando as histórias da amostra), o resto ganha cadeado, e quem clica no cadeado não vai pra tela de login — um modal desce na própria página, com o fundo desfocado, pedindo nome e e-mail. As decisões viraram registro no plano (D7 e D8) e uma migration nova, a V3: flag vitrine em historia e desenho_colorir + tabela solicitacao_acesso (status PENDENTE / APROVADA / RECUSADA).

A ponte com a tabela usuario

O Spring Security consulta uma interface no login, o UserDetailsService. A nossa implementação tem um método só: recebe o e-mail digitado, busca com o findByEmail que nasceu na etapa anterior, e devolve o pacote que o Security entende:

@Override
public UserDetails loadUserByUsername(String email) throws UsernameNotFoundException {
    Usuario usuario = usuarios.findByEmail(email)
            .orElseThrow(() -> new UsernameNotFoundException("Usuário não encontrado: " + email));
    return User.withUsername(usuario.getEmail())
            .password(usuario.getSenhaHash())
            .roles(usuario.getPapel().name())   // ADMIN ou FAMILIA vira ROLE_
            .disabled(!usuario.isAtivo())
            .build();
}

Só de existir esse bean, o usuário em memória que o Spring gera por padrão morre. O par dele é o PasswordEncoder devolvendo BCryptPasswordEncoder — e aqui tem um detalhe bonito: o admin inicial foi semeado na V2 pelo próprio Postgres, com o crypt() da extensão pgcrypto. O Spring valida aquele hash numa boa: BCrypt é BCrypt dos dois lados. O banco gerou, o Java confere.

A corrente de regras

O SecurityConfig da etapa 2 era uma linha (tudo liberado). Foi reescrito como uma corrente avaliada na ordem — a primeira regra que casa com a URL decide:

.authorizeHttpRequests(auth -> auth
    .requestMatchers("/", "/index.html", "/favicon.ico",
            "/css/**", "/js/**", "/components/**", "/assets/**",
            "/content/**", "/colorir/**").permitAll()          // a rua (vitrine)
    .requestMatchers("/login", "/solicitar-acesso", "/api/sessao").permitAll()
    .requestMatchers("/admin/**").hasRole("ADMIN")             // o escritório
    .anyRequest().authenticated())                             // a loja, só com chave
.formLogin(login -> login
    .loginPage("/login")
    .usernameParameter("email")      // o padrão do Spring é username/password
    .passwordParameter("senha"))
.rememberMe(remember -> remember
    .key(rememberMeKey)              // assina o cookie; chave vem do application.yml
    .tokenValiditySeconds((int) Duration.ofDays(90).toSeconds()))

O remember-me tem validade de 90 dias porque o público é família: sessão não pode expirar toda hora. O logout aceita link simples (GET), porque o botão Sair mora num menu estático sem token CSRF — risco assumido e anotado (pior caso: alguém força um logout). A tela de login em si é Thymeleaf com a cara do site, importando o base.css e usando os tokens; o th:action injeta o token CSRF sozinho.

O modal que desce

kids-acesso.js é componente novo no padrão da casa: injeta o próprio CSS, monta o overlay e expõe window.kidsAcesso.abrir(). O visual pedido: fundo com backdrop-filter: blur(6px) e o cartão descendo por keyframes:

@keyframes kidsAcessoDesce {
    from { transform: translateY(-60px); opacity: 0; }
    to   { transform: translateY(0);     opacity: 1; }
}

Dentro do form, além de nome e e-mail, mora o campo armadilha (honeypot): um input chamado site, posicionado fora da tela, invisível pra gente. Robô de spam preenche tudo — inclusive ele — e se entrega. O envio é fetch POST com JSON; sucesso troca o form pela mensagem de "pedido enviado". No live-server (sem app por trás) o fetch falha com mensagem amigável.

O endpoint público com defesa própria

O /solicitar-acesso ficou fora do CSRF de propósito (quem posta é um form estático, sem token). Endpoint público sem CSRF precisa de defesas próprias, e são três:

if (pedido.site() != null && !pedido.site().isBlank()) {
    return ok(); // bot no honeypot: FINGE sucesso e não grava
}
if (estourouLimite(request.getRemoteAddr())) {
    return ResponseEntity.status(HttpStatus.TOO_MANY_REQUESTS)
            .body(Map.of("ok", false, "mensagem", "Muitos pedidos seguidos. ..."));
}
if (!solicitacoes.existsByEmailIgnoreCaseAndStatus(pedido.email(), StatusSolicitacao.PENDENTE)) {
    // grava; e-mail que já tem pedido PENDENTE responde o MESMO sucesso
}

O rate limit é um mapa em memória de IP pra fila de timestamps: mais de 3 pedidos em 10 minutos, resposta 429. E o pedido repetido é idempotente: não duplica no banco e responde igual, sem vazar quem já pediu. A validação é Bean Validation num record, com mensagens em português no tom do site.

O cadeado que abre o modal

Pro clique trancado abrir o modal em vez de navegar, o front precisa saber se tem alguém logado. Nasceu o /api/sessao (público, responde {autenticado, papel}), e no kids-acesso.js o interceptor: se anônimo, escuta os cliques do documento na fase de captura; link .html que não seja o index, preventDefault() e abre o modal.

document.addEventListener('click', function (e) {
    var link = e.target.closest ? e.target.closest('a[href]') : null;
    if (!link || link.target === '_blank') return;
    var arquivo = new URL(link.href, location.href).pathname.split('/').pop();
    if (!/\.html$/i.test(arquivo) || arquivo === 'index.html') return;
    e.preventDefault();
    abrir();          // o modal desce AQUI, sem sair da página
}, true);

Logado, nada é interceptado; no live-server o fetch da sessão falha e o site segue normal; e o redirect do servidor continua como rede de segurança pra URL digitada na mão. O kids-menu faz a mesma consulta e, só quando autenticado, ganha "Trocar senha" e "Sair" — a página /trocar-senha exige a senha atual, e a senha temporária do admin morre nela.

O beco: a arte do colorir sumiu

Teste visual e o relato: o card do colorir perdeu a arte, mas o quebra-cabeça aparecia normal. Diagnóstico com dois curls no mesmo arquivo: logado, 200 image/jpeg; anônimo, 302 pro login. A causa: as imagens das histórias e do quebra-cabeça moram em content/ (público na corrente), mas as artes do colorir moram em site/colorir/ — que caía no anyRequest().authenticated(). Correção de uma linha: /colorir/** entra na lista pública, com anotação no plano de avaliar mover essas artes pra content/ de vez na etapa da carga.

O resultado

Bateria de curl nos três perfis. Anônimo: home 200, rotina 302 pro login, api/sessao dizendo falso. FAMILIA: loga, navega no site inteiro, /admin devolve 403. ADMIN: loga, /admin 200, cookie remember-me no pote. O pedido de acesso: válido grava PENDENTE, repetido não duplica, armadilha finge sucesso, quarto seguido toma 429. E no navegador, o modal descendo com o fundo desfocado, na própria página.

Resumo

A etapa 4 em seis movimentos: a decisão de vitrine antes do código (V3 com flag e tabela de solicitação); o UsuarioDetailsService ligando o Security na tabela usuario com o BCrypt validando o hash que o Postgres semeou; a corrente de regras separando rua, loja e escritório; o modal kids-acesso descendo com blur e campo armadilha; o endpoint público com honeypot, rate limit e resposta idempotente; e o beco da arte do colorir resolvido comparando o mesmo curl logado e anônimo. Próximo capítulo: o endpoint de dados que gera o kids-dados.js do banco e liga a vitrine de verdade.
