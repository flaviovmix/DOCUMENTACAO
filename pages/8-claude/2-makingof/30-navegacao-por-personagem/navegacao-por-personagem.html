Navegação por Personagem

0:00 / 0:00

O site do Kaco tem uma página de histórias (cada história é um livrinho de imagens) e uma seção de personagens na home, com a turma: Kaco, Mãe, Gato, Rato, Pato. A gente quis duas navegações novas. Na home, o pai filtra os personagens por condição (o site é pra criança com necessidade especial, então ele clica em "Autismo" ou "Paralisia" e vê quem representa o filho dele). E uma página pra filtrar as histórias por personagem: escolhe o Gato e vê só as histórias em que o Gato aparece. O problema é que nada disso existia no dado, a lista de histórias não sabia quem aparecia em cada uma. Então o primeiro passo foi criar esse vínculo.

O vínculo história-personagem

O kids-stories.js é a fonte única: a lista das histórias e o mapa de categorias moram nele. A gente somou duas coisas. Em cada história, um campo personagens, que é a lista dos apelidos (os slugs) de quem aparece:

{
    id: 'o-passeio-feliz', t: 'O Passeio Feliz', cat: 'emocoes',
    cenas: 9,
    personagens: ['kaco', 'mae'],   // quem aparece na história
    // ...
}

E um mapa novo, KIDS_PERSONAGENS, que pra cada slug guarda o nome de exibição, o ícone e a cor:

window.KIDS_PERSONAGENS = {
    kaco: { nome: 'Kaco', icone: 'img/icone-kaco.png', cor: '#a9744f' },
    mae:  { nome: 'Mãe',  icone: 'img/icone-mae.png',  cor: '#e0729e' },
    // ...
};

Esse mapa é a fonte dos botões de filtro da página de personagens, e o campo personagens das histórias aponta pros mesmos slugs. Com isso, o dado finalmente sabe quem está em cada história.

Filtro por condição na home

Na home, a seção de personagens é um carrossel: uma fileira de miniaturas em cima e um palco que desliza embaixo. O slide é por margin-left negativo, e ele usava a posição original do personagem na lista. Primeiro a gente colou a etiqueta: um data-cond em cada miniatura, e os botões "Todos / Autismo / Paralisia" em cima.

Aí veio o beco. Esconder um personagem com display:none tira ele da fileira, e os que sobram encostam um no outro. Mas o slide ainda contava a posição antiga, então clicar num personagem visível deslizava pro lugar errado. A correção foi parar de usar a posição original e usar a posição entre os visíveis: a gente guarda numa lista order os índices de quem passou no filtro, e desliza pelo lugar dentro dela.

const pos = order.indexOf(domIndex);   // posição ENTRE OS VISÍVEIS
track.style.marginLeft = '-' + (pos * 100) + '%';

Quem faz a faxina é o applyFilter: percorre as miniaturas, compara o data-cond com o filtro clicado, liga ou desliga o display da miniatura e do painel (com um ajudante show), e monta o order. No fim, se ninguém bate, esconde o palco e mostra "Ainda não temos personagens nessa categoria"; se tem gente, seleciona o primeiro.

function applyFilter(cond) {
    order = [];
    thumbs.forEach((t, i) => {
        const match = cond === 'all' || (t.dataset.cond || 'autismo') === cond;
        show(t, match);
        show(panels[i], match);
        if (match) order.push(i);
    });
    const tem = order.length > 0;
    show(stage, tem);
    show(vazio, !tem);
    if (tem) select(order[0]);
}

A página reusa o componente

A página de histórias já usava o kids-historias.js, que monta o cabeçalho, os filtros e a grade de cards. A página nova precisava do mesmo desenho, filtrando por personagem em vez de categoria. Em vez de clonar a página, a gente deu dois modos pro componente, escolhidos por um parâmetro:

const filtros = cfg.filtros;             // presente = modo personagem
const chipDefs = filtros || categorias;  // de onde saem os botões de cima

Cada card ganha uma etiqueta com as chaves que ele casa. No modo categoria isso é só a categoria; no modo personagem, a lista de personagens:

const chavesCard = filtros ? (h.filtros || []) : (h.cat ? [h.cat] : []);
a.dataset.filtros = chavesCard.join(' ');

E o filtro virou um teste de pertencer à lista, em vez de igualdade. Antes era "a categoria do card é igual ao filtro"; agora é "a etiqueta do card contém o filtro":

const chaves = (c.dataset.filtros || '').split(' ');
c.style.display = (f === 'all' || chaves.indexOf(f) !== -1) ? '' : 'none';

Um detalhe visual: no modo personagem os botões não são pílulas de texto, são as mesmas miniaturas redondas da home (foto e nome), usando o ícone e a cor do KIDS_PERSONAGENS, com um "Todos" (estrela) na frente. A grade embaixo continua mostrando a categoria de cada história como selo; só os botões de cima mudaram. A personagens.html em si é curtinha: passa o KIDS_PERSONAGENS como filtros e joga o personagens de cada história como etiqueta.

Ligação por URL

Faltava o atalho: clicar no Kaco na home e cair na página já filtrada nele. O botão "Ver histórias" de cada personagem virou um link com parâmetro:

<a class="btn-saiba-mais" href="personagens.html?p=kaco">Ver histórias</a>

E o componente, depois de montar os botões, lê o parâmetro e clica no botão certo sozinho. Repara que ele não reimplementa o filtro: acha o botão e dá um clique de mentira, e o clique de verdade já faz o resto. Tem uma trava: só age se a chave existir no chipDefs, pra um valor estranho na URL não quebrar a busca.

const inicial = new URLSearchParams(location.search).get('p');
if (inicial && chipDefs[inicial]) {
    const alvo = chips.querySelector('[data-filter="' + inicial + '"]');
    if (alvo) alvo.click();
}

Decisões e becos

O beco do slide vale a pena guardar: esconder elemento com display:none quebra qualquer coisa que dependa da posição na lista. A correção é contar a posição entre os visíveis, não a original.

Uma decisão pragmática: o KIDS_PERSONAGENS repete nomes que já estão na seção de personagens da home. Em tese é duplicação. A gente aceitou de propósito, porque transformar a seção inteira em algo gerado por JavaScript era uma obra muito maior do que a tarefa pedia. Ficou um mapa pequeno e separado, com a anotação de que os dois andam juntos. Nem toda duplicação vale a obra de remover na hora.

E tem o lado da ferramenta: todo esse vínculo não é pra preencher na mão. O comando que publica histórias no site já olha quem aparece nas cenas, grava a lista de personagens e registra cada personagem novo no KIDS_PERSONAGENS. A página e o componente são código fixo; o dado que os alimenta nasce pronto.

Resumo

A navegação por personagem foi quatro movimentos. O vínculo no kids-stories.js (campo personagens + mapa KIDS_PERSONAGENS). O filtro por condição na home, com a etiqueta data-cond e a correção do slide pra contar a posição entre os visíveis. A página personagens.html, que reusa o componente de histórias num segundo modo, filtrando por lista em vez de igualdade e trocando as pílulas por miniaturas. E a ligação por URL, com o "Ver histórias" abrindo a página já filtrada por um clique automático. Por cima, a regra: reusar o componente com um parâmetro a mais, em vez de clonar a página.
