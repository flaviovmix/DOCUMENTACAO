As Fábricas

0:00 / 0:00

Episódio focado no código, o mais denso da trinca. Par do conceitual Tudo num Lugar Só: a gente lê os três arquivos das fábricas (createListStore, ModalStore e o crud) linha por linha, pra entender como uma função consegue fabricar o código que antes era copiado.

O que esse código faz

Três arquivos pequenos, mesma dor: código copiado. O createListStore resolve a lista que toda área carregava na mão (itens, carregando, erro, buscar). O ModalStore resolve o miolo que todo modal repetia (o "tá aberto", o fechar, a limpeza). O crud resolve o quinteto de servidor (listar, pegar, criar, editar, excluir) copiado em dezessete recursos. Todos: uma função (ou classe) que fabrica a peça, em vez de a peça ser copiada.

O conceito mínimo: generic

Uma ideia, porque aparece nos três. Quando uma função serve pra vários tipos de dado (lista de tags, de tarefas, de clientes), você não quer uma versão pra cada. O generic é um tipo coringa: você põe uma letra (normalmente T, de type) entre < e >, e ela vira um espaço em branco que quem chama preenche. createListStore<T> é "uma lista de alguma coisa, você me diz qual quando usar".

Lendo o código: createListStore

A assinatura. export function createListStore<T>(apiFn: () => Promise<T[]>). O export deixa outros arquivos importarem. O <T> é o generic. O parâmetro apiFn tem tipo "uma função que não recebe nada e devolve uma Promise de uma lista de T" (Promise = algo que chega do servidor mais tarde, não na hora). Resumindo: "me dá uma função que busca a lista lá fora, eu monto a store em cima".

export function createListStore<T>(apiFn: () => Promise<T[]>) {
  let items = $state<T[]>([]);
  let loading = $state(false);
  let error = $state<string | null>(null);

Os três estados. O $state é o rune de reatividade do Svelte 5: marcar a variável com $state faz a tela que a usa se redesenhar sozinha quando ela muda. items começa lista vazia, loading começa false, error é string | null (começa null, "sem erro ainda").

O ponto que pega de surpresa: o getter. A função devolve um objeto, mas o retorno não é items: items. É get items() { return items } — um getter. Por quê? Se devolvesse o valor direto, quem recebesse pegaria uma foto congelada do valor naquele instante e nunca mais veria a lista mudar. Com o getter, toda vez que alguém lê .items, a função roda de novo e devolve o valor atual. Por isso a regra de quem usa: sempre store.items, nunca desestruturar (desestruturar volta a pegar a foto congelada).

return {
  get items()   { return items; },    // getter, nao "items: items"
  get loading() { return loading; },
  get error()   { return error; },
  async refresh() {
    loading = true; error = null;
    try { items = await apiFn(); }
    catch (e) { error = e instanceof Error ? e.message : 'erro'; }
    finally { loading = false; }
  },
  clear() { items = []; error = null; }
};

O refresh. O async marca que tem espera dentro. Liga loading, zera error, abre try. A linha principal: items = await apiFn() — o await espera a promessa chegar; o resultado cai em items e a tela redesenha (porque é $state). Se der ruim, o catch pega a mensagem; o finally roda dando certo ou não e desliga o loading.

O rodapé. Store com lógica própria (polling, get de um item) não reescreve essa base: ela compõe por cima, com Object.assign(createListStore(...), { extras }), e assim os getters da base continuam vivos. A fábrica cobre o caso base; o especial soma em cima.

Lendo o código: ModalStore

Mais curto, e tem o melhor beco. export class ModalStore — aqui é uma classe (não função), porque os modais vão estender essa base, herdar dela. Dentro: open = $state(false), o único estado que todo modal tem em comum, começa fechado. O hide() faz duas coisas: baixa o open e chama this.reset(). E protected reset() {} — protected quer dizer que só a própria classe e as filhas chamam; está vazio na base de propósito (a base não sabe os campos da filha; cada modal reescreve o reset pra limpar os dele).

export class ModalStore {
  open = $state(false);
  hide() { this.open = false; this.reset(); }
  protected reset() {}   // a filha limpa os campos dela aqui
}

O beco (a parte que mais ensina): por que o show() NÃO está na base. Parece o lugar óbvio pra ele. O problema: cada modal abre de um jeito — um com show(id), outro com show(opts), outro com show() puro. Se a base declarasse um show() sem parâmetro e a filha declarasse show(id), o TypeScript reclamaria de sobrescrita incompatível ("esperava 1+ argumentos, recebeu 0"). A solução foi recuar: a base centraliza só o que é de fato igual (open, hide, reset); o show, que varia, cada filho declara o seu (e no fim faz this.open = true).

class MemberModal extends ModalStore {
  memberId = $state<number | null>(null);
  show(id: number) { this.memberId = id; this.open = true; }
  protected reset() { this.memberId = null; }
}

A lição: a base puxa pra dentro o que é igual; o que muda fica de fora. Forçar o igual onde não é igual quebra na cara do compilador.

Lendo o código: o crud

O menor e mais redondo. function crud<T, CreateReq = Partial<T>, UpdateReq = CreateReq>(base: string) — três generics. O T é a coisa (ex: uma Tag). CreateReq é o formato do que você manda pra criar, e o = Partial<T> quer dizer "se você não disser, assumo um T parcial". UpdateReq por padrão é igual ao de criar. O parâmetro de verdade é um só: base, o começo do endereço (ex: /tags).

function crud<T, CreateReq = Partial<T>, UpdateReq = CreateReq>(base: string) {
  return {
    list:   () => request<T[]>(base),
    get:    (id: number) => request<T>(`${base}/${id}`),
    create: (body: CreateReq) => request<T>(base, { method: 'POST', body }),
    update: (id, body: UpdateReq) => request<T>(`${base}/${id}`, { method: 'PUT', body }),
    remove: (id: number) => request<void>(`${base}/${id}`, { method: 'DELETE' })
  };
}

O retorno é um objeto com as cinco funções. Repare que nenhuma fala com a rede sozinha: todas chamam o mesmo request, a função central que já cuida do token, do cabeçalho e do erro. O remove devolve request<void> (apagar não retorna dado). O endereço de get/update usa template literal (crase + ${...}) pra montar /tags/5. A fábrica não reinventa a chamada, só monta as cinco rotas em cima do request que já existia.

const tagsCrud = crud<Tag, TagRequest>('/tags');   // uma linha = 5 operacoes
const tasksCrud = crud<Task, TaskRequest>('/tasks');
// ...17 recursos. list/get fora do padrao (query string) fica a mao ao lado.

Decisões no código

  • A regra dos três: a fábrica cobre o caso base, e o especial compõe por cima, nunca reimplanta. No createListStore é o Object.assign; no crud é o método à mão ao lado; no ModalStore é o show próprio de cada filho.
  • Os getters não são firula: são obrigatórios pela reatividade do Svelte 5. Devolver o valor direto congelaria a lista. Por fora, store.items parece propriedade comum; por dentro é uma função rodando toda vez.
  • O show fora da base: foi a tentativa que o compilador barrou, e o recuo foi o certo. Base magra (só o comum) é mais saudável que base inchada tentando prever toda assinatura de abrir.

Resumo

Três fábricas, mesma ideia: parar de copiar. createListStore é uma função com generic T que devolve items/loading/error por getter (pra não congelar a reatividade) mais refresh/clear. ModalStore é uma classe base com open/hide/reset, e o show fica de fora porque cada modal abre com assinatura própria. O crud é uma função com três generics que monta as cinco rotas REST em cima do request central, e cada recurso vira uma linha. O fio comum: a fábrica cobre o caso base, o especial soma por cima.
