Nome Igual, Banco Diferente

0:00 / 0:00

No Financeiro do Nexus você cadastra contas: a conta de um banco, o dinheiro na carteira, o cartão. O Flávio queria duas contas correntes chamadas "FLÁVIO", uma no Inter e outra no Itaú, com o banco diferenciando as duas nos botões de filtro. Mas o sistema barrava: nome de conta já em uso: FLÁVIO. A gente afrouxou a regra de duplicata. Arquivos reais: a migration V83__finance_contas_unique_per_banco.sql, o FinanceContaRepository.java e o FinanceContaService.java.

O que a gente queria

Cada conta tem um nome, um tipo (corrente, poupança, dinheiro, cartão) e, opcionalmente, um banco. A regra de duplicata olhava só nome + tipo: duas "FLÁVIO" corrente eram a mesma coisa pro sistema, mesmo com bancos diferentes. Por isso a segunda não entrava. A gente queria deixar o mesmo nome repetir, desde que o banco mude.

Os conceitos

1. A chave de identidade

Em todo cadastro existe um conjunto de campos que o sistema usa pra decidir se dois registros são "a mesma coisa". Se esse conjunto bate, é duplicata, e ele bloqueia. Aqui a chave era nome + tipo. A mudança do episódio é uma mudança nessa chave: a gente acrescentou o banco. Agora é nome + tipo + banco. "FLÁVIO corrente Inter" e "FLÁVIO corrente Itaú" têm a mesma dupla nome-tipo, mas banco diferente, então a chave não bate e as duas convivem. É como duas pessoas com o mesmo primeiro nome: só o nome não identifica, você olha o documento. O banco virou o documento da conta.

2. O nulo que conta como igual

Banco é opcional, uma conta pode não ter banco. No Postgres, quando a regra de unicidade é sobre vários campos e um deles está vazio (nulo), o padrão é tratar cada nulo como único, diferente de qualquer outro nulo. Ou seja: se a chave inclui o banco, duas contas com nome igual, tipo igual e banco nulo passariam (nulo não bate com nulo). Isso abriria um buraco: duas contas sem banco com o mesmo nome. Pra fechar, usamos NULLS NOT DISTINCT, que manda o banco tratar nulo como um valor igual a outro nulo. Assim, conta sem banco com mesmo nome e tipo continua bloqueada.

3. Migrations append-only

O esquema do banco do Nexus é versionado por arquivos numerados (Flyway). Migration que já rodou nunca se edita: pra mudar o esquema, você escreve uma nova, com número maior. A regra antiga vivia na V67; a gente não voltou lá. Criou a V83, que apaga o índice antigo e cria o novo. O histórico fica honesto, dá pra ver a regra como era e como ficou.

Como montamos

A migration V83. Duas linhas. A primeira apaga o índice único antigo (sobre user, nome e tipo). A segunda cria o novo, sobre user, nome, tipo e banco, com o NULLS NOT DISTINCT no fim. Índice único é o que garante a regra no banco: mesmo que o código deixasse passar, o banco recusaria a duplicata. A troca é segura porque a regra nova é mais frouxa que a velha: tudo que já estava gravado respeitava a antiga (mais exigente), então respeita a nova.

DROP INDEX uk_finance_contas_user_nome_tipo;

CREATE UNIQUE INDEX uk_finance_contas_user_nome_tipo_banco
    ON finance_contas (user_id, lower(nome), tipo, banco_id) NULLS NOT DISTINCT;

O repositório. Tinha um método que buscava conta por nome e tipo e devolvia no máximo uma (um Optional). Isso valia quando nome+tipo era único. Agora pode existir mais de uma (uma por banco), e um método "no máximo uma" estouraria ao achar duas. A troca pra List não é enfeite, é obrigatória:

// antes — no maximo uma
Optional<FinanceConta> findByUserAndNomeIgnoreCaseAndTipo(User user, String nome, TipoConta tipo);

// agora — todas com esse nome e tipo; o desempate de banco e no codigo
List<FinanceConta> findAllByUserAndNomeIgnoreCaseAndTipo(User user, String nome, TipoConta tipo);

O serviço. A checagem virou o ajudante existsConflito. Ele recebe as candidatas (mesmo nome e tipo), o banco que estou salvando e o id da própria conta na edição. Tem conflito se alguma candidata tem o mesmo banco e não é ela mesma. Dois cuidados: comparar bancos com Objects.equals (trata "sem banco" igual a "sem banco" sem estourar no nulo), e ignorar a própria conta na edição pelo selfId (senão ela acharia conflito consigo mesma). Na criação o selfId é nulo.

private static boolean existsConflito(List<FinanceConta> candidatas, Long bancoId, Long selfId) {
    return candidatas.stream().anyMatch(c ->
        !c.getId().equals(selfId) && java.util.Objects.equals(c.getBancoId(), bancoId));
}

// no update — passa o id da conta editada pra ela se ignorar
if (existsConflito(contas.findAllByUserAndNomeIgnoreCaseAndTipo(conta.getUser(), nome, request.tipo()),
                   request.bancoId(), conta.getId())) {
    throw new FinanceContaNameAlreadyInUseException(nome);
}

Decisões e becos

Incluir o banco, não tirar a regra. Seria mais fácil deixar repetir nome à vontade. Mas a regra existe pra evitar duas contas iguais de verdade. Incluir o banco na chave mantém o bloqueio da duplicata real (mesmo nome, tipo e banco) e abre só o caso legítimo (bancos diferentes).

O NULLS NOT DISTINCT é fácil passar batido. Sem ele, a mudança teria um efeito colateral que ninguém pediu: liberar duas contas sem banco com o mesmo nome. O banco faria isso calado, e só apareceria lá na frente. Pensar no caso do nulo na hora evitou o buraco silencioso.

Reiniciar o backend. A migration só roda quando a aplicação sobe, e o código novo do serviço e do repositório só vale com o processo reiniciado. Enquanto o servidor velho estava no ar, a regra antiga continuava e o erro "nome já em uso" persistia. Subiu de novo, a V83 rodou na inicialização, e a segunda conta FLÁVIO entrou.

Resumo

A conta do Financeiro agora pode ter nome repetido, desde que o banco seja outro. A chave que decide duplicata deixou de ser nome + tipo e passou a ser nome + tipo + banco: uma migration nova (V83) que troca o índice único, com NULLS NOT DISTINCT pra conta sem banco continuar bloqueada; um método de repositório que virou lista; e um ajudante no serviço que desempata pelo banco e ignora a própria conta na edição. A coisa pra levar: toda regra de "já existe" é uma chave, um conjunto de campos. E quando um campo pode estar vazio, decida de propósito se vazio conta como igual, senão o banco decide por você.
