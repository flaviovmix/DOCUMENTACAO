PDF que Nasce de um HTML

0:00 / 0:00

O orçamento do Nexus gera um PDF pra mandar pro cliente. Ele era desenhado célula a célula em Java (biblioteca OpenPDF) e nunca ficava igual ao modelo de proposta que a gente já tinha em HTML. A gente virou a chave: o backend agora renderiza um HTML em PDF. Arquivos reais: OrcamentoPdfService.java e resources/orcamento/proposta.html.

O que a gente queria

O PDF era a cara da proposta, então precisava ficar bonito. Mas ele nascia desenhado no código: "escreve o título aqui, com essa fonte; abre uma tabela; pinta essa célula; alinha à direita", tudo na mão, peça por peça. E a gente já tinha um modelo lindo em HTML e CSS (cabeçalho azul, tabela listrada, barra de total). Por mais que ajustasse cor e tamanho no Java, o desenho nunca batia com o modelo. A virada foi óbvia: o modelo já é um HTML, então deixa o servidor renderizar o próprio HTML em vez de imitá-lo desenhando.

Os conceitos

1. Desenhar um documento vs renderizar

São dois mundos. Desenhar (OpenPDF) é montar a página com régua e mão firme: você marca cada coordenada, posiciona cada caixa, escolhe cada cor. Funciona, mas é trabalhoso e difícil de deixar igual a um modelo. Renderizar é escrever o conteúdo e o estilo num HTML/CSS e entregar pra um motor que faz a diagramação. É a diferença entre desenhar o jornal à mão e dar o texto pro diagramador. Quando você já tem o modelo em HTML, renderizar é o caminho curto: o modelo praticamente já é a resposta.

2. Renderizar HTML no servidor: dois jeitos

Um jeito é um navegador sem tela (o motor do Chrome/Edge) "imprimindo pra PDF": renderização perfeita, mas exige um navegador instalado no servidor, o que vira dor de cabeça quando o Nexus for pro ar num container Docker no Linux. O outro é uma biblioteca Java pura, a openhtmltopdf, que lê o HTML e gera o PDF sozinha, sem navegador. Escolhemos a biblioteca pura: ela sobrevive ao Docker sem depender de nada instalado. Tem um preço (ela só entende CSS antigo, ver becos).

3. Template com lacunas

O HTML do orçamento é quase todo fixo. Só uns pedaços mudam: a lista de serviços, o total, o que não foi incluído. Esses pedaços ficam como lacunas marcadas no HTML ({{INVEST_ROWS}}, {{NAO_INCLUIDO}}), e na hora de gerar o código troca cada lacuna pelo conteúdo montado a partir dos dados. É um molde com espaços em branco, igual carta modelo onde só o nome muda.

Como montamos

A biblioteca. Adicionamos a openhtmltopdf-pdfbox no pom.xml e o código velho da OpenPDF (que desenhava tudo) saiu de cena.

O template. O modelo da proposta virou um arquivo HTML dentro do projeto (resources/orcamento/proposta.html), lido na hora de gerar.

O código que gera. Ficou em três passos: lê o template, troca as lacunas, manda renderizar. Compare com o código antigo, cheio de métodos pra desenhar cada tabela e célula:

public byte[] generate(Orcamento o) {
    String html = loadTemplate()
        .replace("{{INVEST_ROWS}}", buildInvestRows(o))
        .replace("{{NAO_INCLUIDO}}", buildNaoIncluido(o));
    ByteArrayOutputStream baos = new ByteArrayOutputStream();
    PdfRendererBuilder builder = new PdfRendererBuilder();
    builder.useFastMode();
    builder.withHtmlContent(html, null);
    builder.toStream(baos);
    builder.run();
    return baos.toByteArray();
}

As lacunas. O código percorre os serviços e monta uma linha por item contratado (descrição, quantidade, valor) e a barra de total. Os serviços desmarcados (existem, mas não foram contratados) saem da tabela e vão pro quadro "não incluído", mostrando pro cliente o que dá pra fazer mas não entrou. Detalhe de capricho: o cifrão fica colado à esquerda e o número à direita, estilo nota. Como a lib não tem flexbox, isso é feito com uma mini-tabela dentro da célula:

private static String money(BigDecimal v) {
    String n = BR_MONEY.format(v == null ? BigDecimal.ZERO : v);
    return "<table class=\"money\"><tr>"
         + "<td class=\"rs\">R$</td>"
         + "<td class=\"amt\">" + n + "</td>"
         + "</tr></table>";
}

Decisões e becos

O preço da lib pura: sem flexbox. A openhtmltopdf entende um manual de layout antigo do CSS. Os jeitos modernos de alinhar coisas lado a lado (flexbox) ela não conhece, igual uma gráfica que só lê um manual de anos atrás. Todo trecho do modelo que usava flexbox (o cabeçalho, a caixa de dados, os dois quadros incluído/não-incluído) foi reescrito com tabela. Visualmente igual; por baixo, tabela.

Os símbolos sumiram. O modelo tinha um ✓ e um ✕ nos títulos dos quadros. A fonte padrão que a lib usa pra gerar o PDF não tem esses caracteres no acervo, então sairiam quebrados. Tiramos. Lição: ao gerar PDF, o que aparece depende da fonte ter aquele caractere.

O beco mais chato: reiniciar. Toda vez que a gente mexia no HTML ou no código, nada mudava. Por quê? O template não é lido do disco a cada geração; ele entra junto com o programa quando o servidor sobe, e o Nexus em desenvolvimento não recarrega sozinho. Enquanto não reiniciava o backend, ele gerava com a versão velha. Reiniciamos um monte de vezes. Lição: mexeu no servidor e nada mudou? Antes de caçar bug, confira se ele subiu com a versão nova.

Uma briga de CSS. A quantidade devia ficar alinhada na coluna, mas o número e o título "Qtd" do cabeçalho não batiam. Uma regra geral dizia "todo cabeçalho de tabela alinha à esquerda" e vencia a regra da coluna. Quando duas regras de CSS brigam, vence a mais específica. Subimos a especificidade (table.data .col-qt) e os dois passaram a alinhar juntos.

Por que a lib e não o navegador. O navegador daria resultado idêntico ao modelo, mas o objetivo não é só funcionar hoje na máquina do dev, é funcionar quando o Nexus for pro ar num container. Um resultado um pouquinho menos perfeito que roda em qualquer lugar vale mais que um perfeito que depende de navegador instalado no servidor.

Resumo

O PDF do orçamento parou de ser desenhado peça por peça em Java e passou a ser um HTML renderizado pela openhtmltopdf. O template espelha o modelo da proposta, com lacunas que o código preenche com os dados do orçamento. O código de gerar encolheu pra três passos: ler, trocar, renderizar. A coisa pra levar: quando você já tem um modelo em HTML, renderizar é quase sempre melhor que redesenhar no código. Só lembre dos dois preços: a lib pura só entende CSS antigo (alinhamento moderno vira tabela), e o template entra junto com o programa (mexeu, reinicia o servidor).
