Atualizando os Repos no GitHub

0:00 / 0:00

Este making-of é diferente: não é sobre uma tela, é sobre uma tarefa de bastidor que a gente faz toda hora e quase nunca para pra entender. Salvar o trabalho no GitHub. No fim do dia, um comando só (/commit) levou três projetos inteiros pro GitHub, cada um arrumadinho. O episódio abre essa caixa preta: os conceitos de Git por trás (branch, commit, push, squash, merge ours) e o passo mais valioso do roteiro, que nem é um comando: é a pausa pra olhar a lista antes de empurrar.

O que a gente queria

Pegar tudo que mudou em três projetos diferentes e mandar pro GitHub de um jeito organizado, sem misturar lixo e sem quebrar o histórico. Os três: o XT-ZERO (o repositório principal na raiz c:\src, com o Java do treinamento, as instruções, o backup das memórias e a raiz da doc); o XT-DOCUMENTACAO (só o site de estudo: páginas, áudios, imagens); e o Nexus (o projeto grande, Spring Boot + SvelteKit). Cada um é um repositório Git independente ("repositório" é só o nome chique pra a pasta que o Git está vigiando). E a primeira ideia: isso não foi feito na mão, comando por comando. Foi um slash command.

Os conceitos

Slash command. Um roteiro salvo. A lista exata de passos pra atualizar um repositório, escrita uma vez num arquivo (dezesseis passos, na ordem, com avisos de onde parar e perguntar). Depois, em vez de lembrar de cada passo, é só digitar /commit. O valor não é fazer mais rápido, é fazer sempre igual, sem pular passo por esquecimento.

Branch. Uma linha do tempo paralela do projeto. A gente trabalha na dev, onde a sujeira acontece (commit pela metade, testa, conserta). E tem a main, a versão limpa, a oficial. A bagunça fica na dev; na main entra só o resultado final. Assim o histórico da main fica legível: uma sequência de "essa feature ficou pronta", em vez de duzentos "ops, corrigi um typo".

Commit e push. Commit é tirar uma foto do estado dos arquivos e guardar no histórico, com uma mensagem. Essa foto fica na sua máquina. Push é empurrar as fotos pra nuvem, pro GitHub. Commit salva, push publica. Um detalhe que o roteiro insiste: antes de commitar, escolher a dedo o que entra. O atalho preguiçoso "adicionar tudo" é cego, pega qualquer coisa na pasta. O roteiro manda olhar a lista primeiro.

Merge squash. Merge é juntar duas branches; squash quer dizer "amassar". Então pega todos os commits bagunçados da dev e amassa num único commit limpo na main. Os quinze passos intermediários ficam guardados na dev, mas a main recebe um commit só, tipo "essa feature ficou pronta".

Merge ours (o passo crítico). Depois de amassar a dev na main, um último passo: um merge com a estratégia ours ("a nossa"). Quando você faz squash, a main ganha o conteúdo, mas o Git não registra que a dev "já foi mesclada". Pro Git, aqueles commits ainda estão pendentes, e no dia seguinte ele tenta trazer tudo de novo, dando conflito. O merge ours é um aperto de mão: diz pro Git "registra que dev e main estão em dia, mas não muda o conteúdo da main". Sem ele, todo merge vira briga de conflitos no dia seguinte.

Como montamos, o fluxo real

Digitei o comando e disse "todos os projetos". O assistente foi numa lista que eu mantenho (os repositórios ativos), achou os três (XT-ZERO, XT-DOCUMENTACAO, Nexus) e, antes de tocar no Git, olhou o status dos três (git status mostra o que mudou desde a última foto).

Depois de confirmar o que entrava, um passo de segurança que o roteiro exige: copiar as memórias e configurações pra dentro do repositório, pra irem junto no backup. Só então o ciclo do Git. Pro XT-ZERO: adicionar os arquivos confirmados, commitar na dev, push da dev; depois o merge pra main (trocar pra main, merge squash, commitar o squash, merge ours pra fechar, push da main, voltar pra dev). Esse vai-e-volta de branch é o coração do roteiro. O XT-DOCUMENTACAO foi a mesma dança.

O Nexus tem uma diferença: ele não faz merge pra main. Por enquanto só commita e dá push na dev (a regra está na própria lista de repositórios). Por quê? Porque o Nexus ainda muda muito; não faz sentido manter uma "versão oficial limpa" de algo que muda toda semana. Pra ele o ciclo parou no push da dev (foram 76 arquivos modificados e 30+ novos, toda a feature de finanças, com onze migrations).

Decisões e becos

Os três arquivos estranhos (o melhor pedaço). No XT-ZERO o status mostrou três suspeitos: um CLAUDE(1).md (com aquele "(1)" de download duplicado), um virus.png e um Screenshot.png. Em vez de mandar tudo pra dentro, o assistente parou e perguntou. O CLAUDE(1).md era lixo mesmo, apaguei. Já o virus.png eu mandei manter de propósito (quero estudar depois). Se tivesse usado o "adicionar tudo" cego, ou apagaria minha anotação, ou comitaria o lixo. A pergunta evitou os dois. A lição: o passo mais valioso do roteiro não é um comando do Git, é a pausa pra um humano olhar a lista.

Cada projeto pela sua regra. Dois fazem merge na main, o Nexus não. Um roteiro bom não é "faz igual pra todos", é "faz o certo pra cada um".

O tamanho dos arquivos. No push da documentação, o GitHub soltou um aviso (não erro): o menu.m4a tem 53,87 MB, acima dos 50 recomendados. Passou normal. Isso levantou a dúvida: o limite é da conta inteira ou por projeto? No Git comum, o limite é por arquivo e por repositório, não há soma da conta: 50 MB dá aviso amarelo, 100 MB é o limite duro; por repo, recomenda-se abaixo de 1 GB. Não existe teto somando os três projetos, cada um vive isolado. A exceção é o Git LFS (pra arquivos grandes, tipo vídeo e áudio): aí sim existe uma cota da conta inteira (no plano grátis, 1 GB de armazenamento e 1 GB de tráfego/mês). Ponta solta pro futuro: uma hora vale mover os .m4a pesados pro LFS pra não inflar o repo. Fica anotado.

Resumo

Um comando salvo dispara um roteiro de dezesseis passos: olha o que mudou, pergunta antes de mandar qualquer coisa estranha, faz backup, commita e dá push na branch de trabalho, e pros projetos que pedem, amassa tudo num commit limpo na main e fecha o histórico com o aperto de mão do merge ours. Cada projeto seguindo a sua regra. Os conceitos que ficam: branch é linha do tempo paralela, commit salva, push publica, squash amassa, ours reconcilia, e o limite de tamanho no GitHub é por arquivo e por repositório, não da conta inteira (a não ser que você entre no mundo do LFS).
