Link Público do Orçamento

0:00 / 0:00

Um cliente pede um orçamento. Em vez de mandar um PDF anexo, a ideia é mandar um link: o cliente abre uma página, vê o orçamento e, se quiser, imprime ou gera o PDF ele mesmo. O Nexus já tinha um botão HTML que montava essa página, mas ela só abria pra quem estava logado. A gente transformou isso num link de verdade na web, que o cliente abre sem conta e sem senha. O conceito por trás é a URL que carrega o próprio segredo: um token impossível de adivinhar no lugar de uma senha. Arquivos reais: PublicOrcamentoController.java, OrcamentoService.java, SecurityConfig.java no backend, e OrcamentoDetailModal.svelte no front.

O que a gente queria

O cenário: o cliente pede um orçamento, e em vez de um PDF no email a gente quer mandar um link. Ele abre, vê o orçamento numa página bonita, e imprime ou gera o PDF ali mesmo no navegador. Os dados são dele, e só ele recebe o link.

O Nexus já tinha um botão HTML que montava a página. O problema: ela só abria pra quem estava logado. O botão pegava a página pronta e abria numa aba como um arquivo temporário, um rascunho que só existe naquela aba, naquele computador logado. Servia pra gente conferir, não pra mandar pro cliente, que não tem login no sistema. O pedido foi virar isso num link de verdade na web: um endereço que o cliente abre no celular dele, sem senha.

Os conceitos

Rota protegida vs pública. Quase tudo no Nexus pede crachá. Quando você faz login, o sistema te dá um crachá digital (um token), e a cada tela um porteiro confere esse crachá. Sem crachá, a porta não abre. Mas a página do orçamento, pra ir pro cliente, não pode pedir crachá: o cliente não tem.

O token inadivinhável como credencial. Como deixar uma página aberta sem escancarar o resto? Põe o segredo dentro do próprio link. Em vez de uma senha separada, o endereço carrega um código secreto: um número gigante e aleatório que ninguém digita por acaso nem adivinha tentando, um UUID. O link fica tipo /public/orcamentos/fbfc5a34-3d8f-4981-a659-236afd47158a. Quem tem o link entra; quem não tem teria que adivinhar uma chave entre bilhões de bilhões de combinações. Na prática, impossível. O link virou a credencial, a mesma ideia do "qualquer um com o link pode ver" do Google Drive.

O encurtador (de passagem). Esse link com o código gigante fica comprido. Pra mandar mais fácil, passamos ele por um encurtador: um link curtinho que, ao ser aberto, redireciona pro comprido. É só um atalho, opcional. O link comprido já funcionaria sozinho.

Como montamos

1. O token. Demos ao orçamento um campo novo, o token, que nasce vazio e só é gerado na primeira vez que você compartilha (uma coluna public_token no banco).

2. A rota pública. Uma porta nova no servidor: alguém abre /public/orcamentos/{token}, o sistema acha o orçamento daquele token e devolve a página HTML. Sem pedir crachá.

@GetMapping("/{token}")
public ResponseEntity<byte[]> page(@PathVariable UUID token) {
    Orcamento o = service.loadPublicByToken(token);
    byte[] bytes = html.generate(o);
    return ResponseEntity.ok().contentType(/* text/html */).body(bytes);
}

3. Avisar o porteiro. O porteiro de segurança tem uma lista de endereços que entram sem crachá (o login, as páginas públicas de notas). A gente só adicionou o endereço novo na lista.

.requestMatchers(HttpMethod.GET, "/public/orcamentos/*").permitAll()

4. O botão compartilhar. Ao clicar, ele garante que o orçamento tem um token (cria na hora se não tinha), monta o link da página pública e cria, ou reaproveita, um link curto apontando pra lá. Reaproveita pra não encher o encurtador de links repetidos.

5. O modal. Na tela, clicar HTML agora abre um modalzinho com o link pronto e um botão de copiar, pra você mandar pro cliente.

Decisões, becos e o toque de quem tem mais experiência

Antes de codar, a gente parou num detalhe fácil de passar batido, e é a lição do episódio. A ideia ingênua seria: "já existe um botão que gera a página e um encurtador pronto, então é só apontar um link curto pra página que a gente tem". Mas isso quebraria na mão do cliente. Por quê? Porque um link curto é só um redirecionamento: o navegador do cliente é mandado pra página sem crachá nenhum junto, e a página velha pedia crachá. O cliente tomaria um "acesso negado". O ponto cego: um link que você manda pra fora nunca leva o seu login junto. Foi isso que decidiu o desenho inteiro, ter que criar uma rota pública de propósito, com o token fazendo o papel da senha. Perceber antes de codar evitou refazer tudo depois.

O beco da aba anônima. Na primeira versão, o botão ainda abria a página como aquele arquivo temporário. Pra testar de verdade, abrimos o link numa aba anônima (o cenário do cliente: nenhum login). Deu "arquivo não encontrado", porque o rascunho só existe na aba que o criou. Foi a prova viva de que precisava de um link na web de verdade.

O 401 enganoso. Testando a rota nova, em vez de "não existe" (404) vinha "não autorizado" (401). O motivo: a rota nova ainda nem tinha entrado no servidor rodando, e o porteiro, ao não reconhecer o endereço e não ver crachá, já respondia "não autorizado". A correção foi reiniciar o servidor. Lição pequena: no backend, código novo só vale depois de reiniciar.

Direto vs encurtado, e público é público. Duas decisões de fechamento. O encurtador é opcional: o link comprido com o token já resolve, o /s/ curtinho só deixa mais bonito de mandar (mantivemos porque já estava pronto). Pra ter um link curto na raiz de um domínio só dele precisaria de um subdomínio dedicado, coisa de produção. E a página é pública de propósito: qualquer um com o link vê os dados do cliente. No nosso caso tudo bem, o dono dos dados é o próprio cliente que recebe o link. Mas é uma escolha consciente, não um descuido.

Resumo

A gente transformou a página do orçamento, que só abria logado, num link que o cliente abre sem conta. O truque é a URL que carrega o próprio segredo: um token gigante e aleatório que faz o papel da senha. Por baixo, uma rota pública no servidor, liberada no porteiro, e um token que nasce só na hora de compartilhar. A coisa pra levar é a do começo: um link que você manda pra fora nunca leva o seu login junto, então a página do outro lado precisa de uma chave própria. Esse mesmo desenho serve pra qualquer coisa compartilhável por link: uma nota, uma foto, um relatório.
