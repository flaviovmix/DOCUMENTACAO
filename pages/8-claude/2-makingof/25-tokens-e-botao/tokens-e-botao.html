Tokens e Botão

0:00 / 0:00

Episódio focado no código. Par do conceitual Tudo num Lugar Só: lá a faxina é contada por cima; aqui a gente lê o app.css e o Button.svelte linha por linha pra entender como o knob do botão laranja funciona por dentro.

O que esse código faz

Dois arquivos. O app.css é onde moram os valores de estilo com um nome cada (a cor da marca, os cinzas, os números de empilhamento, os raios). O Button.svelte é o componente do botão, e a parte que importa é uma linha: a cor de fundo do primário não é um número escrito ali, é "me dá a cor da marca". Juntando os dois, você tem o knob: muda a cor num lugar, muda em todo botão.

O conceito mínimo: custom property

Só uma ideia pra o código fazer sentido. O CSS deixa você criar suas próprias variáveis, as custom properties. O nome sempre começa com dois traços (--color-brand). Você guarda um valor nela num lugar, e lê em qualquer outro com var(--nome). Guarda num lugar, lê em mil. Agora o código.

Lendo o código: app.css

A abertura. :root { — o :root é um seletor especial que casa com a raiz do documento, o elemento mais de fora. Declarar as variáveis aqui faz elas valerem pra página inteira: todo elemento enxerga. É o lugar canônico de token global.

:root {
  --color-text: #1a1a1a;   /* nome : valor ; */
  --gray-700: #374151;     /* maior numero = mais escuro */
  --gray-50:  #fafafa;
}

Os grupos. Cada linha é nome: valor;. Os neutros (a escala de cinza): repare que o número menor é mais claro (--gray-50 é quase branco, --gray-700 é escuro). Os números (700, 500, 300) são só convenção de "quão escuro". A paleta não é nova; o que mudou foi dar nome aos hexes mais repetidos.

--color-brand: #ff3e00;        /* o knob do botao primario */
--color-brand-hover: #e03600;
--color-danger: #dc2626;       /* cada cor tem par normal + hover */

O z-index com nome. Antes era número solto (alguém escrevia 1000 e ninguém sabia por quê). Virou escala nomeada: --z-modal: 1001, --z-modal-nested: 1100 (modal em cima de modal), --z-toast: 2000. Agora a ordem é óbvia ao ler.

--z-modal: 1001;
--z-modal-nested: 1100;
--z-toast: 2000;

O detalhe que confunde: esse arquivo não tem nenhuma regra de estilo de verdade, nenhum botão pintado. Ele só declara nomes. Quem usa os nomes é o resto do app. E pra os nomes existirem em todo lugar, o app.css é importado uma vez no +layout.svelte (o componente que embrulha todas as telas). Carregou ali, o :root vale pro app inteiro.

Lendo o código: Button.svelte

O import. import type { Snippet } from 'svelte' — o import type traz só o tipo, some quando compila. Snippet é o tipo do conteúdo que vai dentro do botão.

As props. type Props = { ... } lista o que o botão aceita. A ? depois do nome quer dizer opcional. O variant é uma union (lista de textos fixos separados por |): só pode ser primary, cancel, neutral, link ou danger — o editor não deixa outro. O children: Snippet não tem ?, então é obrigatório.

variant?: 'primary' | 'cancel' | 'neutral' | 'link' | 'danger';
form?: string;        // id do form, pra botao fora dele enviar
children: Snippet;    // sem '?' = obrigatorio

A entrada das props. let { ... } = $props() — o $props é um rune do Svelte 5, a porta de entrada das props. Os = dentro das chaves são os valores padrão: variant = 'neutral' quer dizer que, se ninguém passar, ele nasce neutral.

let { variant = 'neutral', type = 'button', disabled = false,
      form, onclick, children }: Props = $props();

O HTML do botão. Três coisas: (1) {type} sozinho é atalho — quando o nome do atributo é igual ao da variável, escreve uma vez (vira type={type}); (2) class="btn btn-{variant}" monta a classe na hora: se variant for primary, vira btn btn-primary; (3) onclick={(e) => onclick?.(e)} — o ?. é a chamada opcional: só chama se o onclick existir, senão não quebra. Dentro, {@render children()} desenha o conteúdo que veio de fora.

<button {type} {form} class="btn btn-{variant}" {disabled}
        onclick={(e) => onclick?.(e)}>
    {@render children()}
</button>

A linha que fecha tudo. No <style>, o .btn é o tronco comum (inline-flex, gap, padding, raio). E então:

.btn-primary { background: var(--color-brand); color: #fff; font-weight: 600; }
.btn-primary:hover:not(:disabled) { background: var(--color-brand-hover); }

A cor de fundo do primário não é um número, é o var() lendo o token --color-brand lá do app.css. O :hover:not(:disabled) quer dizer "no hover, mas só se não estiver desabilitado".

Decisões no código

  • O #fff ficou na mão. O branco do texto não virou token porque branco não muda com a marca; o que muda é o fundo. Token é pra valor que você quer girar de um lugar; o branco não é knob.
  • O knob depende de uma disciplina: apontar pro var, nunca repetir o número. No momento que um botão escrevesse #ff3e00 na mão, ele sairia de baixo do knob e mudar a marca o deixaria pra trás.
  • O :not(:disabled) no hover evita um bug chato: sem ele, um botão desabilitado mudaria de cor no hover, dando impressão de que dá pra clicar.

Resumo

O app.css só declara nomes dentro do :root (cores, cinzas, z-index nomeado, raios, sombras), e é carregado uma vez no +layout. O Button.svelte recebe props por $props, monta a classe na hora com btn-{variant}, e a linha-chave é background: var(--color-brand) no .btn-primary. Esse é o knob: o número do laranja mora num lugar só (o token) e todo primário aponta pra ele. A regra pra continuar funcionando: nunca repetir o número, sempre ler o var.
