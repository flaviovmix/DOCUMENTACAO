Globo de Conexões no Login

0:00 / 0:00

Making-of do planeta animado que gira no fundo da tela de login do Nexus: uma bola de pontos laranja com continentes e arcos de luz viajando entre cidades. Tudo em canvas 2D puro, sem nenhuma biblioteca 3D. Arquivo real: frontend/src/routes/login/+page.svelte.

O que a gente queria

Um globo terrestre girando devagar atrás do formulário de login, no mesmo clima quente (laranja/fogo) do cenário que já existia ali (montanhas e floresta). Não um mapa exato: um enfeite de fundo, com cara de planeta de rede, pontinhos formando os continentes e linhas de conexão acendendo entre eles.

Os conceitos

Cinco ideias sustentam o efeito.

1. Pontos numa esfera por latitude e longitude

Cada ponto do globo nasce de dois ângulos: latitude (o quanto sobe ou desce) e longitude (o quanto gira em volta). Com seno e cosseno desses dois ângulos vira um vetor 3D {x, y, z} em cima de uma bola de raio 1. É o mesmo jeito que se aponta um lugar no globo terrestre.

2. Projeção ortográfica: 3D virando 2D

A tela é plana, o ponto é 3D. A projeção ortográfica é a mais simples possível: joga fora o z e usa direto o x e o y como posição na tela. É a sombra reta do ponto na parede. O z não some de vez: ele decide se o ponto está na frente da bola (z > 0, aparece) ou atrás (some), e regula o brilho pra dar sensação de volume.

3. Rotação no tempo

O planeta gira porque a cada quadro da animação um ângulo cresce junto com o relógio. Esse ângulo roda os pontos em torno do eixo vertical antes de projetar. Mais um leve tombo fixo no eixo (o tilt) só pra dar pose de planeta inclinado.

4. Continentes por ponto-dentro-do-polígono

Os continentes são contornos grosseiros desenhados à mão em [lon, lat] (baixa resolução de propósito, é fundo de tela, não atlas). Pra saber quais pontos da grade são terra, usa-se ray casting: dispara um raio a partir do ponto e conta quantas vezes ele cruza a borda do polígono. Número ímpar de cruzamentos = está dentro.

5. Arcos por interpolação esférica (slerp)

As linhas de conexão entre cidades não cortam reto pelo meio da bola: elas acompanham a superfície. Isso é o slerp (spherical linear interpolation): caminhar pela casca da esfera, pelo trajeto mais curto em cima da bola, entre dois pontos. Um pequeno "bojo" no meio infla o arco pra fora, e um pulso de luz percorre ele.

Como montamos

Primeiro, gerar os pontos de terra. Varre uma grade de latitude/longitude, testa cada ponto com inLand() e, se for terra, converte pra vetor 3D na esfera:

const land = [];
for (let lat = -56; lat <= 80; lat += 2) {
    for (let lon = -180; lon < 180; lon += 2) {
        if (!inLand(lon, lat)) continue;
        const la = (lat * Math.PI) / 180;
        const lo = (lon * Math.PI) / 180;
        land.push({
            x: Math.cos(la) * Math.sin(lo),
            y: Math.sin(la),
            z: Math.cos(la) * Math.cos(lo)
        });
    }
}

O teste de terra é o ray casting, que conta cruzamentos da borda de cada contorno:

const inLand = (lon, lat) => {
    for (const poly of LAND) {
        let inside = false;
        for (let i = 0, j = poly.length - 1; i < poly.length; j = i++) {
            const xi = poly[i][0], yi = poly[i][1];
            const xj = poly[j][0], yj = poly[j][1];
            if (yi > lat !== yj > lat &&
                lon < ((xj - xi) * (lat - yi)) / (yj - yi) + xi)
                inside = !inside;
        }
        if (inside) return true;
    }
    return false;
};

A rotação + tilt acontecem no project(). Repare que a projeção em si é ortográfica: no fim, quem desenha usa p.x e p.y como posição, e p.z só diz frente ou trás:

const project = (p, c, s) => {
    const x1 = p.x * c + p.z * s;       // rotação em Y (planeta girando)
    const z1 = -p.x * s + p.z * c;
    const y2 = p.y * tc - z1 * ts;      // tilt em X (planeta tombado)
    const z2 = p.y * ts + z1 * tc;
    return { x: x1, y: y2, z: z2 };
};

O slerp anda pela superfície entre dois pontos:

const slerp = (a, b, t) => {
    let dot = a.x * b.x + a.y * b.y + a.z * b.z;
    dot = Math.max(-1, Math.min(1, dot));
    const om = Math.acos(dot);
    if (om < 1e-4) return a;
    const sn = Math.sin(om);
    const w1 = Math.sin((1 - t) * om) / sn;
    const w2 = Math.sin(t * om) / sn;
    return { x: a.x*w1 + b.x*w2, y: a.y*w1 + b.y*w2, z: a.z*w1 + b.z*w2 };
};

Tudo isso roda dentro de um loop de animação (requestAnimationFrame): a cada quadro o ângulo cresce com o tempo, limpa o canvas, desenha a grade curva (meridianos e paralelos), os continentes pontilhados, os arcos com o pulso de luz e os nós das cidades. Dois detalhes de polimento: o device pixel ratio evita borrão em tela retina, e um PRNG de semente fixa faz o layout nascer igual a cada reload, sem ficar pulando.

Decisões e becos

Cenário trocado por engano. No vai e vem das tentativas o cenário de fundo chegou a ser trocado sem querer, e teve que voltar pro clima de fogo (montanhas e floresta) que já estava ali.

Reposicionamento. O planeta não fica centralizado: ele é grande e fica empurrado pra fora pela direita, mostrando só uma fatia. E entra no markup depois das estrelas e antes das montanhas, pra elas cobrirem a base da bola, dando o efeito de planeta nascendo no horizonte.

A borda que precisou ser removida. O maior beco: no começo o planeta tinha um disco escuro de fundo e um halo em volta. Contra o céu laranja, isso criava uma borda visível, um círculo nítido demarcando a bola. A solução não foi suavizar a borda: foi remover o disco e o halo. Sem eles, o planeta se revela só pela grade curva e pelos continentes pontilhados, sem nenhuma silhueta delimitando.

Resumo

Um globo de fundo feito em canvas 2D: pontos numa esfera por lat/lon, projeção ortográfica jogando o 3D na tela, rotação crescendo com o tempo, continentes decididos por ponto-dentro-do-polígono e arcos desenhados por slerp na superfície. A lição de design do beco final: às vezes o conserto certo é tirar a peça que incomoda, não tentar disfarçá-la.
