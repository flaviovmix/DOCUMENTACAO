Tabela Vira Classe

0:00 / 0:00

Etapa 3 da migração do site do Kaco pra Java. O app já servia o site e o banco kids já existia com 15 tabelas versionadas em Flyway, mas o Java e o banco ainda não se conheciam. Esta etapa constrói a ponte: cada tabela vira uma entidade JPA, cada consulta vira um método de repository, e um teste de fumaça prova que o espelho reflete direito, contra o schema real. Regra que guiou tudo: a fonte de verdade é o V1__schema.sql, não o rascunho do plano.

A entidade simples: Usuario

Dois pacotes novos, dominio e repositorio. O Usuario mostra o kit básico. @Entity marca a classe, e o nome Usuario casa com a tabela usuario sozinho: a naming strategy do Spring converte camelCase pra snake_case (por isso senhaHash acha senha_hash sem @Column(name) nenhum).

@Entity
public class Usuario {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String email;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private Papel papel;          // enum ADMIN / FAMILIA, gravado como TEXTO

    @Column(nullable = false)
    private boolean ativo = true; // espelha o DEFAULT TRUE do DDL

Peça por peça: o @GeneratedValue(IDENTITY) diz que o id quem gera é o banco (a coluna é BIGSERIAL); o Java recebe o número de volta depois do insert. O @Enumerated(STRING) grava o texto do enum, casando com o CHECK da coluna — sem ele o JPA gravaria a posição numérica e o CHECK estouraria. E como o schema é do Flyway (ddl-auto: none), o @Column(nullable, unique) não cria nada: é documentação do contrato, o espelho declarando o que a tabela exige. Os defaults do banco (ativo, criado_em) nascem iguais no campo Java.

A entidade rica: Historia

A Historia é o coração do domínio e carrega as três relações do JPA:

@ManyToOne(fetch = FetchType.LAZY, optional = false)
@JoinColumn(name = "categoria_id")
private Categoria categoria;

@OneToMany(mappedBy = "historia", cascade = CascadeType.ALL, orphanRemoval = true)
@OrderBy("ordem")
private List<Cena> cenas = new ArrayList<>();

@ManyToMany
@JoinTable(name = "historia_personagem",
        joinColumns = @JoinColumn(name = "historia_id"),
        inverseJoinColumns = @JoinColumn(name = "personagem_id"))
@OrderBy("ordem")
private List<Personagem> personagens = new ArrayList<>();

O @ManyToOne(LAZY) não busca a categoria junto; só quando alguém chamar getCategoria(). As cenas são os vagões engatados: cascade = ALL salva as cenas junto da história, e orphanRemoval = true apaga do banco a cena que sair da lista — cena sem história não existe. O @OrderBy("ordem") garante a lista sempre na ordem de leitura. E no @ManyToMany, a tabela de junção historia_personagem existe no banco mas não vira classe: o JPA cuida dela por baixo.

Um método pequeno evita o bug clássico de relação bidirecional (esquecer um lado e o insert sair sem historia_id):

public void addCena(Cena cena) {
    cena.setHistoria(this);   // casa os DOIS lados da relação
    cenas.add(cena);
}

No resto do domínio, duas curiosidades: a Rotina aponta pra ela mesma (proximaRotina, a trilha encadeada do site) e a Emocao carrega duas listas filhas (sinais e passos) — o desenho real que a leitura dos HTMLs impôs lá na etapa 1, não o rascunho genérico do plano.

Repositories: o nome é a consulta

Oito interfaces, uma por raiz do domínio, todas estendendo JpaRepository — ganham save, findById, findAll e delete de graça. As consultas específicas são derivadas do nome do método, zero SQL:

public interface HistoriaRepository extends JpaRepository<Historia, Long> {

    Optional<Historia> findBySlug(String slug);

    /** Só o que aparece no site (o endpoint da Etapa 5 usa este). */
    List<Historia> findByPublicadaTrueOrderByOrdem();
}

findByPublicadaTrueOrderByOrdem lê como frase: publicada igual a true, ordenado por ordem. O Spring Data monta o SQL no boot. No UsuarioRepository, o findByEmail já nasceu esperando o login da etapa 4.

O teste de fumaça contra o banco real

Decisão antes do código: nem H2, nem Testcontainers. O pom não tem nenhum dos dois, de propósito — o teste roda contra o Postgres real do container, o mesmo schema Flyway de dev. Duas anotações fazem isso, e a transação garante que nada fica:

@DataJpaTest                                                      // só a fatia JPA
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class HistoriaRepositoryTest {                                    // banco REAL, sem H2

    @Test
    void salvaELeHistoriaComCenasEPersonagens() {
        Categoria aventura = categorias.findByChave("aventura").orElseThrow();
        // ... monta história com 2 cenas + personagem, saveAndFlush ...
        em.clear();  // limpa o cache de sessão: a releitura vem do BANCO

        Historia lida = historias.findBySlug("teste-o-passeio").orElseThrow();
        assertThat(lida.getCenas()).hasSize(2);
        assertThat(lida.isPublicada()).isFalse();  // default: nasce rascunho
    }
}

O @DataJpaTest é transacional: tudo que o teste insere sofre rollback no fim, e o banco de dev fica limpo. A categoria vem do seed da V2 (já existe no banco). E a linha fácil de ignorar é o em.clear(): sem ela, o findBySlug devolveria o mesmo objeto do cache de sessão e o teste não provaria nada — com ela, a releitura vem do banco de verdade.

O resultado

Dois testes verdes, e o log do Hibernate como prova visual: insert into historia (audio_path, categoria_id, criada_em, ..., publicada, slug, titulo) — as colunas exatas do V1 — e os selects das cenas com order by ordem. O espelho reflete. Detalhe honesto: sem Lombok no projeto, getters e setters foram escritos na mão; arquivo comprido, zero mágica, e pro objetivo de estudo ver tudo explícito ajuda mais do que esconder.

Resumo

A etapa 3 em quatro movimentos. A entidade simples (@Entity, id gerado pelo banco, enum como texto, defaults espelhados, nomes convertidos sozinhos). A entidade rica (@ManyToOne preguiçoso, @OneToMany com cascade e orphanRemoval, @ManyToMany com junção invisível, addCena casando os dois lados). Os repositories com consulta derivada do nome. E o teste de fumaça @DataJpaTest + Replace.NONE contra o Postgres real, transacional com rollback, com o em.clear() garantindo que a leitura vem do banco.
