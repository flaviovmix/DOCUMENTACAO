A História Sobe pelo Painel

0:00 / 0:00

Etapa 8 da migração do site do Kaco pra Java, parte 2 de 2 — a meta é o critério de pronto da etapa inteira: publicar uma história de ponta a ponta só pela interface. História nova nasce rascunho (o upload precisa do slug, que vira nome de pasta), ganha cenas por upload múltiplo, ordena no arraste e publica com guarda: pelo menos 2 cenas, porque a cena 1 é a capa e capa sozinha não é história.

A lista com filtros

AdminHistoriaController: a lista recebe três filtros opcionais (busca por título, categoria, publicadas/rascunhos) e resolve num stream — findAll ordenado, um filter por critério preenchido, e um map final pra um record Linha com tudo que a tabela mostra (título, slug, categoria com cor, capa = cena 1, contagem, badges). O record existe por causa do lazy com open-in-view=false: o template renderiza depois da transação fechar, então o controller inicializa tudo dentro dela.

O form

Três detalhes com história por trás. Os personagens viram chips: checkbox escondido cujo label é uma pílula com ícone e nome — CSS puro, zero JS:

.chip input { position: absolute; opacity: 0; pointer-events: none; }
.chip input:checked + span { border-color: var(--c-green); background: #e8f8ef; }

A transcrição herda a descrição dos dois lados: no navegador, um JS espelha enquanto o usuário não mexer nela (mexeu, o espelho desliga); no servidor, transcrição vazia recebe a descrição no salvar — cobre o caso sem JS. E publicar tem guarda: menos de 2 cenas, o salvar recusa com "salva como rascunho e adiciona as cenas primeiro".

O bug do dirty checking

O salvar é transacional e carrega a história gerenciada; seta os campos do form e só depois valida. Se a validação falha, re-renderiza com erro, sem chamar save. Parece certo — mas erro de validação não é exceção que estoura: o commit acontece no fim do método, e o dirty checking gravaria as mutações parciais mesmo assim. Você tenta publicar sem cenas, vê a mensagem de erro, mas o título que mudou já foi pro banco por baixo dos panos.

} catch (IllegalArgumentException e) {
    h.setPublicada(false);
    if (id != null) {
        // sem isto o commit do @Transactional gravaria as mutações parciais
        // (dirty checking) mesmo re-renderizando com erro
        em.detach(h);
    }
    prepararForm(model, h, selecionados, (id == null) ? List.of() : cenaViews(h));
    model.addAttribute("erro", e.getMessage());
    return "admin/historia-form";
}

detach tira a entidade da área gerenciada (as mudanças pendentes não vão pro commit), mas o objeto continua com os valores digitados — exatamente o que o form precisa mostrar de volta. Detalhe: as coleções lazy são inicializadas antes (um size() resolve), porque entidade destacada não carrega mais nada.

As cenas

Upload múltiplo com regra de ouro do lote: validar todos antes de gravar qualquer um (um inválido cancela o lote). O nome físico congela no upload (próximo número livre da pasta, convenção 1.jpeg..N) — desvio consciente do plano, que mandava renomear a cada reordenação: o site lê do banco, o nome do arquivo não significa mais nada, e a ordem mora só na coluna.

O arraste é HTML5 drag & drop vanilla — três eventos na grade:

grid.addEventListener('dragover', e => {
    e.preventDefault();
    const alvo = e.target.closest('.cena-card');
    if (!alvo || alvo === arrastada) return;
    const r = alvo.getBoundingClientRect();
    const depois = e.clientX > r.left + r.width / 2;
    grid.insertBefore(arrastada, depois ? alvo.nextSibling : alvo);
});

O insertBefore move o card na hora (feedback imediato); no dragend as etiquetas renumeram ("cena 1 · capa") e o botão "Salvar nova ordem" só aparece se a sequência mudou. No servidor, a renumeração esbarra no unique composto (historia_id, ordem) — não dá pra fazer a cena 3 virar 2 enquanto a 2 existe. Solução em duas fases com flush:

private void renumerar(List<Cena> naOrdem) {
    for (int i = 0; i < naOrdem.size(); i++) {
        naOrdem.get(i).setOrdem(1000 + i);   // fase 1: fora do intervalo final
    }
    em.flush();
    for (int i = 0; i < naOrdem.size(); i++) {
        naOrdem.get(i).setOrdem(i + 1);      // fase 2: posição definitiva
    }
}

Remover cena: orphanRemoval apaga a linha, flush antes de renumerar (senão o UPDATE roda antes do DELETE e o unique reclama), o arquivo sai do disco — e se a história publicada ficar com menos de 2 cenas, volta pra rascunho sozinha, com aviso. Excluir a história: banco (cascade) + a pasta inteira do disco, com dupla proteção (caminho confinado e nunca a raiz do content).

A prova por curl

História de teste: 3 PNGs viram 1.png..3.png no disco e cenas 1..3 no banco; reordenação pra 3,1,2 conferida no banco; publicar ok; remover renumera e apaga o arquivo; remover de novo despublica sozinho; excluir zera banco e pasta. E um falso alarme que vale a lição: título com acento chegou como mojibake no banco — suspeita de bug de encoding no app, mas as histórias da carga tinham acento perfeito. O teste da inocência: repetir o POST com os bytes UTF-8 em percent-encoding na mão (imune ao shell) → "História Bonita", slug historia-bonita. O culpado era o shell do teste no Windows, não o app.

Resumo

O critério da etapa está de pé: história nasce rascunho, ganha cenas por upload em lote (validado inteiro antes de gravar), ordena no arraste com a ordem morando na coluna (nome físico congelado), publica com a guarda das 2 cenas e aparece no site servido pelo banco. Os dois bugs de gente grande — dirty checking desarmado com detach no caminho de erro, e o unique da renumeração vencido em duas fases com flush. O fluxo manual de copiar imagem e editar JavaScript morreu pro conteúdo novo. Próximo: o resto do conteúdo entra no painel (etapa 9).
