Tudo num Lugar Só

0:00 / 0:00

Making-of da faxina grande no frontend do Nexus: como pegamos as coisas que estavam copiadas e espalhadas (a cor dos botões, o código das listas, o CRUD, os hexadecimais soltos, os componentes jogados numa pasta só) e botamos cada uma num lugar único, de onde o resto aponta.

O que a gente queria

O frontend do Nexus cresceu rápido, feature atrás de feature, e foi acumulando repetição. Três exemplos do que a gente via:

  • Os botões. O botão principal de uma tela (salvar, criar) tinha três cores convivendo no app: verde num modal, vermelho noutro, laranja noutro. Ninguém decidiu isso de propósito, foi saindo assim.
  • O código. Quase toda área carrega uma lista (tags, tarefas, inventário, colaboradores, projetos), e o código era sempre o mesmo: uma variável pros itens, uma pro "carregando", uma pro erro, uma função pra buscar. Esse bloco estava copiado em treze lugares. Do lado do servidor, o mesmo: listar, pegar, criar, editar, excluir, copiado em dezessete recursos.
  • As cores soltas. No CSS de cada componente, hexadecimal escrito na mão. O mesmo cinza de borda aparecia de cinco jeitos, espalhado por dezenas de arquivos. E mais de setenta componentes jogados numa pasta única.

O objetivo era um só, e concreto: cada coisa num lugar único. A cor num lugar, o código da lista num lugar, o CRUD num lugar, cada componente na gaveta da área dele. Mexeu lá, mudou em todo lugar.

Os conceitos

São três ideias diferentes, mas todas saem da mesma raiz, que os programadores chamam de "não se repita": se a mesma coisa aparece copiada em vários lugares, dá um nome pra ela e põe num lugar só.

1. Design token (uma etiqueta no valor)

Token é só um nome pra um valor de estilo. Em vez de escrever a cor laranja (#ff3e00) dentro de vinte e quatro botões, você escreve o valor uma vez, dá um nome (a "cor da marca"), e os botões apontam pro nome. Como uma etiqueta colada num pote de tinta: todos os botões pedem "me pinta com a cor da marca". Troca a tinta do pote, todos mudam juntos. Isso é um knob, um botão de ajuste: um lugar só pra girar, e a mudança se espalha sozinha. Vale pra cor, borda, sombra e pros níveis de empilhamento (aquele número que decide o que fica na frente do quê).

2. Fábrica (uma forma de bolo)

Quando o mesmo bloco está copiado em treze lugares mudando só um detalhe, dá pra escrever uma função que fabrica o bloco. Você passa só o que muda (qual endereço buscar) e ela devolve a peça pronta: itens, "carregando", erro, função de buscar. Como uma forma de bolo: a forma é uma só, troca o sabor da massa. Treze listas, uma forma. O mesmo vale pro CRUD.

3. Organizar por área (cada peça na sua gaveta)

Cada componente vai morar na gaveta da área dele: o do financeiro na pasta do financeiro, o de notas na de notas. As peças genéricas, que servem o app inteiro (botão, modal, toggle, ícones), vão pra gaveta ui. Pra achar uma peça, você pensa "de que área ela é" e vai direto. As pastas do frontend passam a espelhar as áreas do servidor.

Como montamos

As cores primeiro. Criamos um arquivo só, o app.css, com um bloco :root onde os tokens moram: a cor da marca, a cor de perigo (excluir), a escala de cinzas, os níveis de empilhamento com nome (o 1000 solto virou "camada do modal"), o raio das bordas, as sombras. Importado uma vez no layout que embrulha o app, todo mundo enxerga os nomes.

:root {
  --color-brand: #ff3e00;   /* o knob do botao primario */
  --color-danger: #dc2626;
  --z-modal: 1001;          /* numero magico vira nome */
  --radius: 10px;
}

O botão. O Button ganhou o tipo primary, cuja cor de fundo não é um número, e sim var(--color-brand). Esse é o knob principal: trocar a cor da marca no app.css repinta todos os primários de uma vez.

.btn-primary { background: var(--color-brand); color: #fff; }

As fábricas. Escrevemos a função que fabrica lista e trocamos as treze stores feitas na mão por uma chamada a ela. Escrevemos a do CRUD e ligamos os dezessete recursos. E pros modais, criamos uma base comum (o pedaço que todo modal tem: "tá aberto", fechar, limpar) que catorze modais passaram a reaproveitar.

function crud<T, CreateReq, UpdateReq>(base: string) {
  return {
    list:   () => request<T[]>(base),
    get:    (id) => request<T>(`${base}/${id}`),
    create: (body) => request<T>(base, { method: 'POST', body }),
    update: (id, body) => request<T>(`${base}/${id}`, { method: 'PUT', body }),
    remove: (id) => request<void>(`${base}/${id}`, { method: 'DELETE' })
  };
}
const tagsCrud = crud<Tag, TagRequest>('/tags');

A mudança mais braba. Mover mais de setenta componentes pras gavetas certas. O perigo: mover um arquivo quebra os imports que apontavam pra ele, e eram quase duzentos. Pra não quebrar na mão, fizemos por script: primeiro um passo reescreveu todos os imports pro formato absoluto (apontando da raiz, não do vizinho), e só depois moveu. Com o caminho absoluto, mover não quebra o aponte. No fim de cada etapa, o verificador do projeto: zero erro novo.

Decisões e becos

A cor do botão (decisão do dono). Cheguei nas três cores convivendo e sinalizei: "tem três cores de primário aqui, é inconsistência, mas escolher uma é decisão sua, não minha". Cor de marca é decisão do dono. O Flávio escolheu o laranja. Só aí o knob fez todo sentido: como tudo já apontava pra um nome, bastou pôr o laranja no nome, e os botões verdes de salvar viraram laranja junto, de propósito. Sem unificar antes, essa escolha teria que ser repetida em vinte e quatro lugares.

O erro da automação. Numa parte da migração, um passo automático trocou os botões laranja por uma variante vermelha, e ainda afirmou "a cor não mudou". Tinha mudado (vermelho e laranja são tokens diferentes). A regra de conferir contra o código real, não contra o que a ferramenta diz, pegou o erro: abrimos os arquivos, vimos o vermelho no lugar do laranja e corrigimos os cinco botões.

O limite da abstração. Na base dos modais, a primeira tentativa foi colocar também o "abrir" (o show) na base. Não deu: cada modal abre com uma assinatura diferente (um com id, outro com um pacote de opções, outro sem nada), e o TypeScript reclamou de sobrescrita incompatível. A solução foi recuar: a base ficou só com "tá aberto", fechar e limpar; o "abrir" cada modal declara do seu jeito. A base centraliza o que é de fato igual; o que varia fica de fora.

O que NÃO migrou (honestidade). Alguns botões ficaram de fora de propósito, porque eram casos genuinamente diferentes: um botão escuro de importação, um azul de detalhe de projeto, barras de ferramenta com largura fixa que quebrariam a grade. A regra: consistente por papel, não idêntico à força. Quando o caso é mesmo diferente, ele continua diferente, e a gente anota o porquê.

Os doze erros antigos. O verificador acusa doze erros, mas são de antes da faxina. A tentação de "já que estou aqui, conserto" foi deixada de lado: misturar conserto de bug com reorganização embola tudo. Os doze ficaram anotados pra uma tarefa própria.

Resumo

A faxina foi botar cada coisa repetida num lugar só. Cor virou token com nome (e o botão primário ganhou um knob: muda a cor da marca, muda todo mundo). Código de lista e CRUD repetido virou fábrica, uma função que gera a peça. E os setenta componentes soltos foram pras gavetas das áreas, com as genéricas na gaveta ui. A lição que serve pra qualquer projeto: quando o mesmo valor (uma cor, um pedaço de lógica, um número mágico) aparece copiado em vários lugares, ele está pedindo um nome e um endereço único. Dá o nome, põe num lugar só, aponta todo mundo pra ele. Depois disso, mudar vira girar um botão, não caçar vinte cópias.
