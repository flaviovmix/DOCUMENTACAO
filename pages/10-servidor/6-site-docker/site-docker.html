Site em Docker — nginx + bind mount

0:00 / 0:00

Como servir um site estático (HTML/CSS/JS) num container Docker com nginx:alpine, em uns 5 comandos. Cobre o caminho do "por que preciso de servidor?" até "como deixo com senha" e "como subo dois sites em paralelo".

Os exemplos usam placeholders genéricos: IP_DA_VM, seu-usuario, seu-site. Substitua pelos valores reais.

Por que precisa de servidor web (e por que Docker) +

Um navegador não abre arquivo pela rede — ele faz uma requisição HTTP numa porta TCP, e precisa ter algum processo na VM escutando essa porta pra responder. Sem servidor, o navegador dá connection refused.

Opções de servidor

CaminhoQuando usar
Tomcat (já rodando)Se for app Java/WAR. Pra HTML estático também serve, mas é matar mosquito com bazuca.
python3 -m http.serverBrincar e abandonar — o Python já vem no Ubuntu, 1 comando e tem servidor. Mata quando fecha o terminal.
Docker + nginxSolução profissional: container reinicia sozinho, isolado do resto, fácil de duplicar, mesmo formato que ía pra produção.

A escolha é Docker aqui porque o objetivo é praticar Docker. Em meia hora você encosta nos 80% dos conceitos do dia-a-dia: image vs container, run / ps / logs / stop / rm, port mapping, bind mount, ciclo "edita arquivo → vê no browser".

Instalando Docker (script oficial) +

O Ubuntu oferece apt install docker.io, mas é versão antiga. O script oficial da Docker instala o docker-ce mais recente, já com docker compose v2 incluso — é o que aparece em todo tutorial.

curl -fsSL https://get.docker.com | sudo bash

Demora ~1 minuto. Vem o pacote completo: docker-ce (engine), docker-ce-cli (comando), containerd.io (runtime), docker-buildx-plugin e docker-compose-plugin.

Adicionar seu usuário ao grupo docker

Por padrão, só root roda docker. Pra não precisar de sudo a cada comando:

sudo usermod -aG docker seu-usuario
exit

O exit é obrigatório — a mudança de grupo só vale na próxima sessão SSH. Reconecta com ssh seu-usuario@IP_DA_VM e confirma:

docker --version

Tem que mostrar a versão sem pedir sudo. Se ainda pedir, você esqueceu de sair e reconectar.

Primeiro site no ar — 5 comandos +

Sequência mínima do "não tem nada" até "tem site rodando em http://IP_DA_VM:8000/":

1. Cria a pasta do site

mkdir -p ~/seu-site

2. Cria o index.html

nano ~/seu-site/index.html

No nano, cola:

<!DOCTYPE html>
<html>
<head><meta charset="UTF-8"><title>Teste</title></head>
<body style="font-family:sans-serif;text-align:center;padding:60px;">
    <h1>Funcionou!</h1>
    <p>Servido por nginx num container Docker.</p>
</body>
</html>

Salva com Ctrl+O → Enter → Ctrl+X.

3. Libera a porta no firewall (uma vez só)

sudo ufw allow 8000/tcp

Cada porta é liberada separadamente — abrir 8000 não abre 8001.

4. Sobe o container

docker run -d --name seu-site -p 8000:80 -v ~/seu-site:/usr/share/nginx/html nginx:alpine

5. Abre no browser

http://IP_DA_VM:8000/ — tem que aparecer "Funcionou!".

Editou o HTML depois? Só recarregar o navegador. Como a pasta está montada (-v), o nginx serve o arquivo novo na hora, sem rebuild nem restart.

Anatomia do docker run +

Cada flag do comando faz uma coisa específica:

PedaçoFunção
docker run"Cria e inicia container"
-dDetached — roda em background, libera o terminal
--name seu-siteApelido do container. Sem isso, Docker inventa nome aleatório engraçado (happy_einstein)
-p 8000:80Port mapping — porta 8000 da VM ↔ porta 80 dentro do container. Nginx sempre escuta na 80 dentro; o -p "abre uma janela" pra fora.
-v ~/seu-site:/usr/share/nginx/htmlBind mount — sua pasta vira o diretório servido pelo nginx
nginx:alpineImagem. nginx é o nome, alpine é a tag (versão minúscula ~40 MB). Sem tag, vem :latest (~140 MB).

O que rola por dentro, em ordem

  1. Docker procura nginx:alpine na máquina. Não tem.
  2. Baixa do Docker Hub (hub.docker.com/_/nginx) — várias camadas em paralelo
  3. Cria o container a partir da imagem
  4. Liga sua pasta do host em /usr/share/nginx/html dentro dele
  5. Liga a porta 8000 do host na 80 do container
  6. Roda o processo principal da imagem (o nginx)
  7. Imprime o ID do container (hash longo) e devolve o terminal

Da próxima vez, pula o passo 2 — a imagem já está em cache local. Subir cai pra < 1 segundo.

Bind mount vs COPY no Dockerfile — quando usar cada +

Soa estranho "mandar HTML pra container", mas com -v você não manda nada. A pasta do host fica visível dentro do container, e o nginx lê dela em tempo real. Edita no host → muda na hora.

ModoQuandoComo fica
Bind mount (-v)Desenvolvimento, site que muda toda horaArquivos vivem no host. Container só lê.
COPY no DockerfileProdução, deploy imutávelArquivos viram parte da imagem. Pra mudar precisa rebuild + restart.

Pra brincar e iterar, bind mount é exatamente o certo. Quando o site estabilizar e você quiser empacotar tudo numa imagem pronta pra docker push → servidor de produção, aí escreve um Dockerfile:

FROM nginx:alpine
COPY ./site /usr/share/nginx/html

Build com docker build -t meu-site . e roda com docker run -d -p 8000:80 meu-site. Imagem auto-contida, não depende de pasta nenhuma do host.

Transferindo arquivos do PC pra VM — scp e a pegadinha do .git +

Site mais complexo? Edita no PC e copia pra VM com scp (vem com o Windows). No PowerShell do Windows:

scp -r c:\caminho\do\projeto\* seu-usuario@IP_DA_VM:~/seu-site/
  • -r = recursivo (copia pasta e tudo dentro)
  • * = todos os arquivos da pasta
  • :~/seu-site/ = destino na VM

Pegadinha: o * não respeita .gitignore e pega tudo — inclusive a pasta .git/. Isso traz dois problemas:

  • Lento: .git/ tem milhares de arquivos pequenos, demora muito
  • Risco de segurança grave: se o nginx servir .git/ via HTTP, qualquer um abre http://IP_DA_VM:8000/.git/config e baixa o repositório inteiro com histórico. Existe ferramenta automatizada (gitdumper) que faz isso em segundos — bots varrem a internet procurando essa falha.

Solução simples: listar explicitamente o que enviar, pulando .git/:

scp -r c:\projeto\index.html c:\projeto\css c:\projeto\js c:\projeto\pages seu-usuario@IP_DA_VM:~/seu-site/

Múltiplos sites em paralelo — um container por porta +

Cada container é isolado: sua pasta, sua porta, seu processo. Dá pra ter quantos quiser rodando ao mesmo tempo — mesma imagem nginx:alpine, comportamentos diferentes:

# Site 1 (com senha, se você configurou auth)
docker run -d --name doc -p 8000:80 \
  -v ~/doc:/usr/share/nginx/html \
  -v ~/nginx-config/default.conf:/etc/nginx/conf.d/default.conf \
  -v ~/nginx-config/.htpasswd:/etc/nginx/conf.d/.htpasswd \
  nginx:alpine

# Site 2 (sem senha, público)
sudo ufw allow 8001/tcp
docker run -d --name site2 -p 8001:80 \
  -v ~/site2:/usr/share/nginx/html \
  nginx:alpine

docker ps mostra os dois rodando. Mesma imagem, mas comportamentos diferentes porque os volumes são diferentes. Pra parar/remover um sem mexer no outro:

docker stop site2 && docker rm site2
Adicionando login — Basic Auth do nginx +

3 peças: arquivo .htpasswd (usuário + hash), config do nginx (default.conf) com auth_basic, e docker run montando os dois como volumes.

1. Instala o utilitário htpasswd

sudo apt install -y apache2-utils

2. Cria o arquivo de senha

mkdir -p ~/nginx-config
htpasswd -c ~/nginx-config/.htpasswd seu-usuario

O -c cria o arquivo — usa só na primeira vez. Pra adicionar mais usuários depois, roda sem o -c (senão apaga o que tinha):

htpasswd ~/nginx-config/.htpasswd outro-usuario

Tem o flag -b que aceita a senha direto na linha (sem prompt interativo) — prático, mas a senha fica visível no histórico do bash:

htpasswd -c -b ~/nginx-config/.htpasswd admin minhasenha

3. Cria o default.conf do nginx

cat > ~/nginx-config/default.conf << 'EOF'
server {
    listen 80;
    server_name _;

    location / {
        auth_basic "Acesso restrito";
        auth_basic_user_file /etc/nginx/conf.d/.htpasswd;
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ =404;
    }
}
EOF

4. Recria o container com os 2 volumes extras

docker stop seu-site && docker rm seu-site
docker run -d --name seu-site -p 8000:80 \
  -v ~/seu-site:/usr/share/nginx/html \
  -v ~/nginx-config/default.conf:/etc/nginx/conf.d/default.conf \
  -v ~/nginx-config/.htpasswd:/etc/nginx/conf.d/.htpasswd \
  nginx:alpine

Abre http://IP_DA_VM:8000/ — o navegador mostra popup pedindo usuário e senha.

Removendo o login — e restart vs reload +

Pra deixar público de novo, edita o default.conf e remove as 2 linhas auth_basic:

nano ~/nginx-config/default.conf

Apaga essas duas:

        auth_basic "Acesso restrito";
        auth_basic_user_file /etc/nginx/conf.d/.htpasswd;

No nano, Ctrl+K apaga a linha inteira. Depois Ctrl+O → Enter → Ctrl+X.

Aplicando a mudança — restart ou reload?

O nginx só lê o default.conf na hora que sobe. Sem reiniciar, continua usando a config antiga em memória.

ComandoComo funcionaQuando usa
docker restart seu-sitePara e sobe o container inteiro (~1-2s downtime)Brincar, dev — mais fácil de lembrar
docker exec seu-site nginx -s reloadSó recarrega config dentro do container (zero downtime)Produção — não derruba conexões em andamento
Riscos de segurança — o que cuidar +

Cada vez que você abre uma porta no firewall, expande a superfície de ataque. Vale conhecer o que está em jogo:

🔴 Urgente — .git/ exposto via HTTP

Se copiar a pasta .git/ junto com os arquivos do site (clássico do scp -r *), qualquer pessoa abre http://IP_DA_VM:8000/.git/config e baixa o repositório inteiro com histórico. Bots varrem a internet procurando isso.

Fix: rm -rf ~/seu-site/.git, ou não copiar no scp.

🟠 Médio — Porta pública sem autenticação

Qualquer um que tope no IP vê o site inteiro. Se ele tem conteúdo sensível (credenciais, paths internos, screenshots de UI privada), vaza.

Fix: Basic Auth (acima), ou colocar atrás de Tailscale (não exposta na internet), ou Cloudflare Tunnel (Fase 5).

🟠 Médio — SSH na porta 22 pública

Bots fazem brute-force 24/7. Conferindo:

sudo journalctl -u ssh --since "1 hour ago" | grep -i fail

Senha forte + chave SSH protegem, mas continua barulhento.

Fix: fail2ban (bane IP que erra 3x), ou Tailscale + fechar a 22.

🟢 Higiene — sudo ainda pede senha

Sua senha forte virou ponto único de falha — se vazar, vira root. Pra cenário sério: 2FA no sudo.

Próximos passos pra praticar +

Sem mudar de domínio (continua sendo "servir site"), dá pra subir o sarrafo gradualmente:

  1. Trocar nginx:alpine por httpd:alpine (Apache) — mesma ideia, servidor diferente. Aprende que imagem é só "ambiente pronto".
  2. Escrever seu próprio Dockerfile com FROM nginx:alpine + COPY ./site /usr/share/nginx/html, dar docker build -t meu-site . e rodar — agora a imagem é sua.
  3. Subir 2 containers ao mesmo tempo em portas diferentes — consolida que cada container é isolado.
  4. docker compose com um docker-compose.yml — declara serviços num arquivo em vez de comando longo. É o que se usa em projeto real.

Quando travar no passo 4, já dá pra encarar coisa séria tipo subir um Postgres + app Java junto com a Fase 2 (Tomcat) e Fase 3 (PostgreSQL).
