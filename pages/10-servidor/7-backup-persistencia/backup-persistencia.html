Backup & Persistência — dados que sobrevivem ao container e à VM

0:00 / 0:00

Como fazer dados durarem além do ciclo de vida do container (volumes), além do ciclo de vida da própria VM (backup off-site), e como restaurar tudo numa máquina nova em uns 5 minutos. Pré-requisito: Site em Docker — você já tem Docker rodando e entende docker run, bind mount e portas.

Os exemplos usam placeholders genéricos: IP_DA_VM, seu-usuario, seu-bucket, meu-app. Substitua pelos valores reais.

O default é perder tudo +

A primeira coisa que assusta quem começa com Docker: dentro de um container, dados são voláteis por design. Você roda um Postgres, popula com tabelas, insere 5 mil registros, dá docker rm no container, e tudo vira pó. Os 5 gigas viraram zero.

Não é bug, é propósito. A ideia de container é ser descartável — você mata e cria outro em segundos, sempre limpo. Ótimo pra app stateless, catastrófico pra banco de dados.

A saída: separar o ciclo de vida do container do ciclo de vida do dado. Isso se faz com volume.

Analogia: container é barraca de camping, volume é o cofre que você planta no chão. Você desmonta a barraca e arma outra em cima — o que tem no cofre continua lá.

Volume nomeado vs bind mount +

Volume é uma pasta de fora do container que você "monta" dentro dele. O programa lá dentro escreve normal no caminho que conhece, mas os bytes vão parar do lado de fora — num lugar que sobrevive quando o container morre.

TipoOnde ficaQuando usar
Volume nomeadoDocker gerencia, em /var/lib/docker/volumes/Banco de dados — recomendado, mais performático, mais portável
Bind mountPasta sua, em qualquer caminhoDesenvolvimento — quando você quer ver e editar arquivos no host

Volume nomeado

docker run -v pgdata:/var/lib/postgresql/data postgres:16-alpine

Docker cria o volume pgdata automaticamente na primeira vez. Você não precisa saber o caminho exato — docker volume ls lista, docker volume inspect pgdata mostra detalhes.

Bind mount

docker run -v ~/meus-dados-pg:/var/lib/postgresql/data postgres:16-alpine

Você vê os arquivos diretamente em ~/meus-dados-pg. Útil pra HTML, configs e logs — não pra dados binários do Postgres.

Pro Postgres, use volume nomeado. Pra HTML (como no episódio anterior), bind mount.

Três camadas de risco — cada uma exige solução diferente +

Antes de entrar em backup, vale separar três camadas de risco. Cada uma tem solução própria, e as três precisam ser cobertas pra você ter "dados seguros" de verdade.

RiscoCenárioSolução
1. Container morredocker rm sem querer, docker compose down sem flag de manter volumesVolume nomeado — container vai, dado fica
2. Dado corrompe ou someVolume sobrevive, mas alguém rodou DROP TABLEBackup histórico com várias versões no tempo — pra poder voltar pro estado de ontem, semana passada, mês passado
3. VM inteira morreDeletada por engano, ransomware, você perdeu o login, datacenter pegou fogoBackup off-site em outro lugar do mundo

Backup local cobre 1 e 2. Backup off-site cobre 3. Os próximos accordions resolvem na ordem.

Backup local automatizado — pg_dump + cron ou imagem pronta +

Backup de Postgres se faz com pg_dump — gera um arquivo SQL com estrutura e dados, restaurável depois com pg_restore. Pra rodar de dentro do container:

docker exec nome-container pg_dump -U postgres -Fc nome-banco > backup.dump

O -Fc gera formato custom comprimido — ~5x menor que SQL puro, e restore mais rápido.

Caminho 1 — cron + script

Script ~/bin/pg-backup.sh:

#!/bin/bash
BACKUP_DIR=~/backups/postgres
mkdir -p "$BACKUP_DIR"
DATE=$(date +%Y-%m-%d-%H)
docker exec pg pg_dump -U postgres -Fc meu-app > "$BACKUP_DIR/meu-app-$DATE.dump"
find "$BACKUP_DIR" -name "meu-app-*.dump" -mtime +7 -delete

Dá permissão (chmod +x), edita o crontab (crontab -e) e adiciona pra rodar de hora em hora:

0 * * * * /home/seu-usuario/bin/pg-backup.sh

Caminho 2 — imagem pronta (recomendado pra projeto com compose)

A imagem prodrigestivill/postgres-backup-local faz tudo isso, com rotação por dia + semana + mês. Adiciona no docker-compose.yml:

backup:
  image: prodrigestivill/postgres-backup-local
  environment:
    POSTGRES_HOST: db
    POSTGRES_DB: meu-app
    POSTGRES_USER: postgres
    POSTGRES_PASSWORD: sua-senha
    SCHEDULE: "@hourly"
    BACKUP_KEEP_DAYS: 7
    BACKUP_KEEP_WEEKS: 4
    BACKUP_KEEP_MONTHS: 6
  volumes:
    - ~/backups:/backups
  depends_on:
    - db

Vira parte da infra declarada — viaja junto com o resto do projeto quando você mover pra outra VM.

Frequência — de hora em hora é exagero saudável

Pra app com poucas escritas/dia, dump por hora pesa kilobytes. Ter 168 versões dos últimos 7 dias custa quase nada de disco e te dá granularidade pra recuperar com perda máxima de 1h.

Por que backup local não basta +

Você tem 168 dumps locais bonitinhos, rotacionados, automatizados. Cenários em que isso não salva:

  • A VM é deletada por engano no provedor (acontece)
  • Você esquece a senha do sudo e perde acesso à máquina inteira
  • A VM é invadida e o atacante criptografa todo o disco (ransomware)
  • O datacenter onde a VM mora pega fogo — sumiu junto com o disco

Em todos esses cenários, os backups locais foram junto. Mesma física — mesmo HD, mesmo datacenter, mesma conta. Pra cobrir isso, você precisa de cópia em outro lugar. Outra geografia, outra conta, outro provedor. É o que se chama backup off-site.

Analogia: backup local é cofre no porão da casa — protege contra raio que queimou seu HD ou exclusão acidental. Não protege contra incêndio que queimou a casa toda. Pra isso você precisa do outro cofre, em outro endereço.

Backup off-site com Backblaze B2 + rclone +

Pra mandar arquivos pra fora da VM tem muita opção — AWS S3, Google Cloud, Dropbox, Backblaze B2. Pra projeto pessoal, recomendo Backblaze B2: tipo um S3 mais barato (US$ 0.005/GB/mês), e free tier de 10 GB.

1. Cria conta e bucket no B2

Em backblaze.com/b2, cria conta, cria um bucket chamado seu-bucket, gera uma Application Key com permissão de escrita.

2. Instala rclone na VM

curl https://rclone.org/install.sh | sudo bash
rclone config

O rclone config é um wizard interativo. Escolhe Backblaze B2, cola Key ID + Application Key. Salva como remote chamado b2.

3. Script de sync

#!/bin/bash
# ~/bin/sync-backups.sh
rclone sync ~/backups b2:seu-bucket/backups \
  --transfers 4 --checksum --log-file=/var/log/rclone.log

4. Cron de hora em hora, 5 minutos depois do backup local

5 * * * * /home/seu-usuario/bin/sync-backups.sh

A cada hora, o estado da pasta ~/backups é espelhado pro bucket. Arquivos novos sobem, deletados pela rotação local somem do bucket também. Zero intervenção sua.

Custo: um banco de poucos MB gera dumps de poucos KB. 168 dumps/dia × 365 dias = alguns MB/ano. Sua conta nem chega perto do free tier de 10 GB.

O que vai no Git vs o que vai no B2 +

Separação crítica — entender o que precisa de cada destino:

ItemDestinoPor quê
Código da aplicaçãoGit (GitHub)Texto, versão, re-criável
Dockerfile, docker-compose.ymlGitDefine COMO a aplicação roda
Scripts de setup, cron, rcloneGitTexto, versão
Dumps do PostgresB2 (storage cloud)Crítico — gerado em runtime, não re-criável
Arquivos enviados por usuários (uploads)B2Também runtime
Imagens Docker customNenhum — ou registry tipo GHCRRe-buildadas do Dockerfile do Git
Containers em execuçãoNenhumRe-criados com docker compose up

Regra geral: tudo re-criável a partir de código vai no Git. Tudo gerado em runtime que não dá pra recriar vai no storage cloud.

Disaster recovery em ~5 minutos +

Cenário: VM morreu. Você tem uma máquina nova (outro VPS, outra VM, ou até seu notebook). Procedimento exato:

# 1. Instala Docker (~1 min)
curl -fsSL https://get.docker.com | sudo bash
sudo usermod -aG docker $USER && newgrp docker

# 2. Clona o repo do projeto (~10s)
git clone git@github.com:seu-user/meu-app.git
cd meu-app

# 3. Configura rclone com a mesma conta B2 e baixa o último dump (~10s)
rclone config   # cola Key ID + Application Key
rclone copy b2:seu-bucket/backups/last/meu-app-latest.sql.gz .

# 4. Sobe a stack (~1-2 min — pull das imagens)
docker compose up -d

# 5. Restaura o dump (segundos pra MBs)
gunzip -c meu-app-latest.sql.gz | docker exec -i meu-app-db-1 psql -U postgres -d meu-app

Total realista: 3 a 7 minutos. Não é literalmente "1 segundo" — pra isso você precisaria de replicação em tempo real e standby ligada (alta disponibilidade real), várias ordens de magnitude mais cara e complexa, exagero pra projeto pessoal.

Pra projeto pessoal ou SaaS pequeno, 5 minutos de downtime no pior caso da vida é absolutamente aceitável. Vale infinitamente mais investir nisso do que sonhar com zero downtime.

Testar restore mensalmente — o passo que ninguém faz +

Backup que você nunca restaurou não é backup, é estatística. Pode estar tudo corrompido. Pode ter mudado a versão do Postgres e o dump não restaurar. Pode ter algum caractere estranho que dá erro no import. Pode ter algum bug no script de backup que silenciosamente gera dumps vazios há 6 meses. Você só descobre no dia da emergência.

Solução — teste mensal de 5 minutos

  1. Sobe uma VM descartável (free tier de qualquer provedor) ou usa seu notebook
  2. Roda o procedimento de disaster recovery do accordion anterior
  3. Verifica que a aplicação sobe e mostra os dados esperados
  4. Destrói a VM

Anota a data num lugar visível ("último teste de restore: 2026-MM-DD"). Se passou mais de 30 dias, você não tem mais backup — tem arquivos com extensão .sql.gz numa pasta.

Mantra: backup não testado não é backup. O teste mensal é o que separa "tenho backup" de "tenho backup que funciona".
